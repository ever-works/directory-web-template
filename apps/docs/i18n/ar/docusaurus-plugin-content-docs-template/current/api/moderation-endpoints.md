# نظام الاعتدال

يوفر نظام الإشراف الإشراف البرمجي على المحتوى من خلال طبقة خدمة بدلاً من نقاط نهاية واجهة برمجة التطبيقات المستقلة. يتم تشغيل إجراءات الإشراف تلقائيًا عندما يقوم المسؤولون بحل تقارير المحتوى عبر Reports API. يدعم النظام تحذير المستخدمين، وتعليق الحسابات، وحظر الحسابات، وإزالة المحتوى، مع سجل التدقيق الكامل وإشعارات البريد الإلكتروني.

## نظرة عامة

لا يتم عرض الإشراف كنقاط نهاية REST منفصلة. بدلاً من ذلك، يتم استدعاؤه من خلال سير عمل تحليل التقرير:

```
PUT /api/admin/reports/[id]  -->  resolution triggers moderation action
```

عندما يقوم المسؤول بتعيين قيمة `resolution` في تقرير، يتم تنفيذ وظيفة الإشراف المقابلة تلقائيًا.

|قيمة القرار|وظيفة الاعتدال|تأثير|
|---|---|---|
|`content_removed`|`removeContent()`|يقوم بحذف التعليق أو العنصر الذي تم الإبلاغ عنه بشكل مبدئي|
|`user_warned`|`warnUser()`|يزيد عدد تحذيرات المستخدم|
|`user_suspended`|`suspendUser()`|يضبط حالة المستخدم على `"suspended"`|
|`user_banned`|`banUser()`|يضبط حالة المستخدم على `"banned"`|
|`no_action`|لا شيء|لم يتم اتخاذ أي إجراء للاعتدال|

## إجراءات الاعتدال

### إزالة المحتوى

```typescript
removeContent(contentType, contentId, reportId, adminId): Promise<ModerationResult>
```

يزيل المحتوى المبلغ عنه بناءً على نوعه. بالنسبة للتعليقات، يؤدي هذا إلى إجراء حذف بسيط (يعين `deletedAt`). بالنسبة للعناصر، يؤدي هذا إلى حذف العنصر من مستودع المحتوى المستند إلى Git.

**المعلمات:**

|المعلمة|اكتب|الوصف|
|---|---|---|
|`contentType`|`"item"` أو `"comment"`|نوع المحتوى المراد إزالته|
|`contentId`|سلسلة|معرف أو سبيكة المحتوى|
|`reportId`|سلسلة|معرف التقرير المرتبط|
|`adminId`|سلسلة|المستخدم المسؤول يقوم بتنفيذ الإجراء|

**خطوات المعالجة:**

1. ابحث عن مالك المحتوى عبر `getContentOwner()`
2. إذا كان التعليق: قم بالحذف الناعم عبر `deleteComment()`
3. إذا كان العنصر: احذفه من مستودع Git عبر `itemRepository.delete()`
4. تسجيل سجل الإشراف باستخدام الإجراء `CONTENT_REMOVED`
5. إرسال إشعار بالبريد الإلكتروني لإزالة المحتوى إلى مالك المحتوى

**المصدر:** `template/lib/services/moderation.service.ts`

### تحذير المستخدم

```typescript
warnUser(userId, reason, reportId, adminId): Promise<ModerationResult>
```

يُصدر تحذيرًا للمستخدم عن طريق زيادة حقل `warningCount` الخاص به. لا يمكن للمستخدمين المحظورين بالفعل تلقي تحذيرات.

**المعلمات:**

|المعلمة|اكتب|الوصف|
|---|---|---|
|`userId`|سلسلة|معرف ملف تعريف العميل للمستخدم|
|`reason`|سلسلة|سبب التحذير|
|`reportId`|سلسلة|معرف التقرير المرتبط|
|`adminId`|سلسلة|المستخدم المسؤول يقوم بتنفيذ الإجراء|

**خطوات المعالجة:**

1. تحقق من وجود المستخدم وأنه ليس محظورًا بالفعل
2. زيادة عدد التحذيرات عبر `incrementWarningCount()`
3. تسجيل سجل الإشراف باستخدام الإجراء `WARN`
4. إرسال إشعار تحذيري عبر البريد الإلكتروني مع عدد التحذيرات الحالي

**نتيجة النجاح:**

```json
{
  "success": true,
  "message": "User warned successfully. Total warnings: 3"
}
```

**المصدر:** `template/lib/services/moderation.service.ts`

### تعليق المستخدم

```typescript
suspendUser(userId, reason, reportId, adminId): Promise<ModerationResult>
```

تعليق حساب مستخدم عن طريق ضبط حالته على `"suspended"` وتسجيل الطابع الزمني `suspendedAt`. لا يمكن للمستخدمين المعلقين إنشاء تعليقات أو إرسال أصوات أو تقديم تقارير.

**الحرس:**

- إرجاع خطأ إذا كان المستخدم معلقًا بالفعل
- إرجاع خطأ إذا كان المستخدم محظورًا بالفعل

**خطوات المعالجة:**

1. تحقق من وجود المستخدم وأنه ليس موقوفًا أو محظورًا بالفعل
2. اضبط الحالة على `"suspended"` باستخدام الطابع الزمني `suspendedAt`
3. تسجيل سجل الإشراف باستخدام الإجراء `SUSPEND`
4. أرسل إشعارًا بالتعليق عبر البريد الإلكتروني

**المصدر:** `template/lib/services/moderation.service.ts`

### حظر المستخدم

```typescript
banUser(userId, reason, reportId, adminId): Promise<ModerationResult>
```

حظر حساب مستخدم نهائيًا عن طريق ضبط حالته على `"banned"` وتسجيل الطابع الزمني `bannedAt`. يتم منع المستخدمين المحظورين من كافة الإجراءات المصادق عليها.

**الحرس:**

- إرجاع خطأ إذا كان المستخدم محظورًا بالفعل

**خطوات المعالجة:**

1. تحقق من وجود المستخدم وأنه ليس محظورًا بالفعل
2. اضبط الحالة على `"banned"` باستخدام الطابع الزمني `bannedAt`
3. تسجيل سجل الإشراف باستخدام الإجراء `BAN`
4. إرسال إشعار الحظر عبر البريد الإلكتروني

**المصدر:** `template/lib/services/moderation.service.ts`

## قرار مالك المحتوى

تحدد الوظيفة `getContentOwner()` من يملك المحتوى المُبلغ عنه:

|نوع المحتوى|مصدر المالك|
|---|---|
|`comment`|`comment.userId` من جدول التعليقات|
|`item`|`item.submitted_by` الحقل من مستودع العناصر|

يتم استخدام هذا من خلال جميع إجراءات الإشراف على مستوى المستخدم (`user_warned`، `user_suspended`، `user_banned`) لتحديد المستخدم المستهدف للإجراء.

**المصدر:** `template/lib/services/moderation.service.ts`

## تاريخ الاعتدال

تقوم كافة إجراءات الإشراف بإنشاء مسار تدقيق في جدول قاعدة البيانات `moderationHistory`.

### حقول سجل التاريخ

|الميدان|اكتب|الوصف|
|---|---|---|
|`id`|سلسلة|معرف السجل الفريد|
|`userId`|سلسلة|معرف ملف تعريف العميل للمستخدم المتأثر|
|`action`|سلسلة|`"CONTENT_REMOVED"`، `"WARN"`، `"SUSPEND"`، أو `"BAN"`|
|`reason`|سلسلة أو فارغة|سبب اتخاذ إجراء الاعتدال|
|`reportId`|سلسلة أو فارغة|معرف التقرير المرتبط|
|`performedBy`|سلسلة أو فارغة|معرف مستخدم المسؤول الذي نفذ الإجراء|
|`contentType`|سلسلة أو فارغة|`"item"` أو `"comment"` (لإزالة المحتوى)|
|`contentId`|سلسلة أو فارغة|معرف المحتوى الذي تمت إزالته|
|`details`|كائن أو فارغة|سياق إضافي (على سبيل المثال، عدد التحذيرات واسم العنصر)|
|`createdAt`|الطابع الزمني|عندما تم تنفيذ الإجراء|

### استعلامات التاريخ

|وظيفة|الوصف|
|---|---|
|`getModerationHistoryByUser(userId, limit)`|الحصول على جميع إجراءات الإشراف للمستخدم (الحد الافتراضي: 50)|
|`getModerationHistoryByReport(reportId)`|احصل على إجراءات الإشراف المرتبطة بتقرير محدد|

تعمل كلتا وظيفتي الاستعلام على إثراء النتائج بمعلومات ملف تعريف المستخدم وتفاصيل المؤدي الإداري.

**المصدر:** `template/lib/db/queries/moderation.queries.ts`

## إدارة حالة المستخدم

### قيم الحالة

|الحالة|الوصف|
|---|---|
|`active`|حساب عادي، كل الميزات متوفرة|
|`suspended`|مقيد مؤقتًا، ولا يمكن إنشاء محتوى|
|`banned`|مقيد بشكل دائم، محظور من جميع الإجراءات|

### عمليات قاعدة البيانات

|وظيفة|الوصف|
|---|---|
|`suspendUser(userId)`|يضبط الحالة على `"suspended"`، ويسجل `suspendedAt`|
|`unsuspendUser(userId)`|يعيد الحالة إلى `"active"`، ويمسح `suspendedAt`|
|`banUser(userId)`|يضبط الحالة على `"banned"`، ويسجل `bannedAt`|
|`unbanUser(userId)`|يعيد الحالة إلى `"active"`، ويمسح `bannedAt`|
|`incrementWarningCount(userId)`|الزيادات `warningCount` باستخدام SQL `COALESCE`|

### الشيكات المستخدم المحظورة

هناك وظيفتان مساعدتان للتحقق من حالة المستخدم عبر التطبيق:

- **`isUserBlocked(status)`** -- يُرجع `true` إذا كانت الحالة هي `"suspended"` أو `"banned"`
- **`getBlockReasonMessage(status)`** - إرجاع رسالة موجهة للمستخدم تشرح سبب تقييد الإجراء

يتم استخدام عمليات التحقق هذه بواسطة نقاط نهاية التعليقات والتصويتات والتقارير لمنع المستخدمين المحظورين من إنشاء المحتوى.

**المصدر:** `template/lib/db/queries/moderation.queries.ts`

## إشعارات البريد الإلكتروني

يرسل `EmailNotificationService` إشعارات غير محظورة لإجراءات الإشراف:

|الطريقة|الزناد|
|---|---|
|`sendContentRemovedEmail(email, type, reason)`|تمت إزالة المحتوى من قبل المشرف|
|`sendUserWarningEmail(email, reason, count)`|صدر تحذير|
|`sendUserSuspensionEmail(email, reason)`|تم تعليق الحساب|
|`sendUserBanEmail(email, reason)`|الحساب محظور|

تستخدم جميع رسائل البريد الإلكتروني `.catch()` لمنع حالات الفشل من مقاطعة سير عمل الإشراف. لا يؤدي فشل البريد الإلكتروني إلى فشل إجراء الإشراف نفسه.

## تفاصيل التنفيذ الرئيسية

- **نمط طبقة الخدمة:** يوجد منطق الإشراف في `lib/services/moderation.service.ts`، وليس في معالجات مسار API. وهذا يسمح بإعادة الاستخدام عبر نقاط دخول مختلفة.
- **مسار التدقيق:** ينشئ كل إجراء إشراف سجل `moderationHistory`، مما يوفر سجل تدقيق كامل للامتثال والمراجعة.
- ** رسائل البريد الإلكتروني غير المحظورة: ** يتم إرسال إشعارات البريد الإلكتروني بشكل غير متزامن مع معالجات `.catch()`. إذا كانت خدمة البريد الإلكتروني غير متوفرة، فسيظل إجراء الإشراف ناجحًا.
- ** حراس العجز: ** يتحقق كل إجراء من حالة المستخدم الحالية قبل المتابعة. يؤدي حظر مستخدم محظور بالفعل إلى ظهور خطأ بدلاً من إنشاء إجراء مكرر.
- **الحذف الناعم مقابل الحذف الصعب:** يتم حذف التعليقات بشكل مبدئي (الإعداد `deletedAt`)، بينما تتم إزالة العناصر بالكامل من مستودع Git. يعكس هذا الاختلاف نموذج التخزين (قاعدة البيانات مقابل المحتوى المستند إلى الملف).
