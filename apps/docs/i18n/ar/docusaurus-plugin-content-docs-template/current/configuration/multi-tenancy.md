# تكوين متعدد المستأجرين

يشرح هذا المستند كيفية عمل دعم تعدد المستأجرين في Directory Web Template.

## نظرة عامة

يستخدم القالب نهج **قاعدة بيانات مشتركة مع عزل على مستوى الصف**:

- تخدم قاعدة بيانات PostgreSQL واحدة **مستأجرين** متعددين (مواقع الدليل).
- تحتوي كل جدول على عمود `tenant_id` يقصر البيانات على مستأجر معين.
- تقوم جميع الاستعلامات بالتصفية تلقائيًا وفقًا للمستأجر الحالي — لا تسرب بيانات بين المستأجرين.

## الإعداد السريع

### 1. تعيين متغير البيئة

في منصة النشر (Vercel أو Docker أو غيرها) أو في `.env.local`:

```bash
TENANT_ID="your-unique-tenant-id"
```

يمكن أن يكون أي سلسلة فريدة (مثل UUID أو slug قابل للقراءة مثل `"my-directory"`).

### 2. النشر

عند التشغيل الأول، سيقوم التطبيق بما يلي:

1. تشغيل هجرات قاعدة البيانات (إضافة عمود `tenant_id` إذا لم يكن موجودًا)
2. إنشاء صف مستأجر يتطابق مع قيمة `TENANT_ID`
3. ترحيل البيانات الموجودة ذات قيمة NULL في `tenant_id` إلى مستأجرك
4. زرع البيانات الافتراضية (مستخدم المشرف والأدوار والأذونات)

لا يلزم SQL يدوي — كل شيء تلقائي.

### 3. التحقق

تحقق من سجلات الخادم بحثًا عن:

```
[DB Init] Ensured environment tenant 'your-unique-tenant-id' exists
[Tenant Migration] ✓ users: updated 3 rows
[Tenant Migration] ✅ Migration complete: 15 total rows updated across all tables.
```

## كيفية عمل تحليل المستأجر

عندما يحتاج التطبيق إلى تحديد المستأجر الحالي، يستخدم استراتيجية **الشلال**:

| الأولوية | المصدر              | الوصف                                                         |
| -------- | ------------------- | ------------------------------------------------------------- |
| 1        | **الجلسة**          | `user.tenantId` من رمز JWT (المستخدمون المصادق عليهم)         |
| 2        | **متغير البيئة**    | متغير البيئة `TENANT_ID`                                      |
| 3        | **رأس HTTP**        | رأس `x-tenant-domain` (لتوجيه النطاق الفرعي)                 |
| 4        | **قاعدة البيانات**  | أول صف مستأجر نشط (الاحتياط النهائي)                          |

تُنفذ الدالة `getTenantId()` من `lib/auth/tenant.ts` هذه السلسلة وتُستدعى بواسطة كل استعلام قاعدة بيانات.

## البنية المعمارية

### الملفات الرئيسية

| الملف                                    | الغرض                                                                    |
| ---------------------------------------- | ------------------------------------------------------------------------ |
| `lib/auth/tenant.ts`                     | `getTenantId()` — تحليل المستأجر من جانب الخادم مع التخزين المؤقت       |
| `lib/config/env.ts`                      | التحقق من صحة متغير البيئة `TENANT_ID`                                   |
| `lib/db/schema.ts`                       | جدول المستأجرين + مفتاح خارجي `tenant_id` على جميع الجداول               |
| `lib/db/initialize.ts`                   | يُنشئ تلقائيًا مستأجر البيئة + يُشغّل هجرة البيانات عند البدء            |
| `lib/db/migrate-tenant-data.ts`          | يُعيّن صفوف `tenant_id` ذات قيمة NULL إلى المستأجر الحالي               |
| `lib/auth/index.ts`                      | ردود نداء JWT/الجلسة تُحقن `tenantId`                                    |
| `components/context/tenant-provider.tsx` | سياق React للوصول إلى المستأجر من جانب العميل                           |
| `app/api/tenant/route.ts`                | `GET /api/tenant` — يُعيد معلومات المستأجر الحالي                        |

### تدفق البيانات

```
طلب المستخدم → getTenantId() → يُحلل من الجلسة/البيئة/الرؤوس/قاعدة البيانات
                                            ↓
                       جميع استعلامات قاعدة البيانات تُصفى بواسطة tenant_id هذا
                                            ↓
                             تُعاد فقط البيانات لهذا المستأجر
```

### التكامل مع المصادقة

- **تسجيل الدخول بالبيانات الاعتمادية**: يحصل مستخدمو المشرف والعميل على `tenantId` من عمود `users.tenant_id`.
- **تسجيل الدخول عبر OAuth**: يُغلَّف محوّل Drizzle لحقن `tenantId` عند إنشاء المستخدم.
- **رد نداء JWT**: يقرأ `tenantId` من سجل المستخدم ويضمّنه في الرمز.
- **رد نداء الجلسة**: ينقل `tenantId` إلى `session.user.tenantId`.
- **مكونات العميل**: تستخدم الخطاف `useTenant()` من `TenantProvider` للحصول على معلومات المستأجر.

## أدلة متعددة (تعدد المستأجرين)

لتشغيل مواقع دليل متعددة على قاعدة بيانات واحدة:

1. **كل موقع** يُعيّن `TENANT_ID` مختلفًا في بيئته:
    - الموقع أ: `TENANT_ID="directory-a-uuid"`
    - الموقع ب: `TENANT_ID="directory-b-uuid"`

2. **جميع المواقع** تتصل بـ**قاعدة البيانات نفسها** (`DATABASE_URL`).

3. **عزل البيانات** تلقائي — الموقع أ يرى فقط الصفوف حيث `tenant_id = 'directory-a-uuid'`.

4. **المستخدمون والأدوار والتعليقات والاشتراكات** وجميع البيانات الأخرى معزولة تمامًا لكل مستأجر.

## التعامل مع البيانات الموجودة

عند الترقية من نسخة بدون مستأجرين:

- يُضاف عمود `tenant_id` كـ**nullable** (لا يكسر البيانات الموجودة)
- عند التشغيل الأول، تُعيّن `migrateNullTenantIds()` تلقائيًا الصفوف ذات NULL إلى المستأجر المُحلَّل
- هذه الهجرة **متحايدة** — آمنة للتشغيل مرات متعددة
- بعد الهجرة، تكون جميع البيانات الموجودة مرئية ضمن المستأجر الحالي

## توجيه النطاق الفرعي (متقدم)

لتوجيه المستأجر المستند إلى النطاق الفرعي (مثل `tenant-a.example.com`):

1. قم بتكوين الوكيل العكسي لإضافة رأس `x-tenant-domain`
2. أنشئ سجلات المستأجرين مع تعيين حقول `domain` أو `slug`:
    ```sql
    INSERT INTO tenant (id, name, domain, slug, status)
    VALUES ('uuid', 'Tenant A', 'tenant-a.example.com', 'tenant-a', 'active');
    ```
3. ستُطابق استراتيجية `resolveFromHeaders()` النطاق وتُحلل المستأجر

## مخطط جدول المستأجرين

```sql
CREATE TABLE tenant (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  website TEXT,
  domain TEXT UNIQUE,
  slug TEXT UNIQUE,
  status TEXT NOT NULL DEFAULT 'active',  -- 'active' | 'inactive'
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMP NOT NULL DEFAULT NOW()
);
```
