# التوسع والتوافر العالي

يغطي هذا الدليل استراتيجيات توسيع Ever Works Template من نشر مثيل واحد إلى تكوين إنتاج ذي توافر عالٍ، بما في ذلك تكوين serverless وتجميع الاتصالات وتحسين CDN ووظائف الحافة.

## معماريات النشر

يدعم القالب معماريات نشر متعددة:

| المعمارية | الأنسب لـ | نموذج التوسع |
|---|---|---|
| Vercel (Serverless) | معظم عمليات النشر | توسع أفقي تلقائي |
| Docker (مستقل) | الاستضافة الذاتية، المحلية | يدوي أو قائم على المُنسق |
| Node.js (مباشر) | التطوير والنشر البسيط | مثيل واحد أو كتلة PM2 |

## تكوين Serverless (Vercel)

### المخرجات المستقلة

تم تكوين القالب بمخرجات مستقلة لنشر serverless محسن:

```typescript
// next.config.ts
const nextConfig: NextConfig = {
  output: "standalone",
};
```

يُنشئ الوضع المستقل بناءً مكتفياً بذاته في `.next/standalone/` يحتوي فقط على الملفات اللازمة لتشغيل التطبيق. يُقلل أوقات البدء البارد بتقليل حجم حزمة النشر.

### تكوين الوظائف

قم بتكوين إعدادات وظيفة serverless في `vercel.json` أو عبر تكوين مستوى المسار:

```typescript
// app/api/heavy-computation/route.ts
export const maxDuration = 60; // ثانية (خطة Pro: حتى 300 ثانية)
export const dynamic = 'force-dynamic';
```

### إعدادات الوظائف الموصى بها

| نوع المسار | المدة القصوى | الذاكرة | ملاحظات |
|---|---|---|---|
| مسارات API (بسيطة) | 10 ثواني | 1024 ميجابايت | الافتراضي لمعظم نقاط النهاية |
| مسارات API (معالجة البيانات) | 30 ثانية | 1024 ميجابايت | للعمليات الدفعية |
| مهام Cron | 60 ثانية | 1024 ميجابايت | تنفيذ المهام الخلفية |
| معالجات Webhook | 30 ثانية | 1024 ميجابايت | عمليات رد الاتصال للمدفوعات وOAuth |
| الصفحات الثابتة | غير مطبق | غير مطبق | مُصيَّرة مسبقاً عند البناء |

### تحسين البدء البارد

تقليل حالات البدء البارد باستخدام هذه التقنيات:

| التقنية | التنفيذ | التأثير |
|---|---|---|
| تقليل حجم الوظيفة | `serverExternalPackages` في التكوين | تقليل وقت التهيئة |
| تجنب الاستيراد على مستوى الوحدة | `import()` ديناميكي للوحدات الثقيلة | تأجيل التحميل حتى الحاجة |
| استخدام وقت تشغيل الحافة حيثما أمكن | `export const runtime = 'edge'` | شبه صفري للبدء البارد |
| إبقاء الوظائف دافئة | نقاط نهاية فحص الصحة مع المراقبة | إبقاء الوظائف نشطة |

## تجميع اتصالات قاعدة البيانات

### المشكلة

في بيئات serverless، قد يفتح كل استدعاء وظيفة اتصالاً جديداً بقاعدة البيانات. بدون تجميع الاتصالات، قد يستنفد ذلك حد اتصالات قاعدة البيانات.

### الحل: مُجمِّع الاتصالات

استخدم مُجمِّع اتصالات بين التطبيق وقاعدة البيانات:

| المُجمِّع | الموفر | التكوين |
|---|---|---|
| PgBouncer | Supabase (مدمج) | استخدام سلسلة اتصال مجمّعة (المنفذ 6543) |
| Neon Pooler | Neon (مدمج) | استخدام سلسلة اتصال `-pooler` |
| PgBouncer | مستضاف ذاتياً | نشر PgBouncer بجانب PostgreSQL |

### التكوين

استخدم سلاسل اتصال مختلفة للاتصالات المجمّعة والمباشرة:

```bash
# اتصال مجمّع لاستعلامات التطبيق (آمن لـ serverless)
DATABASE_URL=postgresql://user:pass@host:6543/db?pgbouncer=true

# اتصال مباشر للترحيلات فقط
DIRECT_DATABASE_URL=postgresql://user:pass@host:5432/db
```

تحديث `drizzle.config.ts` لاستخدام الاتصال المباشر للترحيلات:

```typescript
export default {
  schema: "./lib/db/schema.ts",
  out: "./lib/db/migrations",
  dialect: "postgresql",
  dbCredentials: {
    url: process.env.DIRECT_DATABASE_URL || process.env.DATABASE_URL,
  },
} satisfies Config;
```

### حدود الاتصال

| المستوى | الحد الأقصى للاتصالات | حجم التجميع الموصى به |
|---|---|---|
| Hobby (Neon/Supabase) | 50–100 | 10–20 |
| Pro (Neon/Supabase) | 200–500 | 50–100 |
| Enterprise | 1000+ | 100–200 |

### إدارة الاتصالات في الكود

يجب أن تُعيد وحدة قاعدة البيانات في القالب استخدام تجمع اتصالات واحد لكل مثيل وظيفة:

```typescript
// lib/db/index.ts
import { drizzle } from 'drizzle-orm/postgres-js';
import postgres from 'postgres';

// إنشاء تجمع اتصالات مرة واحدة لكل مثيل serverless
const connectionString = process.env.DATABASE_URL!;
const client = postgres(connectionString, {
  max: 10,          // الحد الأقصى لعدد الاتصالات في التجمع
  idle_timeout: 20, // إغلاق الاتصالات الخاملة بعد 20 ثانية
  connect_timeout: 10,
});

export const db = drizzle(client);
```

## شبكات CDN والتخزين المؤقت

### شبكة Vercel Edge

عند النشر على Vercel، توفر شبكة Edge تلقائياً:

- توزيع CDN عالمي في أكثر من 30 منطقة
- تخزين مؤقت تلقائي للأصول الثابتة
- تخزين مؤقت على الحافة لصفحات ISR (التوليد الثابت التدريجي)
- حماية DDoS

### رؤوس Cache-Control

تكوين التخزين المؤقت لأنواع مختلفة من المحتوى:

```typescript
// مسار API برؤوس التخزين المؤقت
export async function GET() {
  const data = await fetchData();

  return Response.json(data, {
    headers: {
      'Cache-Control': 'public, s-maxage=60, stale-while-revalidate=300',
    },
  });
}
```

### استراتيجية التخزين المؤقت حسب نوع المحتوى

| نوع المحتوى | استراتيجية التخزين المؤقت | المدة | ملاحظات |
|---|---|---|---|
| الأصول الثابتة (JS وCSS والصور) | ثابت | سنة واحدة | أسماء الملفات بتجزئة المحتوى |
| الصفحات العامة | ISR | 60–300 ثانية | إعادة التحقق عند الطلب |
| استجابات API (عامة) | `s-maxage` | 10–60 ثانية | التخزين المؤقت على مستوى CDN |
| استجابات API (مصادَقة) | `no-store` | 0 | عدم تخزين بيانات المستخدم |
| صفحات محتوى CMS | ISR | 300 ثانية | إعادة التحقق بعد مزامنة المحتوى |

### ISR (التوليد الثابت التدريجي)

استخدم ISR للصفحات الغنية بالمحتوى التي نادراً ما تتغير:

```typescript
// app/[locale]/discover/[page]/page.tsx
export const revalidate = 300; // إعادة التوليد كل 5 دقائق

export default async function DiscoverPage({ params }) {
  const items = await fetchItems(params.page);
  return <ItemGrid items={items} />;
}
```

### إعادة التحقق عند الطلب

تشغيل إعادة التحقق بعد تحديثات المحتوى:

```typescript
// app/api/revalidate/route.ts
import { revalidatePath } from 'next/cache';

export async function POST(request: Request) {
  const { secret, path } = await request.json();

  if (secret !== process.env.REVALIDATION_SECRET) {
    return Response.json({ error: 'Invalid secret' }, { status: 401 });
  }

  revalidatePath(path);
  return Response.json({ revalidated: true });
}
```

## وظائف الحافة

### متى تُستخدم وقت التشغيل على الحافة

تعمل وظائف الحافة على Cloudflare Workers (عبر Vercel) وتوفر شبه صفر من وقت البدء البارد. استخدمها من أجل:

| حالة الاستخدام | المثال |
|---|---|
| التوجيه المستند إلى الموقع الجغرافي | إعادة توجيه المستخدمين إلى المحتوى الإقليمي |
| اختبار A/B | التوجيه إلى متغيرات التجربة |
| فحوصات المصادقة | التحقق السريع من الجلسة |
| تحويل الاستجابة | إضافة رؤوس وتعديل الاستجابات |
| نقاط نهاية API بسيطة | جلب البيانات الخفيفة |

### قيود وقت التشغيل على الحافة

| القيد | التفاصيل |
|---|---|
| لا توجد Node.js APIs | لا يمكن استخدام `fs` و`child_process` وغيرها |
| لا توجد وحدات أصلية | لا يمكن استخدام `bcryptjs` و`postgres` مباشرة |
| وقت تنفيذ محدود | الحد الأقصى 30 ثانية (Vercel Pro) |
| ذاكرة محدودة | 128 ميجابايت |
| لا يوجد Drizzle ORM | استخدام عملاء قاعدة بيانات متوافقة مع الحافة |

### مثال وظيفة الحافة

```typescript
// app/api/geo/route.ts
export const runtime = 'edge';

export async function GET(request: Request) {
  const country = request.headers.get('x-vercel-ip-country') || 'US';
  const city = request.headers.get('x-vercel-ip-city') || 'Unknown';

  return Response.json({
    country,
    city,
    timestamp: Date.now(),
  });
}
```

## استراتيجيات التوسع الأفقي

### تصميم التطبيق عديم الحالة

تم تصميم القالب كعديم الحالة على مستوى التطبيق:

| المكوّن | موقع البيانات | التأثير على التوسع |
|---|---|---|
| الجلسات | قاعدة البيانات أو JWT | لا يوجد حالة مشتركة بين المثيلات |
| المهام الخلفية | مدير المهام (لكل مثيل أو Trigger.dev) | استخدام Trigger.dev للمثيلات المتعددة |
| رفع الملفات | تخزين خارجي (S3 وSupabase) | لا تعتمد على نظام الملفات المحلي |
| محتوى CMS | مستودع Git (مُستنسَخ عند البناء/التشغيل) | للقراءة فقط، متطابق لكل مثيل |
| التخزين المؤقت | في الذاكرة (لكل مثيل) أو Redis | فكر في Redis للتخزين المؤقت المشترك |

### اعتبارات تعدد المثيلات

عند تشغيل مثيلات متعددة (Docker Swarm أو Kubernetes أو وظائف Vercel متعددة):

1. **المهام الخلفية**: استخدم Trigger.dev أو Vercel Cron بدلاً من `LocalJobManager` لتجنب التنفيذ المكرر.
2. **اتصالات قاعدة البيانات**: فعّل تجميع الاتصالات لتجنب استنفاد الاتصالات.
3. **تخزين الجلسات**: استخدم جلسات مستندة إلى قاعدة البيانات بدلاً من مخازن الذاكرة.
4. **إبطال التخزين المؤقت**: نفّذ تخزيناً مؤقتاً مشتركاً (Redis) أو اقبل الاتساق النهائي مع التخزين المؤقت لكل مثيل.

## المراقبة أثناء التوسع

### المقاييس الأساسية للتتبع

| المقياس | الأداة | العتبة |
|---|---|---|
| وقت الاستجابة (p95) | Sentry وVercel Analytics | أقل من 500 ميلي ثانية |
| معدل الأخطاء | Sentry | أقل من 1% |
| اتصالات قاعدة البيانات | لوحة تحكم قاعدة البيانات | أقل من 80% من الحد الأقصى |
| عمليات البدء البارد للوظائف | Vercel Analytics | مراقبة التكرار |
| معدل الإصابة في التخزين المؤقت | سجلات التطبيق | أكثر من 80% |
| استخدام الذاكرة | مقاييس Vercel/Docker | أقل من 80% من الحد |

### مراقبة الأداء في Sentry

يُكوّن القالب Sentry مع معدل أخذ عينات التتبع:

```typescript
Sentry.init({
  tracesSampleRate: process.env.NODE_ENV === 'production' ? 0.1 : 1.0,
});
```

اضبط `tracesSampleRate` بناءً على حجم حركة المرور:

| الطلبات اليومية | معدل أخذ العينات الموصى به |
|---|---|
| أقل من 10,000 | 1.0 (100%) |
| 10,000–100,000 | 0.1 (10%) |
| 100,000–1,000,000 | 0.01 (1%) |
| أكثر من 1,000,000 | 0.001 (0.1%) |

## اختبار التحميل

### الأدوات الموصى بها

| الأداة | حالة الاستخدام | التعقيد |
|---|---|---|
| `autocannon` | معايرة HTTP السريعة | منخفض |
| `k6` | اختبار التحميل بالبرمجة النصية | متوسط |
| `Artillery` | سيناريوهات معقدة | متوسط |
| `Locust` | مستند إلى Python وموزع | مرتفع |

### مثال على اختبار التحميل

```bash
# معايرة سريعة باستخدام autocannon
npx autocannon -c 50 -d 30 https://your-app.vercel.app/api/health

# اختبار أكثر تفصيلاً باستخدام k6
k6 run load-test.js
```

### قائمة تحقق الاختبار

| الاختبار | الهدف | معيار النجاح |
|---|---|---|
| تحميل الصفحة الرئيسية | 100 مستخدم متزامن | p95 أقل من 1 ثانية |
| نقطة نهاية API | 200 طلب/ثانية | p95 أقل من 500 ميلي ثانية، 0% أخطاء |
| استعلام البحث | 50 مستخدماً متزامناً | p95 أقل من 2 ثانية |
| تدفق المصادقة | 20 مستخدماً متزامناً | جميعها ناجحة، لا توجد مهل |
