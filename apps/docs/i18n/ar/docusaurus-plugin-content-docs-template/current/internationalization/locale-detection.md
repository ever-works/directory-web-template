# اكتشاف اللغة والتوجيه

يستخدم القالب `next-intl` لاكتشاف اللغة مع مطابقة تلقائية للغة المتصفح، وتوجيه قائم على عنوان URL، وحفظ في ملفات تعريف الارتباط، ونظام رسائل احتياطية.

## عملية الاكتشاف

عند وصول طلب، تُحدَّد اللغة على النحو التالي:

1. **بادئة عنوان URL** — إن احتوى عنوان URL على بادئة لغة (مثل `/fr/about`)، تُستخدم هذه اللغة مباشرةً
2. **ملف تعريف الارتباط** — إن لم تكن ثمة بادئة URL، يتحقق النظام من ملف تعريف ارتباط اللغة الذي يُعيّنه مكوّن LanguageSwitcher
3. **رأس Accept-Language** — إن لم يوجد ملف تعريف ارتباط، يُقرأ رأس تفضيلات لغة المتصفح
4. **الاحتياطي** — إن لم يُعثر على تطابق، تُستخدم اللغة الافتراضية (`en`)

## الملفات المصدرية

| الملف | الدور في الاكتشاف |
|------|-------------------|
| `i18n/routing.ts` | يُعرِّف اللغات المدعومة واستراتيجية البادئة |
| `i18n/request.ts` | يُتحقق من اللغة المكتشفة ويحمّل الرسائل |
| `i18n/navigation.ts` | يوفر Link وrouter وredirect مع دعم اللغة |
| `lib/constants.ts` | المصدر الوحيد للحقيقة لمصفوفتي LOCALES وRTL_LOCALES |
| `components/language-switcher.tsx` | يُعيّن ملف تعريف ارتباط اللغة عبر router.replace |
| `app/[locale]/layout.tsx` | يُتحقق من اللغة ويرفض اللغات غير الصحيحة عبر notFound() |

## إعداد التوجيه

```typescript
import { defineRouting } from "next-intl/routing";
import { DEFAULT_LOCALE, LOCALES } from "@/lib/constants";

export const routing = defineRouting({
  locales: LOCALES,
  defaultLocale: DEFAULT_LOCALE,
  localeDetection: true,
  localePrefix: "as-needed",
});
```

### استراتيجية البادئة

| الطلب | اللغة المكتشفة | عنوان URL المعروض |
|---------|-----------------|-----------|
| `/about` | `en` | `/about` (بلا بادئة للغة الافتراضية) |
| `/fr/about` | `fr` | `/fr/about` (البادئة مطلوبة للغات الأخرى) |
| `/en/about` | `en` | إعادة توجيه إلى `/about` |

## منطق الرسائل الاحتياطية

- تعمل الرسائل الإنجليزية كطبقة أساسية تحتوي على جميع المفاتيح
- رسائل اللغة المحددة تُغطي المفاتيح المعرَّفة فحسب
- المفاتيح المفقودة في ملف اللغة تحتفظ بقيمتها الإنجليزية
- الكائنات المتداخلة تُدمج بشكل متكرر

## الحفاظ على ملفات تعريف الارتباط

عند اختيار المستخدم للغة عبر LanguageSwitcher، تُعيّن `next-intl` ملف تعريف ارتباط:

```typescript
const changeLanguage = useCallback(
  (locale: string) => {
    if (locale === currentLocale || isPending) return;

    startTransition(() => {
      router.replace(pathname, { locale });
    });
    setIsOpen(false);
  },
  [currentLocale, isPending, router, pathname]
);
```

## اكتشاف Accept-Language

```
Accept-Language: fr-FR,fr;q=0.9,en-US;q=0.8,en;q=0.7
```

يبحث النظام عن تطابق في مصفوفة `LOCALES` المدعومة. تفوز أول لغة مطابقة.

## استكشاف أخطاء اللغة

| المشكلة | السبب المحتمل | الحل |
|---------|-------------|----------|
| مفاتيح الترجمة تظهر بدلاً من النص | المفتاح مفقود في ملف اللغة | أضف المفتاح إلى `messages/en.json` (الاحتياطي) |
| تظهر لغة خاطئة | ملف تعريف الارتباط يتغلب على عنوان URL | امسح ملفات تعريف الارتباط أو استخدم وضع التصفح المتخفي |
| خطأ 404 في عناوين URL للغة | اللغة غير موجودة في مصفوفة LOCALES | أضف الرمز إلى `lib/constants.ts` |
| تخطيط RTL لا يُطبَّق | اللغة غير موجودة في RTL_LOCALES | أضفها إلى `RTL_LOCALES` في `lib/constants.ts` |

## أفضل الممارسات

1. **استخدام `Link` من `@/i18n/navigation`** دائماً بدلاً من `next/link`
2. **إضافة جميع المفاتيح الجديدة أولاً إلى `en.json`** إذ هي لغة الاحتياطي
3. **اختبار الاكتشاف** بضبط تفضيلات لغة المتصفح
4. **الاعتماد على احتياطي `deepmerge`** — الملفات المترجمة جزئياً متوقعة ومعالجة بشكل صحيح
