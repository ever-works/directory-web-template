# Оптимизация на производителността

Това ръководство обхваща оптимизациите на производителността, вградени в шаблона Ever Works и техниките за поддържане на бързо време за зареждане, докато вашето приложение расте.

## Конфигурация на Next.js `next.config.ts` на шаблона включва няколко настройки, фокусирани върху производителността:

### Самостоятелен изход

```typescript
const nextConfig: NextConfig = {
  output: "standalone",
  // ...
};
```

Изходният режим `standalone` създава самостоятелна компилация, която включва само файловете, необходими за стартиране на приложението. Това намалява размера на контейнера и времето за стартиране в производството.

### Оптимизация за импортиране на пакети

```typescript
experimental: {
  optimizePackageImports: ["@heroui/react", "lucide-react"],
},
```

Тази настройка позволява дървовидно разклащане за пакети с голямо количество барел-файл. Вместо да се импортира цялата библиотека `@heroui/react` или `lucide-react` , само действително използваните компоненти са включени в пакета.

### Webpack Watch Optimization

```typescript
if (dev) {
  config.watchOptions = {
    ...config.watchOptions,
    ignored: ['**/node_modules/**', '**/.git/**', '**/.content/**']
  };
}
```

Директорията `.content/` (базирана на Git CMS с 220+ файла за маркиране) е изключена от програмата за наблюдение на файлове на webpack в процес на разработка. Това предотвратява ненужните реконструкции при промяна на файлове със съдържание и значително намалява използването на процесора по време на разработката.

### Потиснати предупреждения

Подробното регистриране на инфраструктурата е потиснато в CI и Vercel среди:

```typescript
if (process.env.CI || process.env.VERCEL) {
  config.infrastructureLogging = { level: 'error' };
}
```

## Оптимизация на изображението

### Отдалечени модели

Шаблонът генерира динамично разрешени шаблони на отдалечено изображение, използвайки `generateImageRemotePatterns()` . Това гарантира, че изображенията от конфигурирани CDN и външни източници са оптимизирани чрез вградения канал за изображения на Next.js.

### Обработка на SVG

```typescript
images: {
  dangerouslyAllowSVG: true,
  contentDispositionType: 'attachment',
  contentSecurityPolicy: "default-src 'self'; script-src 'none'; sandbox;",
  unoptimized: false,
},
```

SVG изображенията са разрешени, но са защитени със стриктна политика за сигурност на съдържанието, която деактивира изпълнението на скрипт. Това позволява SVG лога и икони, като същевременно предотвратява XSS чрез инжектиране на SVG.

### Най-добри практики за изображения

| Техника | Изпълнение | Въздействие |
|---|---|---|
| Използвайте `next/image` | Вграден компонент с отложено зареждане | Автоматичен WebP/AVIF, адаптивни размери |
| Задайте изрични размери | `width` и `height` подпори | Предотвратява кумулативното изместване на оформлението (CLS) |
| Използвайте `priority` за LCP | `<Image priority />` за изображения на герои | Предварително зарежда изображение за боядисване с най-голямо съдържание |
| Използвайте `sizes` опора | `sizes="(max-width: 768px) 100vw, 50vw"` | Предотвратява изтеглянето на големи изображения |
| Замъгляване на контейнери | `placeholder="blur"` с `blurDataURL` | Подобрява възприеманата скорост на зареждане |

## Стратегии за кеширане

### HTTP заглавки

Шаблонът задава заглавки, свързани с кеша в `next.config.ts` :

```typescript
headers: [
  { key: "X-DNS-Prefetch-Control", value: "on" },
]
```

Предварителното извличане на DNS е активирано глобално, за да се намали забавянето на DNS търсене за външни ресурси.

### Статично генериране

Шаблонът използва щедро време за изчакване за генериране на статична страница:

```typescript
staticPageGenerationTimeout: 180, // 3 minutes
```

Това включва страници, които извличат данни от външни API или базираната на Git CMS по време на компилация.

### Конфигурация на ETag

```typescript
generateEtags: false,
```

ETags са деактивирани на ниво Next.js, тъй като CDN/обратният прокси (Vercel Edge Network или Cloudflare) обработва валидирането на кеша по-ефективно.

### Кеширане на ниво приложение

Аналитичният фонов процесор загрява предварително кеш паметта на редовни интервали:

| Тип на кеша | Интервал на опресняване | Данни |
|---|---|---|
| Тенденции за растеж на потребителите | 10 минути | Месечен ръст на потребителите за 6, 12, 24 месеца |
| Тенденции в дейността | 5 минути | Данни за дейността за 7, 14, 30 дневни прозорци |
| Класиране на най-добрите елементи | 15 минути | Топ 10, 20, 50 артикула |
| Скорошна дейност | 2 минути | Последни 10 и 20 записа за активност |
| Показатели за ефективност | 30 секунди | Статистика за ефективността на заявката |
| Почистване на кеша | 1 час | Премахване на изтекъл кеш запис |

## Мързеливо зареждане

### Мързеливо зареждане на ниво компонент

Използвайте `next/dynamic` за тежки компоненти, които не са необходими при първоначално рендиране:

```typescript
import dynamic from 'next/dynamic';

const HeavyChart = dynamic(() => import('@/components/charts/HeavyChart'), {
  loading: () => <ChartSkeleton />,
  ssr: false, // disable SSR for client-only components
});
```

### Разделяне на код на ниво маршрут

Next.js App Router автоматично разделя кода по маршрут. Всяка страница в `app/[locale]/` получава свой собствен пакет, така че потребителите изтеглят само необходимия JavaScript за текущата страница.

### Динамично импортиране във фонови задачи

Шаблонът използва динамично импортиране вътре в обратните извиквания на задания, за да попречи на webpack да изтегли модули само за сървър в клиентския пакет:

```typescript
manager.scheduleJob('repository-sync', 'Repository Synchronization', async () => {
  const { syncManager } = await import('@/lib/services/sync-service');
  await syncManager.performSync();
}, 5 * 60 * 1000);
```

## Оптимизация на размера на пакета

### Анализиране на пакета

Изпълнете следното, за да проверите състава на пакета:

```bash
ANALYZE=true pnpm build
```

Ако `@next/bundle-analyzer` е конфигуриран, това създава интерактивна дървовидна карта, показваща кои модули допринасят за размера на пакета.

### Често срещани техники за оптимизация

| Техника | Пример | Спестявания |
|---|---|---|
| Оптимизация на барел файл | `optimizePackageImports` в конфигурацията | Предотвратява импортирането на цели библиотеки с икони/ПИ |
| Модули само за сървър | `import 'server-only'` в lib файлове | Предотвратява случайно групиране на клиенти |
| Динамичен импорт | `await import('@/lib/services/...')` | Отлага зареждането до необходимост |
| Външни пакети | `serverExternalPackages: ['postgres', 'bcryptjs', 'drizzle-orm']` | Изключва от групирането на webpack |

Конфигурацията `serverExternalPackages` е особено важна:

```typescript
serverExternalPackages: ['postgres', 'bcryptjs', 'drizzle-orm'],
```

Тези пакети са изключени от групирането на webpack и се зареждат естествено по време на изпълнение, намалявайки времето за изграждане и избягвайки проблеми със съвместимостта с оригиналните модули.

## Съвети за оптимизация на Lighthouse

### Основни цели за уеб показатели

| Метрика | Цел | Ключови фактори |
|---|---|---|
| **LCP** (Най-голямо боядисване на съдържание) | < 2,5s | Оптимизация на изображението, приоритетно зареждане, време за отговор на сървъра |
| **FID** (закъснение при първо въвеждане) | < 100ms | Разделяне на код, минимално блокиране на главния поток |
| **CLS** (Кумулативно изместване на оформлението) | < 0,1 | Изрични размери на изображението, стратегия за зареждане на шрифт |
| **TTFB** (Време до първия байт) | < 800ms | CDN кеширане, крайни функции, оптимизация на заявки към база данни |

### Практически контролен списък

1. **Изображения**: Използвайте `next/image` с изрични `width` , `height` и `sizes` реквизити. Маркирайте изображения на видимата на екрана част от страницата с `priority` .
2. **Шрифтове**: Използвайте `next/font` за самостоятелно хостване на шрифтове с `display: swap` и предварително зареждане на критични файлове с шрифтове.
3. **JavaScript**: Прегледайте `optimizePackageImports` и добавете всички големи библиотеки, които използват барелни файлове.
4. **CSS**: Шаблонът използва CSS на Tailwind, който вече е изчистен в производствените версии. Избягвайте импортирането на неизползвани CSS модули.
5. **Скриптове на трети страни**: Отложете некритичните скриптове, като използвате `next/script` с `strategy="lazyOnload"` .
6. **Сървърни компоненти**: По подразбиране React Server Components (RSC) и използвайте само `"use client"` , където се изисква интерактивност.

### Бягащ фар

Шаблонът включва конфигурация `lighthouse-test.json` . Изпълнете автоматизирани тестове на Lighthouse:

```bash
npx lhci autorun --config=lighthouse-test.json
```

Или използвайте панела Lighthouse на Chrome DevTools за ръчни проверки.

## Ефективност на заявката за база данни

### Обединяване на връзки

Използвайте групиране на връзки, за да избегнете отварянето на нова връзка с база данни за всяка заявка. Вижте [Ръководството за мащабиране](/deployment/scaling) за подробности за конфигурацията.

### Оптимизация на заявките

- Използвайте модела на хранилището ( `lib/repositories/` ), за да централизирате и оптимизирате заявките.
- Хранилището за анализи включва вградени кеш слоеве с конфигурируем TTL.
- Наблюдавайте бавните заявки чрез фоновото задание за показатели за ефективност.

### Стратегия за индексиране

Прегледайте `lib/db/schema.ts` за съществуващи индекси. Добавяне на индекси за:
- Колони, използвани в `WHERE` клаузи
- Колони с външен ключ
- Колони, използвани в `ORDER BY` клаузи
- Композитни индекси за търсене в няколко колони

## Мониторинг на ефективността

### Интеграция на Sentry

Шаблонът интегрира Sentry за наблюдение на производителността в `instrumentation.ts` :

```typescript
Sentry.init({
  tracesSampleRate: process.env.NODE_ENV === 'production' ? 0.1 : 1.0,
});
```

Следите се вземат проби при 10% в производството и 100% в разработката. Коригирайте `tracesSampleRate` въз основа на обема на трафика и ограниченията на плана Sentry.

### Персонализирани маркери за ефективност

Използвайте API за уеб ефективност за персонализирано време:

```typescript
performance.mark('data-fetch-start');
const data = await fetchData();
performance.mark('data-fetch-end');
performance.measure('data-fetch', 'data-fetch-start', 'data-fetch-end');
```

## Резюме

| Площ | Вградена оптимизация | Допълнителни стъпки |
|---|---|---|
| Изображения | Автоматичен WebP/AVIF, SVG пясъчник | Добавете `priority` към LCP изображения, използвайте `sizes` |
| JavaScript | Оптимизация на пакети, разделяне на код | Добавете библиотеки към `optimizePackageImports` |
| Кеширане | Подгряване на фоновия кеш, предварително извличане на DNS | Конфигурирайте правилата за кеш на CDN |
| База данни | Обединяване на връзки, модел на хранилище | Добавяне на индекси, наблюдение на бавни заявки |
| Изграждане | Самостоятелен изход, външни пакети | Активиране на анализатора на пакети |
| Мониторинг | Sentry следи, показатели за работа | Настройте сигнали за влошени показатели |
