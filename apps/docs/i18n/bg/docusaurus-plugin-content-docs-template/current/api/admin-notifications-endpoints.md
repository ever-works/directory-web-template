# Крайни точки на API за администраторски известия

API за администраторски известия управлява известия в приложението за администраторски потребители. Той поддържа изброяване на известия с броя на непрочетените, създаване на нови известия за конкретни потребители и маркиране на известия като прочетени (поотделно или групово). Уведомленията се съхраняват в базата данни и са обхванати от отделни потребители.

## Резюме на маршрута

|Метод|Пътека|авт|Описание|
|--------|------|------|-------------|
|`GET`|`/api/admin/notifications`|Админ|Списък с известия за текущия администратор|
|`POST`|`/api/admin/notifications`|Удостоверен|Създайте ново известие|
|`PATCH`|`/api/admin/notifications/{id}/read`|Удостоверен|Маркирайте едно известие като прочетено|
|`PATCH`|`/api/admin/notifications/mark-all-read`|Удостоверен|Маркирайте всички известия като прочетени|

## Удостоверяване

Крайните точки за уведомяване използват две нива на удостоверяване:

**Само за администратор (GET списък):** Изисква както удостоверяване, така и роля на администратор.

```typescript
const session = await auth();
if (!session?.user?.id) {
  return NextResponse.json({ success: false, error: 'Unauthorized' }, { status: 401 });
}
if (!session.user.isAdmin) {
  return NextResponse.json({ success: false, error: 'Forbidden' }, { status: 403 });
}
```

**Удостоверен потребител (POST, PATCH):** Изисква валидна сесия, но не изисква администраторска роля. Крайните точки за маркиране като прочетени са обхванати от собствените известия на удостоверения потребител.

## Крайни точки

### ВЗЕМЕТЕ `/api/admin/notifications`

Извлича последните 50 известия за удостоверения администраторски потребител, сортирани по дата на създаване (първо най-новите). Също така връща общия брой непрочетени известия.

**Отговор (200):**

```json
{
  "success": true,
  "data": {
    "notifications": [
      {
        "id": "notif_123abc",
        "userId": "user_456def",
        "type": "item_approved",
        "title": "Item Approved",
        "message": "Your item 'Awesome Tool' has been approved and is now live.",
        "data": "{\"itemId\": \"item_789ghi\", \"itemName\": \"Awesome Tool\"}",
        "isRead": false,
        "readAt": null,
        "createdAt": "2024-01-20T10:30:00.000Z",
        "updatedAt": "2024-01-20T10:30:00.000Z"
      }
    ],
    "unreadCount": 3
  }
}
```

**Подробности за поведението:**
- На заявка се връщат максимум 50 известия
- Резултатите са подредени по `createdAt` в низходящ ред (първо най-новите)
- `unreadCount` се изчислява отделно чрез преброяване на известията, където `isRead = false`
- Известията са обхванати от ID на удостоверения потребител

### ПУБЛИКУВАНЕ `/api/admin/notifications`

Създава ново известие за определен потребител. Полето `data` приема обект, който ще бъде JSON-stringified преди съхранение. Тази крайна точка не изисква администраторски привилегии -- всеки удостоверен потребител може да създава известия (обикновено извиквани от системата вътрешно).

**Тяло на заявката:**

```json
{
  "type": "item_approved",
  "title": "Item Approved",
  "message": "Your item 'Awesome Tool' has been approved and is now live.",
  "userId": "user_456def",
  "data": {
    "itemId": "item_789ghi",
    "itemName": "Awesome Tool",
    "action": "approved"
  }
}
```

|Поле|Тип|Задължително|Описание|
|-------|------|----------|-------------|
|`type`|низ|да|Идентификатор на типа известие (напр. `"item_approved"`, `"comment_received"`)|
|`title`|низ|да|Кратко заглавие на известието|
|`message`|низ|да|Пълно съобщение за известяване|
|`userId`|низ|да|Целеви потребителски идентификатор за получаване на известието|
|`data`|обект|не|Допълнителни метаданни (JSON-stringified в хранилището)|

**Отговор (200):**

```json
{
  "success": true,
  "notification": {
    "id": "notif_123abc",
    "userId": "user_456def",
    "type": "item_approved",
    "title": "Item Approved",
    "message": "Your item 'Awesome Tool' has been approved and is now live.",
    "data": "{\"itemId\": \"item_789ghi\", \"itemName\": \"Awesome Tool\", \"action\": \"approved\"}",
    "isRead": false,
    "readAt": null,
    "createdAt": "2024-01-20T10:30:00.000Z",
    "updatedAt": "2024-01-20T10:30:00.000Z"
  }
}
```

### ПАТЧ `/api/admin/notifications/{id}/read`

Маркира конкретно известие като прочетено. Задава `isRead` на `true`, записва текущото времево клеймо в `readAt` и актуализира `updatedAt`. Само собственикът на известието може да маркира своите собствени известия -- заявката филтрира както по ИД на известие, така и по ИД на удостоверен потребител.

**Параметри на пътя:**

|Параметър|Тип|Описание|
|-----------|------|-------------|
|`id`|низ|Уникален идентификатор на известието|

**Отговор (200):**

```json
{
  "success": true,
  "notification": {
    "id": "notif_123abc",
    "userId": "user_456def",
    "type": "item_approved",
    "title": "Item Approved",
    "message": "Your item 'Awesome Tool' has been approved and is now live.",
    "isRead": true,
    "readAt": "2024-01-20T16:45:00.000Z",
    "createdAt": "2024-01-20T10:30:00.000Z",
    "updatedAt": "2024-01-20T16:45:00.000Z"
  }
}
```

### ПАТЧ `/api/admin/notifications/mark-all-read`

Маркира всички непрочетени известия за удостоверения потребител като прочетени в една групова операция. Актуализира `isRead`, `readAt` и `updatedAt` за всяко съответстващо известие. Връща броя на известията, които са били актуализирани.

**Отговор (200):**

```json
{
  "success": true,
  "updatedCount": 5
}
```

**Подробности за поведението:**
- Актуализира само известия, където `isRead = false` за текущия потребител
- `updatedCount` може да е `0`, ако няма непрочетени известия
- Всички съвпадащи известия се актуализират в една заявка към базата данни

## Модел на данни за уведомяване

|Поле|Тип|Nullable|Описание|
|-------|------|----------|-------------|
|`id`|низ|не|Уникален идентификатор за известяване|
|`userId`|низ|не|ID на потребителя, който получава известието|
|`type`|низ|не|Тип известие (напр. `"item_approved"`, `"comment_received"`)|
|`title`|низ|не|Кратко заглавие за показване|
|`message`|низ|не|Пълно съобщение за известяване|
|`data`|низ|да|JSON-stringified допълнителни метаданни|
|`isRead`|булево|не|Дали известието е прочетено|
|`readAt`|дата и час|да|Времево клеймо, когато е маркирано като прочетено|
|`createdAt`|дата и час|не|Дата на създаване|
|`updatedAt`|дата и час|да|Дата на последна актуализация|

## Кодове за грешки

|Статус|Грешка|причина|
|--------|-------|-------|
| `400` |Липсващи задължителни полета|POST липсва тип, заглавие, съобщение или потребителски идентификатор|
| `400` |Изисква се ID за известие|PATCH с празен ID параметър|
| `401` |Неразрешено|Няма активна сесия|
| `403` |Забранено|Потребител без администратор в крайна точка на списък GET|
| `404` |Известието не е намерено|Невалиден идентификатор или известието принадлежи на друг потребител|
| `500` |Вътрешна грешка на сървъра|Повреда в база данни или сървър|

## Често срещани типове известия

Полето `type` е низ в свободна форма, но приложението обикновено използва тези стойности:

|Тип|Описание|
|------|-------------|
|`item_approved`|Артикул е одобрен от администратор|
|`item_rejected`|Артикул е отхвърлен|
|`comment_received`|Беше публикуван нов коментар за елемент|
|`submission_received`|Беше получен нов артикул|
