# Действия на сървъра за бюлетин

Системата за бюлетин използва Next.js сървърни действия, а не традиционни манипулатори на API маршрути. Тези действия управляват имейл абонаменти, включително абониране, отписване и извличане на статистика. Имейл известията се изпращат както за събития за абониране, така и за отписване, като се използват конфигурируеми доставчици на имейл.

## Преглед

|Действие|авт|Описание|
|---|---|---|
|`subscribeToNewsletter`|Обществен|Абонирайте се за имейл за бюлетина|
|`unsubscribeFromNewsletter`|Обществен|Отпишете имейл от бюлетина|
|`getNewsletterStatistics`|Няма|Вземете статистически данни за абонамента|

Това са действия на сървъра, дефинирани с `'use server'` и извикани от компонентите на React чрез изпращане на формуляр или директни повиквания, а не чрез HTTP крайни точки.

## Действия на сървъра

### Абонирайте се за бюлетин

```typescript
subscribeToNewsletter(data: { email: string })
```

Абонира имейл адрес за бюлетина. Потвърждава имейла с помощта на Zod, проверява за дублиращи се активни абонаменти, създава запис в базата данни и изпраща имейл за добре дошли. Имейлът автоматично се нормализира до малки букви и се изрязва.

**Валидиране на въвеждане (Zod):**

|Поле|Тип|Задължително|Ограничения|
|---|---|---|---|
|`email`|низ|да|Трябва да е валиден имейл формат|

**Успешен отговор:**

```json
{
  "success": true
}
```

**Отговори за грешка:**

```json
{
  "error": "Email is already subscribed to the newsletter",
  "email": "user@example.com"
}
```

|Грешка|Състояние|
|---|---|
|`"Please enter a valid email address"`|Невалиден имейл формат (Zod валидиране)|
|`"Email is already subscribed to the newsletter"`|Вече съществува активен абонамент|
|`"Failed to create subscription. Please try again."`|Неуспешно вмъкване на база данни|
|`"Failed to subscribe to newsletter. Please try again."`|Неочаквана грешка|

**Стъпки на обработка:**

1. Валидиране и нормализиране на имейл (малки букви, изрязване)
2. Проверете за съществуващ активен абонамент чрез `getNewsletterSubscriptionByEmail`
3. Създаване на абонаментен запис с източник `"footer"` чрез `createNewsletterSubscription`
4. Изпратете приветстващ имейл с помощта на конфигурирания имейл доставчик (Повторно изпращане или Novu)

Грешките при изпращане на имейл се улавят тихо и не пречат на абонамента да успее.

**Източник:** `template/app/[locale]/newsletter/actions.ts`

### Отписване от бюлетина

```typescript
unsubscribeFromNewsletter(data: { email: string })
```

Прекратява абонамента за имейл от бюлетина, като зададе `isActive` на `false`. Изпраща имейл за потвърждение за отписване.

**Успешен отговор:**

```json
{
  "success": true
}
```

**Отговори за грешка:**

|Грешка|Състояние|
|---|---|
|`"Email is not subscribed to the newsletter"`|Няма намерен активен абонамент|
|`"Failed to unsubscribe. Please try again."`|Неуспешна актуализация на базата данни|

**Източник:** `template/app/[locale]/newsletter/actions.ts`

### Вземете статистика за бюлетин

```typescript
getNewsletterStatistics()
```

Връща обобщена статистика за бюлетин. Не са необходими входни параметри.

**Успешен отговор:**

```json
{
  "success": true,
  "data": {
    "totalActive": 1250,
    "recentSubscriptions": 45
  }
}
```

|Поле|Тип|Описание|
|---|---|---|
|`totalActive`|цяло число|Брой текущо активни абонаменти|
|`recentSubscriptions`|цяло число|Абонаменти, създадени през последните 30 дни|

Връща нули и за двете полета, ако заявката е неуспешна, осигурявайки елегантно деградиране.

**Източник:** `template/app/[locale]/newsletter/actions.ts`

## Заявки към бази данни

Данните за абонамент за бюлетин се управляват чрез специални функции за заявки в `lib/db/queries/newsletter.queries.ts`.

### Абонаментни операции

|функция|Описание|
|---|---|
|`createNewsletterSubscription(email, source)`|Създава нов абонаментен запис|
|`getNewsletterSubscriptionByEmail(email)`|Търси абонамент по имейл|
|`updateNewsletterSubscription(email, updates)`|Актуализира полетата за абонамент|
|`unsubscribeFromNewsletter(email)`|Задава `isActive: false` и записва `unsubscribedAt`|
|`resubscribeToNewsletter(email)`|Задава `isActive: true` и изчиства `unsubscribedAt`|
|`getNewsletterStats()`|Връща активен брой и 30-дневен абонамент|

Всички търсения на имейли нормализират въвеждането до малки букви и изрязват празното пространство преди заявка.

**Източник:** `template/lib/db/queries/newsletter.queries.ts`

## Конфигурация

Константите за конфигуриране на бюлетин са дефинирани в `lib/newsletter/config.ts`:

```
NEWSLETTER_CONFIG.DEFAULT_PROVIDER = "resend"
NEWSLETTER_CONFIG.DEFAULT_FROM = "onboarding@resend.dev"
NEWSLETTER_CONFIG.DEFAULT_COMPANY_NAME = "Ever Works"
```

### Източници на абонамент

|Източник|Описание|
|---|---|
|`footer`|Абонамент от долната форма на сайта|
|`popup`|Абонамент от изскачащ диалогов прозорец|
|`signup`|Абонамент по време на регистрация на потребител|

### Схеми за валидиране

Две Zod схеми се експортират за валидиране:

- **`emailSchema`** -- валидира и нормализира едно поле за имейл
- **`newsletterSubscriptionSchema`** -- потвърждава имейла и източника (по подразбиране `"footer"`)

### Доставчици на имейл

Системата поддържа два доставчика на имейл, конфигурирани чрез `config.yml` и променливи на средата:

|Доставчик|Променлива на средата|Описание|
|---|---|---|
|Повторно изпращане|`RESEND_API_KEY`|Доставчик на имейл по подразбиране|
|нову|`NOVU_API_KEY`|Алтернативен доставчик с поддръжка на шаблони|

Доставчикът се избира въз основа на полето `mail.provider` в `config.yml`. Конфигурацията на имейл се изгражда динамично от конфигурацията на приложението с помощта на `createEmailConfig()`.

**Източник:** `template/lib/newsletter/config.ts`

## Ключови подробности за внедряването

- **Действия на сървъра:** Това не са крайни точки на REST API. Те използват `validatedAction` обвивката от `lib/auth/middleware`, която осигурява валидиране на Zod схема преди изпълнението на действието.
- **Нормализиране на имейли:** Всички имейли се нормализират до малки букви и се изрязват както на ниво действие, така и на ниво заявка към базата данни за последователни търсения.
- **Изящни грешки в имейла:** Имейлите за приветствие и потвърждение за отписване се изпращат чрез `sendEmailSafely()`, който улавя грешките тихо. Неуспешният имейл не пречи на операцията по абониране да завърши.
- **Предотвратяване на дублиране:** Преди да създаде абонамент, системата проверява за съществуващ активен абонамент с помощта на `validateExistingSubscription()`.
- **Меко отписване:** Прекратяването на абонамента задава `isActive: false` вместо да изтрива записа, като запазва хронологията на абонамента.
