# Дневник на Промените

Тази страница обяснява как Directory Web Template управлява версиите, изданията и пътищата за надграждане.

## Семантично Версиране

Шаблонът следва [Semantic Versioning (SemVer)](https://semver.org/). Номерата на версии използват формата **MAJOR.MINOR.PATCH**:

| Компонент  | Кога да се увеличава                                            |
| ---------- | --------------------------------------------------------------- |
| **MAJOR**  | Несъвместими промени, изискващи стъпки за миграция              |
| **MINOR**  | Нови функции, добавени по обратно съвместим начин               |
| **PATCH**  | Обратно съвместими поправки на грешки и малки подобрения        |

Предварителните версии могат да използват суфикси като `-alpha.1`, `-beta.2` или `-rc.1` за ранно тестване.

## Миграции на База Данни

Шаблонът използва **Drizzle ORM** с PostgreSQL. Промените в схемата на базата данни се управляват чрез Drizzle Kit:

```bash
# Generate migration files from schema changes
pnpm db:generate

# Apply migrations to the database
pnpm db:migrate

# Open Drizzle Studio for visual database management
pnpm db:studio
```

Файловете за миграция се съхраняват в директорията `lib/db/migrations/`. Всяка миграция е SQL файл, генериран от промени в дефинициите на схемата Drizzle в `lib/db/schema/`.

## Надграждане на Шаблона

При надграждане до по-нова версия:

```bash
cd directory-web-template

# Pull latest changes
git pull origin main

# Install updated dependencies
pnpm install

# Apply database migrations
pnpm db:migrate

# Verify build
pnpm build
```

### Обработка на Конфликти при Надграждане

Ако сте персонализирали Шаблона, може да срещнете конфликти при сливане при изтегляне на актуализации. Препоръчителният подход:

1. **Запазете персонализациите в отделни файлове**, когато е възможно (персонализирани компоненти, нови маршрути, допълнителни услуги).
2. **Използвайте CMS базиран на Git** за промени в съдържанието вместо да модифицирате основните файлове.
3. **Прегледайте бележките за изданието** преди надграждане, за да разберете кои файлове са се променили.
4. **Тествайте задълбочено** след разрешаване на конфликтите, като изпълните `pnpm lint`, `pnpm tsc --noEmit` и `pnpm build`.

## Проследяване на Изданията

### GitHub Releases

Изданията се публикуват в GitHub на [github.com/ever-works/directory-web-template/releases](https://github.com/ever-works/directory-web-template/releases).

Всяко издание включва:

- Таг на версията (напр. `v0.1.0`)
- Бележки за изданието, описващи промени, нови функции, поправки на грешки и критични промени
- Връзки към съответните pull request-и и issues

### История на Комитите

Хранилището използва [Conventional Commits](https://www.conventionalcommits.org/), което улеснява прегледа на историята на комитите за промени:

```bash
# View recent commits with conventional commit prefixes
git log --oneline --since="2025-01-01"

# Filter for feature commits only
git log --oneline --grep="^feat:"

# Filter for breaking changes
git log --oneline --grep="BREAKING CHANGE"
```

## Политика за Критични Промени

Критичните промени се приемат сериозно. Проектът следва тези принципи:

1. **Предварително уведомление.** Критичните промени се обявяват поне едно минорно издание преди влизането им в сила, когато е възможно.
2. **Ръководства за миграция.** Всяка критична промяна включва ръководство за миграция в бележките за изданието.
3. **Минимизиране на смущенията.** Критичните промени се групират в мажорни издания, вместо да се разпределят в множество минорни издания.
4. **Обратна съвместимост на базата данни.** Миграциите са проектирани да бъдат неразрушителни. Добавянето на колони и създаването на таблици се предпочитат пред премахванията или преименуванията.

### Примери за Критични Промени

- Премахване или преименуване на публичен API ендпойнт
- Промяна на структурата на тела на заявки или отговори на API
- Премахване или преименуване на колони или таблици в базата данни
- Промяна на изискваните променливи на средата
- Прекратяване на поддръжката за версия на Node.js
- Промяна на поведението при удостоверяване или оторизация
- Премахване или преименуване на експортирани TypeScript типове или интерфейси

### Примери за Некритични Промени

- Добавяне на нови API ендпойнти
- Добавяне на нови незадължителни полета към тела на заявки или отговори
- Добавяне на нови колони в базата данни със стойности по подразбиране
- Добавяне на нови променливи на средата с разумни стойности по подразбиране
- Добавяне на нови функции или интеграции
- Подобрения в производителността
- Поправки на грешки

## Формат на Дневника на Промените

Бележките за изданието следват тази структура:

```markdown
## [0.2.0] - 2025-04-15

### Added

- Category-based directory filtering
- New Polar payment provider integration

### Changed

- Improved authentication flow with better error messages

### Fixed

- Resolved race condition in concurrent directory updates
- Fixed pagination offset calculation for search results

### Deprecated

- Legacy REST endpoints under /api/v1/ (use /api/v2/ instead)

### Breaking Changes

- Removed `LEGACY_AUTH_MODE` environment variable
- Renamed `DirectoryItem` type to `Item` across all APIs
```

Този формат следва конвенциите на [Keep a Changelog](https://keepachangelog.com/).
