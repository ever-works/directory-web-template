# Конфигурация за Много Наематели

Този документ обяснява как работи поддръжката за много наематели в Directory Web Template.

## Преглед

Шаблонът използва подход на **споделена база данни с изолация на ниво ред**:

- Една единствена база данни PostgreSQL обслужва множество **наематели** (уебсайтове-директории).
- Всяка таблица има колона `tenant_id`, която ограничава данните до конкретен наемател.
- Всички заявки автоматично се филтрират по текущия наемател — без изтичане на данни между наемателите.

## Бързо Конфигуриране

### 1. Задаване на Променливата на Средата

На платформата за разгръщане (Vercel, Docker и т.н.) или в `.env.local`:

```bash
TENANT_ID="your-unique-tenant-id"
```

Това може да бъде произволен уникален низ (например UUID или четим slug като `"my-directory"`).

### 2. Разгръщане

При първото стартиране приложението ще:

1. Изпълни миграции на базата данни (добавя колона `tenant_id`, ако не е налична)
2. Създаде ред на наемател, съответстващ на стойността на `TENANT_ID`
3. Мигрира съществуващите данни с NULL в `tenant_id` към вашия наемател
4. Зареди данни по подразбиране (администраторски потребител, роли, разрешения)

Не е необходим ръчен SQL — всичко е автоматично.

### 3. Проверка

Проверете сървърните журнали за:

```
[DB Init] Ensured environment tenant 'your-unique-tenant-id' exists
[Tenant Migration] ✓ users: updated 3 rows
[Tenant Migration] ✅ Migration complete: 15 total rows updated across all tables.
```

## Как Работи Разрешаването на Наемателя

Когато приложението трябва да определи текущия наемател, то използва **каскадна** стратегия:

| Приоритет | Източник           | Описание                                                           |
| --------- | ------------------ | ------------------------------------------------------------------ |
| 1         | **Сесия**          | `user.tenantId` от JWT токена (удостоверени потребители)           |
| 2         | **Проm. на средата**| Променлива на средата `TENANT_ID`                                 |
| 3         | **HTTP заглавие**  | Заглавие `x-tenant-domain` (за маршрутизиране по поддомейн)        |
| 4         | **База данни**     | Първият активен ред на наемател (краен резервен вариант)           |

Функцията `getTenantId()` от `lib/auth/tenant.ts` реализира тази верига и се извиква от всяка заявка към базата данни.

## Архитектура

### Ключови Файлове

| Файл                                     | Предназначение                                                                      |
| ---------------------------------------- | ----------------------------------------------------------------------------------- |
| `lib/auth/tenant.ts`                     | `getTenantId()` — сървърно разрешаване на наемател с кеширане                       |
| `lib/config/env.ts`                      | Валидация на променливата на средата `TENANT_ID`                                    |
| `lib/db/schema.ts`                       | Таблица на наемателите + FK `tenant_id` за всички таблици                           |
| `lib/db/initialize.ts`                   | Автоматично създава наемателя на средата + изпълнява миграция на данни при стартиране |
| `lib/db/migrate-tenant-data.ts`          | Присвоява редове с NULL в `tenant_id` на текущия наемател                           |
| `lib/auth/index.ts`                      | JWT/сесийните callbacks инжектират `tenantId`                                       |
| `components/context/tenant-provider.tsx` | React контекст за клиентски достъп до наемателя                                     |
| `app/api/tenant/route.ts`                | `GET /api/tenant` — връща информация за текущия наемател                            |

### Поток на Данните

```
Потребителска заявка → getTenantId() → Разрешава от сесия/среда/заглавия/БД
                                                 ↓
                             Всички заявки към БД се филтрират по този tenant_id
                                                 ↓
                               Връщат се само данни за този наемател
```

### Интеграция с Удостоверяването

- **Влизане с идентификационни данни**: Администраторите и клиентите получават `tenantId` от колоната `users.tenant_id`.
- **OAuth влизане**: Адаптерът Drizzle е обвит, за да инжектира `tenantId` при създаване на потребител.
- **JWT callback**: Чете `tenantId` от записа на потребителя и го вгражда в токена.
- **Сесиен callback**: Предава `tenantId` на `session.user.tenantId`.
- **Клиентски компоненти**: Използват hook-а `useTenant()` от `TenantProvider` за информация за наемателя.

## Множество Директории (Много Наематели)

За стартиране на множество уебсайтове-директории върху единствена база данни:

1. **Всеки уебсайт** задава различен `TENANT_ID` в своята среда:
    - Уебсайт A: `TENANT_ID="directory-a-uuid"`
    - Уебсайт B: `TENANT_ID="directory-b-uuid"`

2. **Всички уебсайтове** се свързват с **една и съща база данни** (`DATABASE_URL`).

3. **Изолацията на данните** е автоматична — Уебсайт A вижда само редове, при които `tenant_id = 'directory-a-uuid'`.

4. **Потребители, роли, коментари, абонаменти** и всички останали данни са напълно изолирани по наемател.

## Обработка на Съществуващите Данни

При надграждане от версия без наематели:

- Колоната `tenant_id` се добавя като **nullable** (не нарушава съществуващите данни)
- При първото стартиране `migrateNullTenantIds()` автоматично присвоява редовете с NULL на разрешения наемател
- Тази миграция е **идемпотентна** — безопасна за многократно изпълнение
- След миграцията всички съществуващи данни са видими под текущия наемател

## Маршрутизиране по Поддомейн (Разширено)

За маршрутизиране на наемател, базирано на поддомейн (напр. `tenant-a.example.com`):

1. Конфигурирайте обратния прокси да добавя заглавие `x-tenant-domain`
2. Създайте записи на наематели с полетата `domain` или `slug`:
    ```sql
    INSERT INTO tenant (id, name, domain, slug, status)
    VALUES ('uuid', 'Tenant A', 'tenant-a.example.com', 'tenant-a', 'active');
    ```
3. Стратегията `resolveFromHeaders()` ще съпостави домейна и ще разреши наемателя

## Схема на Таблицата на Наемателите

```sql
CREATE TABLE tenant (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  website TEXT,
  domain TEXT UNIQUE,
  slug TEXT UNIQUE,
  status TEXT NOT NULL DEFAULT 'active',  -- 'active' | 'inactive'
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMP NOT NULL DEFAULT NOW()
);
```
