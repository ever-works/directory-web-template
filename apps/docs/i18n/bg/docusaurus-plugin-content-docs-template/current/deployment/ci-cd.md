# CI/CD Тръбопровод

Шаблонът Ever Works включва пълен CI/CD тръбопровод, изграден с GitHub Actions. Това ръководство обхваща структурата на работните потоци, сканирането за сигурност, стратегията за защита на клоновете и потока за промотиране на внедряванията.

## Преглед на Работните Потоци

Тръбопроводът се състои от шест файла с работни потоци в `.github/workflows/`:

| Работен Поток | Файл | Тригер | Цел |
|---|---|---|---|
| CI | `ci.yml` | Push/PR към `main`, `develop` | Lint, проверка на типове, сборка |
| CodeQL | `codeql.yml` | Push/PR към `main`, `develop` + седмично разписание | Анализ на уязвимости в сигурността |
| Dev Deploy | `deploy_dev.yaml` | Push към `develop` | Внедряване в среда за предварителен преглед |
| Prod Deploy | `deploy_prod.yaml` | Push към `main` | Внедряване в производствена среда |
| Vercel Deploy | `deploy_vercel.yaml` | Извикван от dev/prod | Споделена логика за внедряване на Vercel |
| Disable CodeQL | `disable-default-codeql.yml` | Само ръчно | Инструмент за разрешаване на конфликти на CodeQL |

### Поток на Тръбопровода

```
Feature Branch --> PR to develop --> CI runs
                                     |
                                     v
                               Merge to develop --> Dev Deploy (preview)
                                     |
                                     v
                               PR to main --> CI runs
                                     |
                                     v
                               Merge to main --> Prod Deploy (production)
```

## Работен Поток CI (ci.yml)

CI се изпълнява при всеки push и pull request към `main` и `develop`. Той проверява качеството на кода и гарантира, че проектът се изгражда успешно.

### Задания

Работният поток съдържа едно задание `lint-and-build`, изпълнявано на `ubuntu-latest`:

**Стъпки**:

1. **Checkout на кода** -- Клонира хранилището
2. **Разпознаване на мениджъра на пакети** -- Автоматично разпознава pnpm, yarn или npm от lockfiles
3. **Настройка на pnpm** -- Инсталира pnpm v9 при разпознаване
4. **Настройка на Node.js** -- Инсталира Node 20 с кеш на мениджъра на пакети
5. **Инсталиране на зависимости** -- Изпълнява `pnpm install`
6. **Стартиране на lint** -- Изпълнява `pnpm lint` (продължава при грешки за PRs)
7. **Проверка на типове** -- Изпълнява `pnpm typecheck` или `pnpm check:types`
8. **Създаване на директории за съдържание** -- Създава `.content/data` за сборката
9. **Изграждане на проекта** -- Изпълнява `pnpm build` с всички необходими променливи на средата
10. **Проверка на успеха на сборката** -- Проверява дали директорията `.next` е създадена

### Управление на Паралелизма

```yaml
concurrency:
  group: ${{ github.ref }}-${{ github.workflow }}
  cancel-in-progress: true
```

Ако в същия клон се направи нов push по време на изпълнение на CI, предишното изпълнение се отменя автоматично. Това пести CI минути и гарантира, че само последният commit се валидира.

### Променливи на Средата

CI работният поток използва комбинация от твърдо кодирани стойности по подразбиране и GitHub тайни:

| Променлива | Източник | Цел |
|---|---|---|
| `NEXT_PUBLIC_APP_URL` | Твърдо кодирана | URL на приложението за сборката |
| `DATABASE_URL` | Тайна или по подразбиране | Връзка с БД за сборката |
| `AUTH_SECRET` | Твърдо кодирана CI стойност | Подписване на токени за удостоверяване (не за производство) |
| `DATA_REPOSITORY` | Тайна или по подразбиране | URL на хранилището за съдържание |
| `CONTENT_WARNINGS_SILENT` | Твърдо кодирано `true` | Заглушава предупрежденията за съдържание в CI |
| `CI` | Твърдо кодирано `true` | Указва CI среда |
| OAuth тайни | GitHub тайни | Идентификационни данни за Google, GitHub, Facebook, Twitter |
| `RESEND_API_KEY` | GitHub тайна | Имейл услуга за проверки при сборката |

### Разрешения

Работният поток изисква минимални разрешения:

```yaml
permissions:
  contents: read
```

Необходим е само достъп за четене на съдържанието на хранилището за CI заданието.

## Анализ на Сигурността CodeQL (codeql.yml)

### Какво Прави

CodeQL извършва семантичен анализ на кода за открива уязвимости в сигурността в JavaScript/TypeScript код. Изпълнява се:

- При всеки push и PR към `main` и `develop`
- Всяка седмица в понеделник в 6 AM UTC (планиран анализ)
- При ръчно стартиране

### Стъпки за Анализ

1. **Checkout** и **настройка** на Node.js + pnpm
2. **Инициализиране на CodeQL** с езика `javascript-typescript`
3. **Настройка на CodeQL среда** чрез `scripts/codeql-setup.js`
4. **Инсталиране на зависимости** за контекст на анализа
5. **Autobuild** -- Автоматично разпознаване на сборката от CodeQL
6. **Анализ с качване** -- Качва резултати в раздела Security на GitHub
7. **Резервен анализ** -- Ако качването не успее, анализ без качване

### Разрешения

CodeQL изисква по-широки разрешения за докладване на събития за сигурност:

```yaml
permissions:
  actions: read
  contents: read
  security-events: write
  pull-requests: read
```

### Преглед на Резултатите

След успешно изпълнение с качване:
1. Отидете в хранилището си в GitHub
2. Отидете в **Security** > **Code scanning**
3. Прегледайте резултатите, филтрирайте по тежест и управлявайте предупрежденията

### Разрешаване на Конфликти CodeQL

Ако срещнете конфликти при обработка на SARIF от стандартната конфигурация на CodeQL на GitHub, използвайте работния поток `disable-default-codeql.yml`:

```bash
# Trigger manually from GitHub Actions tab
# This disables the default configuration that may conflict with your custom setup
```

## Потоци на Внедряване

### Съответствие Клон-Среда

| Клон | Работен Поток | Среда | Домейн |
|---|---|---|---|
| `develop` | `deploy_dev.yaml` | `preview` | URL за предварителен преглед на Vercel |
| `main` | `deploy_prod.yaml` | `production` | Производствен домейн |

### Порта на Доставчика за Внедряване

И двата потока за внедряване проверяват хранилищна променлива преди да продължат:

```yaml
jobs:
  Vercel:
    if: ${{ vars.DEPLOY_PROVIDER == 'vercel' }}
```

Задайте `DEPLOY_PROVIDER=vercel` в **Настройки > Променливи** на хранилището, за да активирате внедрявания на Vercel. Това позволява промяна на доставчиците за внедряване без промяна на файловете с работни потоци.

### Внедряване на Vercel (deploy_vercel.yaml)

Споделеният поток за внедряване на Vercel обработва внедрявания както за предварителен преглед, така и за производство.

**Стратегия за Внедряване**: Работният поток използва двуфазен подход:

1. **Внедряване чрез API** (основно): Задейства внедряване чрез API на Vercel за по-бързи сборки
2. **Резервен CLI**: Ако API повикването не успее, преминава към `vercel build` + `vercel deploy --prebuilt`

**Стъпки**:

1. **Checkout** на кода
2. **Разпознаване на мениджъра на пакети** и настройка на pnpm
3. **Глобална инсталация на Vercel CLI**
4. **Свързване на проекта Vercel** с `VERCEL_TOKEN` и незадължителна област на екипа
5. **Задаване на променливи на средата** (DATA_REPOSITORY, GH_TOKEN, CRON_SECRET) чрез Vercel CLI
6. **Извличане на настройките на Vercel** за целевата среда
7. **Задействане на внедряване чрез API** или преминаване към CLI сборка/внедряване
8. **Актуализиране на cron разписанието** чрез `scripts/update-cron.ts`

### Необходими Тайни

Конфигурирайте ги в тайните на GitHub хранилището:

| Тайна | Необходима | Цел |
|---|---|---|
| `VERCEL_TOKEN` | Да | Удостоверяване за API на Vercel |
| `VERCEL_TEAM_SCOPE` | При използване на екипи | Slug на екипа в Vercel |
| `DATA_REPOSITORY` | Да | Наименование на хранилището за съдържание |
| `GH_TOKEN` | Да | GitHub токен за клониране на съдържание |
| `CRON_SECRET` | Препоръчително | Удостоверява повиквания към cron разточката |
| `DATABASE_URL` | За сборката | Низ за връзка с базата данни |
| OAuth тайни | При използване на OAuth | Идентификационни данни на доставчика |

### Актуализации на Cron Разписанието

След успешно внедряване работният поток изпълнява `scripts/update-cron.ts` за синхронизиране на cron разписанията:

```yaml
- name: Update cron schedule
  if: success() && steps.trigger_deployment.outputs.deployment_id != ''
  run: npx tsx scripts/update-cron.ts
```

## Правила за Защита на Клоновете

### Препоръчителни Настройки за `main`

| Настройка | Стойност | Цел |
|---|---|---|
| Изискване на pull request | Да | Без директни push към производство |
| Необходими рецензии | 1+ | Преглед на кода преди сливане |
| Изискване на проверки на статуса | CI (lint-and-build) | CI трябва да премине преди сливане |
| Изискване на CodeQL | CodeQL анализ | Проверката на сигурността трябва да премине |
| Изискване на актуални клони | Да | PR трябва да е ребейсван върху последния main |
| Включване на администратори | Да | Правилата се прилагат за всички |

### Препоръчителни Настройки за `develop`

| Настройка | Стойност | Цел |
|---|---|---|
| Изискване на pull request | По избор | Директните push са разрешени за бърза итерация |
| Необходими проверки на статуса | CI (lint-and-build) | Основна порта за качество |
| Изискване на актуални клони | Не | Позволява по-бърза итерация |

### Конфигуриране на Защита на Клоновете

1. Отидете в **Настройки** > **Клони** на хранилището
2. Кликнете **Добавяне на правило за защита на клон**
3. Въведете шаблона за имена на клона (напр. `main`)
4. Конфигурирайте настройките от таблиците по-горе
5. Запазете промените

## Поток на Промотиране

Шаблонът следва стандартен поток на промотиране:

### Цикъл на Разработка

```
1. Create feature branch from develop
2. Implement changes
3. Open PR to develop
4. CI validates (lint, type check, build)
5. CodeQL scans for vulnerabilities
6. Code review and approval
7. Merge to develop --> automatic preview deployment
```

### Производствен Релийз

```
1. Open PR from develop to main
2. CI validates against main
3. CodeQL security scan
4. Final code review
5. Merge to main --> automatic production deployment
```

### Процес на Hotfix

```
1. Create hotfix branch from main
2. Implement fix
3. Open PR directly to main
4. CI + CodeQL validation
5. Emergency review and merge
6. Backport to develop
```

## Персонализация

### Добавяне на Нови CI Стъпки

За да добавите тестове или допълнителна валидация, добавете стъпки към заданието в `ci.yml`:

```yaml
- name: Run unit tests
  run: ${{ steps.detect-pm.outputs.run-cmd }} test

- name: Run E2E tests
  run: ${{ steps.detect-pm.outputs.run-cmd }} test:e2e
```

### Добавяне на Известия за Внедряване

Добавете стъпка за известие в края на потока за внедряване:

```yaml
- name: Notify Slack
  if: success()
  uses: slackapi/slack-github-action@v1
  with:
    payload: '{"text": "Deployed to ${{ inputs.environment }}"}'
  env:
    SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK_URL }}
```

### Променливи за Конкретна Среда

Използвайте **Среди** на GitHub за ограничаване на тайни до конкретни цели за внедряване:

1. Отидете в **Настройки** > **Среди**
2. Създайте среди `production` и `preview`
3. Добавете тайни, специфични за средата
4. Референцирайте ги в работните потоци с конфигурация `environment:`
