# Мащабиране и Висока Наличност

Това ръководство обхваща стратегии за мащабиране на Ever Works Template от внедряване с единичен екземпляр до производствена конфигурация с висока наличност, включително конфигурация serverless, connection pooling, оптимизация на CDN и edge функции.

## Архитектури на Внедряване

Шаблонът поддържа множество архитектури на внедряване:

| Архитектура | Най-подходяща за | Модел на Мащабиране |
|---|---|---|
| Vercel (Serverless) | Повечето внедрявания | Автоматично хоризонтално мащабиране |
| Docker (Standalone) | Self-hosted, on-premise | Ръчно или базирано на оркестратор |
| Node.js (Директно) | Разработка, прости внедрявания | Единичен екземпляр или PM2 клъстер |

## Конфигурация Serverless (Vercel)

### Standalone Изход

Шаблонът е конфигуриран с standalone изход за оптимизирано serverless внедряване:

```typescript
// next.config.ts
const nextConfig: NextConfig = {
  output: "standalone",
};
```

Standalone режимът създава самостоятелна сборка в `.next/standalone/`, съдържаща само файловете, необходими за стартиране на приложението. Това минимизира времето за студен старт чрез намаляване на размера на пакета за внедряване.

### Конфигурация на Функциите

Конфигурирайте настройките на serverless функциите в `vercel.json` или чрез конфигурация на ниво маршрут:

```typescript
// app/api/heavy-computation/route.ts
export const maxDuration = 60; // секунди (Pro план: до 300с)
export const dynamic = 'force-dynamic';
```

### Препоръчителни Настройки на Функциите

| Тип Маршрут | Макс. Продължителност | Памет | Бележки |
|---|---|---|---|
| API маршрути (прости) | 10с | 1024 МБ | По подразбиране за повечето крайни точки |
| API маршрути (обработка на данни) | 30с | 1024 МБ | За пакетни операции |
| Cron задачи | 60с | 1024 МБ | Изпълнение на фонови задачи |
| Webhook обработчици | 30с | 1024 МБ | Обратни извиквания за плащания, OAuth |
| Статични страници | Н/П | Н/П | Предварително рендерирани при изграждане |

### Оптимизация на Студения Старт

Минимизирайте студените стартове с тези техники:

| Техника | Имплементация | Въздействие |
|---|---|---|
| Минимизиране на размера на функцията | `serverExternalPackages` в конфигурацията | Намалява времето за инициализация |
| Избягване на импорти на ниво модул | Динамичен `import()` за тежки модули | Отлага зареждането до момента на нужда |
| Използване на edge runtime където е възможно | `export const runtime = 'edge'` | Почти нулев студен старт |
| Поддържане на функциите топли | Крайни точки за проверка на здравето с мониторинг | Поддържа функциите активни |

## Connection Pooling на Базата Данни

### Проблемът

В serverless среди всяко извикване на функция може да отваря ново свързване с базата данни. Без pooling това може да изчерпи лимита на свързванията на базата данни.

### Решение: Connection Pooler

Използвайте connection pooler между приложението и базата данни:

| Pooler | Доставчик | Конфигурация |
|---|---|---|
| PgBouncer | Supabase (вграден) | Използвайте pooled низ за свързване (порт 6543) |
| Neon Pooler | Neon (вграден) | Използвайте `-pooler` низ за свързване |
| PgBouncer | Self-hosted | Внедрете PgBouncer заедно с PostgreSQL |

### Конфигурация

Използвайте различни низове за свързване за pooled и директни връзки:

```bash
# Pooled свързване за заявки на приложението (безопасно за serverless)
DATABASE_URL=postgresql://user:pass@host:6543/db?pgbouncer=true

# Директно свързване само за миграции
DIRECT_DATABASE_URL=postgresql://user:pass@host:5432/db
```

Актуализирайте `drizzle.config.ts` да използва директното свързване за миграции:

```typescript
export default {
  schema: "./lib/db/schema.ts",
  out: "./lib/db/migrations",
  dialect: "postgresql",
  dbCredentials: {
    url: process.env.DIRECT_DATABASE_URL || process.env.DATABASE_URL,
  },
} satisfies Config;
```

### Лимити на Свързванията

| Ниво | Макс. Свързвания | Препоръчителен Размер на Пула |
|---|---|---|
| Hobby (Neon/Supabase) | 50–100 | 10–20 |
| Pro (Neon/Supabase) | 200–500 | 50–100 |
| Enterprise | 1000+ | 100–200 |

### Управление на Свързванията в Кода

Модулът за базата данни на шаблона трябва да използва повторно единичен пул от свързвания на функционален екземпляр:

```typescript
// lib/db/index.ts
import { drizzle } from 'drizzle-orm/postgres-js';
import postgres from 'postgres';

// Създаване на пул от свързвания веднъж на serverless екземпляр
const connectionString = process.env.DATABASE_URL!;
const client = postgres(connectionString, {
  max: 10,          // Максимален брой свързвания в пула
  idle_timeout: 20, // Затваряне на неактивни свързвания след 20с
  connect_timeout: 10,
});

export const db = drizzle(client);
```

## CDN и Кеширане

### Мрежа Vercel Edge

При внедряване на Vercel мрежата Edge автоматично предоставя:

- Глобална CDN дистрибуция в повече от 30 региона
- Автоматично кеширане на статичните активи
- Edge кеширане за ISR страници (Incremental Static Regeneration)
- DDoS защита

### Cache-Control Заглавия

Конфигурирайте кеширането за различни типове съдържание:

```typescript
// API маршрут с кеш заглавия
export async function GET() {
  const data = await fetchData();

  return Response.json(data, {
    headers: {
      'Cache-Control': 'public, s-maxage=60, stale-while-revalidate=300',
    },
  });
}
```

### Стратегия за Кеширане по Тип Съдържание

| Тип Съдържание | Стратегия за Кеширане | TTL | Бележки |
|---|---|---|---|
| Статични активи (JS, CSS, изображения) | Immutable | 1 година | Имена на файловете с хеш на съдържанието |
| Публични страници | ISR | 60–300с | Ревалидиране при поискване |
| API отговори (публични) | `s-maxage` | 10–60с | Кеширане на ниво CDN |
| API отговори (удостоверени) | `no-store` | 0 | Никога да не се съхраняват потребителски данни |
| Страници с CMS съдържание | ISR | 300с | Ревалидиране след синхронизация на съдържанието |

### ISR (Incremental Static Regeneration)

Използвайте ISR за страници с много съдържание, което рядко се променя:

```typescript
// app/[locale]/discover/[page]/page.tsx
export const revalidate = 300; // Регенериране на всеки 5 минути

export default async function DiscoverPage({ params }) {
  const items = await fetchItems(params.page);
  return <ItemGrid items={items} />;
}
```

### Ревалидиране при Поискване

Задействайте ревалидиране след актуализации на съдържанието:

```typescript
// app/api/revalidate/route.ts
import { revalidatePath } from 'next/cache';

export async function POST(request: Request) {
  const { secret, path } = await request.json();

  if (secret !== process.env.REVALIDATION_SECRET) {
    return Response.json({ error: 'Invalid secret' }, { status: 401 });
  }

  revalidatePath(path);
  return Response.json({ revalidated: true });
}
```

## Edge Функции

### Кога да Използвате Edge Runtime

Edge функциите работят на Cloudflare Workers (чрез Vercel) и осигуряват почти нулево време за студен старт. Използвайте ги за:

| Случай на Употреба | Пример |
|---|---|
| Маршрутизация базирана на геолокация | Пренасочване на потребители към регионално съдържание |
| A/B тестване | Насочване към варианти на експеримент |
| Проверки за удостоверяване | Бърза валидация на сесията |
| Трансформация на отговора | Добавяне на заглавия, промяна на отговорите |
| Прости API крайни точки | Лека извличане на данни |

### Ограничения на Edge Runtime

| Ограничение | Детайли |
|---|---|
| Без Node.js API | Не може да се използва `fs`, `child_process` и др. |
| Без нативни модули | Не може директно да се използва `bcryptjs`, `postgres` |
| Ограничено време за изпълнение | Макс. 30 секунди (Vercel Pro) |
| Ограничена памет | 128 МБ |
| Без Drizzle ORM | Използвайте edge-съвместими клиенти за база данни |

### Пример за Edge Функция

```typescript
// app/api/geo/route.ts
export const runtime = 'edge';

export async function GET(request: Request) {
  const country = request.headers.get('x-vercel-ip-country') || 'US';
  const city = request.headers.get('x-vercel-ip-city') || 'Unknown';

  return Response.json({
    country,
    city,
    timestamp: Date.now(),
  });
}
```

## Стратегии за Хоризонтално Мащабиране

### Stateless Дизайн на Приложение

Шаблонът е проектиран като stateless на ниво приложение:

| Компонент | Местоположение на Данни | Влияние върху Мащабирането |
|---|---|---|
| Сесии | База данни или JWT | Без споделено състояние между екземплярите |
| Фонови задачи | Мениджър на задачи (на екземпляр или Trigger.dev) | Използвайте Trigger.dev за множество екземпляри |
| Качвания на файлове | Външно хранилище (S3, Supabase) | Без зависимост от локалната файлова система |
| CMS съдържание | Git хранилище (клонирано при изграждане/стартиране) | Само за четене, идентично за всеки екземпляр |
| Кеш | In-memory (на екземпляр) или Redis | Обмислете Redis за споделен кеш |

### Съображения за Множество Екземпляри

При стартиране на множество екземпляри (Docker Swarm, Kubernetes или множество Vercel функции):

1. **Фонови задачи**: Използвайте Trigger.dev или Vercel Cron вместо `LocalJobManager`, за да избегнете дублирани изпълнения.
2. **Свързвания с базата данни**: Активирайте connection pooling, за да избегнете изчерпване на свързванията.
3. **Съхранение на сесии**: Използвайте базирани на база данни сесии вместо in-memory хранилища.
4. **Инвалидиране на кеша**: Имплементирайте споделен кеш (Redis) или приемете крайна консистентност с кешове на екземпляр.

## Мониторинг при Мащабиране

### Ключови Показатели за Проследяване

| Показател | Инструмент | Праг |
|---|---|---|
| Време за отговор (p95) | Sentry, Vercel Analytics | < 500мс |
| Честота на грешки | Sentry | < 1% |
| Брой свързвания с базата данни | Табло на базата данни | < 80% от максимума |
| Студени стартове на функции | Vercel Analytics | Мониторинг на честотата |
| Честота на попаденията в кеша | Логове на приложението | > 80% |
| Използване на паметта | Vercel/Docker показатели | < 80% от лимита |

### Мониторинг на Производителността в Sentry

Шаблонът конфигурира Sentry с трасиращо вземане на проби:

```typescript
Sentry.init({
  tracesSampleRate: process.env.NODE_ENV === 'production' ? 0.1 : 1.0,
});
```

Настройте `tracesSampleRate` въз основа на обема на трафика:

| Дневни Заявки | Препоръчителна Честота на Вземане на Проби |
|---|---|
| < 10 000 | 1,0 (100%) |
| 10 000–100 000 | 0,1 (10%) |
| 100 000–1 000 000 | 0,01 (1%) |
| > 1 000 000 | 0,001 (0,1%) |

## Тестване на Натоварване

### Препоръчителни Инструменти

| Инструмент | Случай на Употреба | Сложност |
|---|---|---|
| `autocannon` | Бързи HTTP бенчмаркове | Ниска |
| `k6` | Скриптирано тестване на натоварване | Средна |
| `Artillery` | Сложни сценарии | Средна |
| `Locust` | Базиран на Python, разпределен | Висока |

### Пример за Тест на Натоварване

```bash
# Бърз бенчмарк с autocannon
npx autocannon -c 50 -d 30 https://your-app.vercel.app/api/health

# k6 скрипт за по-детайлно тестване
k6 run load-test.js
```

### Контролен Списък на Тестовете

| Тест | Цел | Критерий за Преминаване |
|---|---|---|
| Зареждане на началната страница | 100 едновременни потребители | p95 < 1с |
| API крайна точка | 200 заявки/секунда | p95 < 500мс, 0% грешки |
| Заявка за търсене | 50 едновременни потребители | p95 < 2с |
| Поток за удостоверяване | 20 едновременни потребители | Всички успешни, без timeouts |
