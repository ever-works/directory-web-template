# Internationalisierung (i18n)

Das Ever Works Template verwendet `next-intl` für die Internationalisierung und unterstützt 21 Sprachregionen mit URL-basiertem Routing, automatischer Browser-Erkennung, Nachrichten-Fallback, Rechts-nach-Links-Layouts und SEO-Hreflang-Tags. Diese Seite behandelt die i18n-Architektur, Konfiguration und Verwendungsmuster.

## Architektur

| Datei | Zweck |
|------|---------|
| `i18n/routing.ts` | Routenkonfiguration (unterstützte Sprachregionen, Präfix-Strategie) |
| `i18n/request.ts` | Anfragespezifische Sprachregions-Auflösung und Nachrichtenladung |
| `i18n/navigation.ts` | Sprachregions-bewusste Link-, Redirect-, usePathname-, useRouter-Exporte |
| `lib/constants.ts` | Sprachregionsliste, Standard-Sprachregion, RTL-Sprachregionsdefinitionen |
| `lib/seo/hreflang.ts` | SEO-Hreflang-Tag-Generierung für Suchmaschinen |
| `components/language-switcher.tsx` | Benutzeroberflächen-Sprachauswahl-Dropdown |
| `messages/*.json` | Übersetzungsdateien (eine pro Sprachregion) |
| `app/[locale]/layout.tsx` | Root-Layout mit NextIntlClientProvider |

## Routing-Konfiguration

Das Routing-Modul in `i18n/routing.ts` legt fest, wie Sprachregionen URLs zugeordnet werden:

```typescript
import { defineRouting } from "next-intl/routing";
import { DEFAULT_LOCALE, LOCALES } from "@/lib/constants";

export const routing = defineRouting({
  locales: LOCALES,
  defaultLocale: DEFAULT_LOCALE,
  localeDetection: true,
  localePrefix: "as-needed",
});
```

### Lokalpräfix-Strategie

Die `"as-needed"`-Strategie hält URLs für die Standard-Sprachregion sauber:

| URL | Aufgelöste Sprache | Erklärung |
|-----|-----------------|-------------|
| `/about` | `en` (Standard) | Kein Präfix für die Standard-Sprachregion erforderlich |
| `/fr/about` | `fr` | Präfix für Nicht-Standard-Sprachregionen erforderlich |
| `/ar/about` | `ar` | Präfix für Nicht-Standard-Sprachregionen erforderlich |

Die Standard-Sprachregion (`en`) erscheint nie in der URL. Alle anderen Sprachregionen erfordern ein explizites Präfix.

### Spracherkennung

Mit `localeDetection: true` prüft das System automatisch:

1. Ob ein Sprachregions-Cookie vorhanden ist (gesetzt durch die LanguageSwitcher-Komponente)
2. Den `Accept-Language`-HTTP-Header des Browsers
3. Fällt auf `DEFAULT_LOCALE` (`'en'`) zurück, wenn keine Übereinstimmung gefunden wird

## Nachrichtenladung und Fallback

Das Modul `i18n/request.ts` läuft bei jeder Serveranfrage, um die Sprachregion aufzulösen und Übersetzungsnachrichten zu laden:

```typescript
export default getRequestConfig(async ({ requestLocale }) => {
  let locale = await requestLocale;

  // Sprachregion gegen unterstützte Liste validieren
  if (!locale || !routing.locales.includes(locale as any)) {
    locale = routing.defaultLocale;
  }

  // Sprachregionsspezifische Nachrichten und englischen Fallback laden
  const userMessages = (await import(`../messages/${locale}.json`)).default;
  const defaultMessages = (await import(`../messages/en.json`)).default;
  // Tiefes Zusammenführen: Sprachregionsnachrichten überschreiben englische Standards
  const messages = deepmerge(defaultMessages, userMessages);

  return { locale, messages };
});
```

Die `deepmerge`-Strategie stellt sicher, dass nicht übersetzte Schlüssel den englischen Text anzeigen, anstatt rohe Schlüsselnamen anzuzeigen. Teilweise übersetzte Sprachregionsdateien funktionieren korrekt – jeder fehlende Schlüssel fällt auf sein englisches Äquivalent zurück.

### Datums- und Zahlenformate

Die Anfragekonfiguration exportiert Formatdefinitionen, die von `next-intl`-Formatierungsfunktionen verwendet werden:

```typescript
export const formats = {

  },
  number: {
    precise: { maximumFractionDigits: 5 },
  },
  list: {
    enumeration: { style: "long", type: "conjunction" },
  },

```

## Übersetzungsdateien

Übersetzungsnachrichten werden als JSON-Dateien im Verzeichnis `messages/` gespeichert, eine Datei pro Sprachregion:

```
messages/
  en.json    # Englisch (Quelldatei, alle Schlüssel vorhanden)
  fr.json    # Französisch
  es.json    # Spanisch
  de.json    # Deutsch
  ar.json    # Arabisch
  zh.json    # Chinesisch
  ... (21 Dateien insgesamt)
```

### Nachrichtenstruktur

Nachrichten verwenden eine flache Namespace-Struktur mit durch Punkte getrennten Schlüsseln:

```json
{
  "auth": {
    "SIGN_IN": "Sign In",
    "SIGN_UP": "Sign up",
    "FORGOT_PASSWORD": "Forgot password?",
    "ENTER_YOUR_EMAIL": "Enter your email"
  },
  "common": {
    "DISCOVER": "Discover",
    "HOME": "Home",
    "ABOUT": "About",
    "SAVE": "Save",
    "SHARE": "Share"
  }
}
```

### Verwendung von Übersetzungen in Komponenten

**Client-Komponenten** verwenden den `useTranslations`-Hook:

```tsx
import { useTranslations } from "next-intl";

function LoginButton() {
  const t = useTranslations("auth");
  return <button>{t("SIGN_IN")}</button>;
}
```

**Server-Komponenten** verwenden das asynchrone `getTranslations`:

```tsx
import { getTranslations } from "next-intl/server";

export default async function Page() {
  const t = await getTranslations("common");
  return <h1>{t("DISCOVER")}</h1>;
}
```

## Navigations-Hilfsmittel

Das Modul `i18n/navigation.ts` exportiert sprachregions-bewusste Ersetzungen für Standard-Next.js-Navigation:

```typescript
import { createNavigation } from "next-intl/navigation";
import { routing } from "./routing";

export const { Link, redirect, usePathname, useRouter, getPathname } =
  createNavigation(routing);
```

### Verwendung

```tsx
import { Link, useRouter, usePathname } from "@/i18n/navigation";

// Sprachregions-bewusste Link-Komponente
<Link href="/about">About</Link>
// Rendert: /about (en) oder /fr/about (fr)

// Sprachregions-bewusster Router
const router = useRouter();
router.push("/about");  // Bewahrt die aktuelle Sprachregion

// Aktueller Pfadname ohne Sprachregionspräfix
const pathname = usePathname();  // Gibt "/about" zurück, unabhängig von der Sprachregion
```

Verwenden Sie immer diese Exporte anstelle von `next/link` und `next/navigation`, um sprachregions-bewusstes Routing sicherzustellen.

## Root-Layout-Integration

Das Root-Layout in `app/[locale]/layout.tsx` validiert die Sprachregion und umhüllt den Seitenbaum mit dem i18n-Anbieter:

```typescript
export default async function RootLayout({ children, params }) {
  const { locale } = await params;

  // Ungültige Sprachregionen ablehnen
  if (!routing.locales.includes(locale as Locale)) {
    notFound();
  }

  setRequestLocale(locale);
  const messages = await getMessages();

  return (
    <>
      <NextIntlClientProvider messages={messages}>
        {/* Einstellungen, Theme und andere Anbieter */}
        <Providers config={config}>
          {children}
        </Providers>
      </NextIntlClientProvider>
    </>
  );
}
```

Das Layout generiert auch statische Parameter für die Standard-Sprachregion, während andere Sprachregionen bei Bedarf gerendert werden:

```typescript
export async function generateStaticParams() {
  return [{ locale: 'en' }];
}
```

## SEO: Hreflang-Tags

Das Modul `lib/seo/hreflang.ts` generiert Hreflang-Alternativ-URLs für Suchmaschinen:

```typescript
export function generateHreflangAlternates(
  path: string
): Record<string, string> {
  const languages: Record<string, string> = {};

  for (const locale of LOCALES) {
    const hreflang = LOCALE_TO_HREFLANG[locale];
    languages[hreflang] = getLocalizedUrl(path, locale);
  }

  // x-default zeigt auf die Standard-Sprachregionsversion
  languages['x-default'] = getLocalizedUrl(path, DEFAULT_LOCALE);
  return languages;
}
```

Verwendung in Seiten-Metadaten:

```typescript
export async function generateMetadata({ params }) {
  const { locale } = await params;
  return {
    alternates: {
      canonical: locale === DEFAULT_LOCALE ? '/' : `/${locale}`,
      languages: generateHreflangAlternates('/')
    }
  };
}
```

Dies gibt Hreflang-Tags für alle 21 Sprachregionen plus einen `x-default`-Eintrag aus und hilft Suchmaschinen dabei, Nutzern die korrekte Sprachversion bereitzustellen.

## LanguageSwitcher-Komponente

Die `LanguageSwitcher`-Komponente in `components/language-switcher.tsx` bietet ein Dropdown zum Ändern der Sprachregion:

```tsx
import { LanguageSwitcher } from "@/components/language-switcher";

// Standardgröße mit Flagge + Bezeichnung
<LanguageSwitcher />

// Kompakter Modus (nur Flagge)
<LanguageSwitcher compact />
```

### Funktionsweise

1. Zeigt die aktuelle Sprachregions-Flagge und den zweistelligen Code in einem Button
2. Öffnet ein scrollbares Dropdown mit allen 21 Sprachregionen und Flaggensymbolen
3. Beim Auswählen wird `router.replace(pathname, { locale })` über `useTransition` aufgerufen
4. `next-intl` übernimmt Cookie-Persistenz und URL-Umschreibung automatisch
5. Die aktive Sprachregion wird mit der primären Theme-Farbe hervorgehoben
6. Escape-Taste und Klick außerhalb schließen das Dropdown

### Barrierefreiheit

Die Komponente enthält `aria-label`-, `aria-expanded`- und `aria-controls`-Attribute. Buttons werden während Sprachregions-Übergängen deaktiviert, um Doppelklicks zu verhindern.

## Neue Sprache hinzufügen

1. Den Sprachregionscode zum `LOCALES`-Array in `lib/constants.ts` hinzufügen
2. Falls die Sprachregion RTL ist, zu `RTL_LOCALES` hinzufügen
3. `messages/<code>.json` mit übersetzten Schlüsseln erstellen (fehlende Schlüssel fallen auf Englisch zurück)
4. Eine Flaggen-SVG zu `public/flags/<code>.svg` hinzufügen
5. Den Sprachregionseintrag zur `languageMap` in `components/language-switcher.tsx` hinzufügen
6. Die Hreflang-Zuordnung in `lib/seo/hreflang.ts` hinzufügen

## Best Practices

1. **`Link` aus `@/i18n/navigation` verwenden** anstelle von `next/link`, um sprachregions-bewusstes Routing sicherzustellen
2. **Neue Schlüssel zuerst zu `en.json` hinzufügen**, da es die Fallback-Sprache für alle Sprachregionen ist
3. **Englische Strings niemals fest codieren** in Komponenten – `useTranslations` oder `getTranslations` verwenden
4. **Auf den `deepmerge`-Fallback vertrauen** – teilweise übersetzte Sprachregionsdateien funktionieren korrekt
5. **RTL-Layouts testen**, indem auf Arabisch oder Hebräisch im LanguageSwitcher gewechselt wird
6. **`localePrefix: "as-needed"` verwenden**, um Standard-Sprachregions-URLs sauber zu halten und gleichzeitig SEO-Präfixe für andere Sprachregionen zu unterstützen
