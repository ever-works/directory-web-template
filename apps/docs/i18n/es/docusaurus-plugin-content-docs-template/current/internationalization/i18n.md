# Internacionalización (i18n)

Ever Works Template usa `next-intl` para la internacionalización, soportando 21 locales con enrutamiento basado en URL, detección automática del navegador, fallback de mensajes, layouts de derecha a izquierda y etiquetas SEO hreflang. Esta página documenta la arquitectura i18n, la configuración y los patrones de uso.

## Arquitectura

| Archivo | Propósito |
|------|---------|
| `i18n/routing.ts` | Configuración de rutas (locales soportados, estrategia de prefijo) |
| `i18n/request.ts` | Resolución de locale y carga de mensajes por solicitud |
| `i18n/navigation.ts` | Link, redirect, usePathname, useRouter compatibles con locale |
| `lib/constants.ts` | Lista de locales, locale predeterminado, definiciones RTL |
| `lib/seo/hreflang.ts` | Generación de etiquetas SEO hreflang para motores de búsqueda |
| `components/language-switcher.tsx` | Selector de idioma en la interfaz de usuario |
| `messages/*.json` | Archivos de traducción (uno por locale) |
| `app/[locale]/layout.tsx` | Layout raíz con NextIntlClientProvider |

## Locales Soportados

La plantilla define 21 locales en `lib/constants.ts`:

```typescript
export const DEFAULT_LOCALE = 'en';
export const LOCALES = [
  'en', 'fr', 'es', 'de', 'zh', 'ar', 'he',
  'ru', 'uk', 'pt', 'it', 'ja', 'ko', 'nl',
  'pl', 'tr', 'vi', 'th', 'hi', 'id', 'bg'
] as const;

export type Locale = (typeof LOCALES)[number];

/** Locales que usan dirección de texto de derecha a izquierda */
export const RTL_LOCALES: readonly Locale[] = ['ar', 'he'] as const;
```

| Código | Idioma | Dirección |
|------|----------|-----------|
| `en` | Inglés | LTR |
| `fr` | Francés | LTR |
| `es` | Español | LTR |
| `de` | Alemán | LTR |
| `zh` | Chino simplificado | LTR |
| `ar` | Árabe | **RTL** |
| `he` | Hebreo | **RTL** |
| `it` | Italiano | LTR |
| `pt` | Portugués | LTR |
| `nl` | Neerlandés | LTR |
| `pl` | Polaco | LTR |
| `ru` | Ruso | LTR |

## Configuración de Enrutamiento

El módulo de enrutamiento en `i18n/routing.ts` define cómo los locales se mapean a URLs:

```typescript
import { defineRouting } from "next-intl/routing";
import { DEFAULT_LOCALE, LOCALES } from "@/lib/constants";

export const routing = defineRouting({
  locales: LOCALES,
  defaultLocale: DEFAULT_LOCALE,
  localeDetection: true,
  localePrefix: "as-needed",
});
```

### Estrategia de Prefijo de Locale

La estrategia `"as-needed"` mantiene URLs limpias para el locale predeterminado:

| URL | Locale Resuelto | Explicación |
|-----|-----------------|-------------|
| `/about` | `en` (predeterminado) | Sin prefijo para el locale predeterminado |
| `/fr/about` | `fr` | Prefijo requerido para locales no predeterminados |
| `/ar/about` | `ar` | Prefijo requerido para locales no predeterminados |

### Detección de Locale

Con `localeDetection: true`, el sistema verifica automáticamente:

1. Una cookie de locale (establecida por el componente LanguageSwitcher)
2. El encabezado HTTP `Accept-Language` del navegador
3. Regresa al `DEFAULT_LOCALE` (`'en'`) si no se encuentra coincidencia

## Carga de Mensajes y Fallback

El módulo `i18n/request.ts` se ejecuta en cada solicitud del servidor:

```typescript
export default getRequestConfig(async ({ requestLocale }) => {
  let locale = await requestLocale;

  if (!locale || !routing.locales.includes(locale as any)) {
    locale = routing.defaultLocale;
  }

  const userMessages = (await import(`../messages/${locale}.json`)).default;
  const defaultMessages = (await import(`../messages/en.json`)).default;
  const messages = deepmerge(defaultMessages, userMessages);

  return { locale, messages };
});
```

La estrategia `deepmerge` garantiza que las claves no traducidas muestren texto en inglés.

## Archivos de Traducción

Los mensajes de traducción se almacenan como archivos JSON en el directorio `messages/`:

```
messages/
  en.json    # Inglés (fuente, todas las claves presentes)
  fr.json    # Francés
  es.json    # Español
  ... (21 archivos en total)
```

### Uso de Traducciones en Componentes

**Componentes cliente** usan el hook `useTranslations`:

```tsx
import { useTranslations } from "next-intl";

function LoginButton() {
  const t = useTranslations("auth");
  return <button>{t("SIGN_IN")}</button>;
}
```

**Componentes servidor** usan el `getTranslations` asíncrono:

```tsx
import { getTranslations } from "next-intl/server";

export default async function Page() {
  const t = await getTranslations("common");
  return <h1>{t("DISCOVER")}</h1>;
}
```

## Utilidades de Navegación

El módulo `i18n/navigation.ts` exporta sustitutos compatibles con locale:

```typescript
import { createNavigation } from "next-intl/navigation";
import { routing } from "./routing";

export const { Link, redirect, usePathname, useRouter, getPathname } =
  createNavigation(routing);
```

## Agregar un Nuevo Locale

1. Añadir el código de locale al array `LOCALES` en `lib/constants.ts`
2. Si el locale es RTL, añadirlo a `RTL_LOCALES`
3. Crear `messages/<código>.json` con las claves traducidas
4. Añadir un SVG de bandera en `public/flags/<código>.svg`
5. Añadir la entrada de locale al `languageMap` en `components/language-switcher.tsx`
6. Añadir el mapeo hreflang en `lib/seo/hreflang.ts`

## Mejores Prácticas

1. **Usar `Link` de `@/i18n/navigation`** en lugar de `next/link`
2. **Añadir nuevas claves primero en `en.json`** ya que es el idioma de fallback
3. **Nunca codificar cadenas en inglés** en componentes
4. **Confiar en el fallback `deepmerge`** — los archivos parcialmente traducidos funcionan correctamente
5. **Probar layouts RTL** cambiando al árabe o hebreo
