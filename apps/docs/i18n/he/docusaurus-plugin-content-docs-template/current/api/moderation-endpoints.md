# מערכת ניהול

מערכת הניהול מספקת ניהול תוכן פרוגרמטי דרך שכבת שירות ולא נקודות קצה עצמאיות של API. פעולות ניהול מופעלות באופן אוטומטי כאשר מנהלי מערכת פותרים דוחות תוכן באמצעות ה-API של דוחות. המערכת תומכת באזהרה של משתמשים, השעיית חשבונות, איסור חשבונות והסרת תוכן, עם היסטוריית ביקורת מלאה והתראות באימייל.

## סקירה כללית

מתינות אינה נחשפת כנקודות קצה נפרדות של REST. במקום זאת, הוא מופעל באמצעות זרימת העבודה של פתרון הדוח:

```
PUT /api/admin/reports/[id]  -->  resolution triggers moderation action
```

כאשר מנהל מגדיר ערך `resolution` בדוח, פונקציית הניהול המתאימה מופעלת באופן אוטומטי.

|ערך רזולוציה|פונקציית מתן|אפקט|
|---|---|---|
|`content_removed`|`removeContent()`|מוחק רך את ההערה או הפריט שדווחו|
|`user_warned`|`warnUser()`|מגדיל את ספירת האזהרות של המשתמש|
|`user_suspended`|`suspendUser()`|מגדיר את סטטוס המשתמש ל-`"suspended"`|
|`user_banned`|`banUser()`|מגדיר את סטטוס המשתמש ל-`"banned"`|
|`no_action`|אין|לא ננקטה פעולות מתינות|

## פעולות מתן

### הסר תוכן

```typescript
removeContent(contentType, contentId, reportId, adminId): Promise<ModerationResult>
```

מסיר את התוכן המדווח על סמך סוגו. עבור הערות, זה מבצע מחיקה רכה (מגדיר `deletedAt`). עבור פריטים, הדבר מוחק את הפריט ממאגר התוכן מבוסס Git.

**פרמטרים:**

|פרמטר|הקלד|תיאור|
|---|---|---|
|`contentType`|`"item"` או `"comment"`|סוג התוכן להסרה|
|`contentId`|מחרוזת|מזהה או שבלול של התוכן|
|`reportId`|מחרוזת|מזהה דוח משויך|
|`adminId`|מחרוזת|משתמש מנהל מבצע את הפעולה|

**שלבי עיבוד:**

1. חפש את בעל התוכן באמצעות `getContentOwner()`
2. אם הערה: soft-delete דרך `deleteComment()`
3. אם פריט: מחק ממאגר Git דרך `itemRepository.delete()`
4. רישום היסטוריית ניהול עם פעולה `CONTENT_REMOVED`
5. שלח אימייל התראה על הסרת תוכן לבעל התוכן

**מקור:** `template/lib/services/moderation.service.ts`

### הזהר משתמש

```typescript
warnUser(userId, reason, reportId, adminId): Promise<ModerationResult>
```

מוציא אזהרה למשתמש על ידי הגדלת השדה `warningCount` שלו. משתמשים שכבר חסומים אינם יכולים לקבל אזהרות.

**פרמטרים:**

|פרמטר|הקלד|תיאור|
|---|---|---|
|`userId`|מחרוזת|מזהה פרופיל לקוח של המשתמש|
|`reason`|מחרוזת|סיבת האזהרה|
|`reportId`|מחרוזת|מזהה דוח משויך|
|`adminId`|מחרוזת|משתמש מנהל מבצע את הפעולה|

**שלבי עיבוד:**

1. ודא שהמשתמש קיים ואינו כבר חסום
2. הגדל את ספירת האזהרות באמצעות `incrementWarningCount()`
3. רישום היסטוריית ניהול עם פעולה `WARN`
4. שלח הודעת אזהרה באימייל עם ספירת האזהרות הנוכחית

**תוצאת הצלחה:**

```json
{
  "success": true,
  "message": "User warned successfully. Total warnings: 3"
}
```

**מקור:** `template/lib/services/moderation.service.ts`

### השעיית משתמש

```typescript
suspendUser(userId, reason, reportId, adminId): Promise<ModerationResult>
```

משעה חשבון משתמש על ידי הגדרת הסטטוס שלו ל-`"suspended"` והקלטת חותמת זמן `suspendedAt`. משתמשים מושעים אינם יכולים ליצור הערות, להגיש הצבעות או להגיש דוחות.

**שומרים:**

- מחזיר שגיאה אם המשתמש כבר מושעה
- מחזיר שגיאה אם המשתמש כבר חסום

**שלבי עיבוד:**

1. ודא שהמשתמש קיים ואינו כבר מושעה או חסום
2. הגדר את המצב ל-`"suspended"` עם חותמת הזמן `suspendedAt`
3. רישום היסטוריית ניהול עם פעולה `SUSPEND`
4. שלח הודעת אימייל על השעיה

**מקור:** `template/lib/services/moderation.service.ts`

### חסום משתמש

```typescript
banUser(userId, reason, reportId, adminId): Promise<ModerationResult>
```

אוסר לצמיתות חשבון משתמש על ידי הגדרת הסטטוס שלו ל-`"banned"` והקלטת חותמת זמן `bannedAt`. משתמשים חסומים חסומים בפני כל הפעולות המאומתות.

**שומרים:**

- מחזיר שגיאה אם המשתמש כבר חסום

**שלבי עיבוד:**

1. ודא שהמשתמש קיים ואינו כבר חסום
2. הגדר את המצב ל-`"banned"` עם חותמת הזמן `bannedAt`
3. רישום היסטוריית ניהול עם פעולה `BAN`
4. שלח הודעת אימייל

**מקור:** `template/lib/services/moderation.service.ts`

## החלטה של בעל תוכן

הפונקציה `getContentOwner()` קובעת מי הבעלים של התוכן המדווח:

|סוג תוכן|מקור הבעלים|
|---|---|
|`comment`|שדה `comment.userId` מטבלת ההערות|
|`item`|שדה `item.submitted_by` ממאגר הפריטים|

זה משמש את כל פעולות הניהול ברמת המשתמש (`user_warned`, `user_suspended`, `user_banned`) כדי לזהות את משתמש היעד של הפעולה.

**מקור:** `template/lib/services/moderation.service.ts`

## היסטוריית ניהול

כל פעולות הניהול יוצרות שובל ביקורת בטבלת מסד הנתונים `moderationHistory`.

### שדות תיעוד היסטוריה

|שדה|הקלד|תיאור|
|---|---|---|
|`id`|מחרוזת|מזהה רשומה ייחודי|
|`userId`|מחרוזת|מזהה פרופיל לקוח של המשתמש המושפע|
|`action`|מחרוזת|`"CONTENT_REMOVED"`, `"WARN"`, `"SUSPEND"`, או `"BAN"`|
|`reason`|מחרוזת או ריק|הסיבה לפעולת הממתן|
|`reportId`|מחרוזת או ריק|מזהה דוח משויך|
|`performedBy`|מחרוזת או ריק|מזהה משתמש מנהל שביצע את הפעולה|
|`contentType`|מחרוזת או ריק|`"item"` או `"comment"` (להסרת תוכן)|
|`contentId`|מחרוזת או ריק|מזהה התוכן שהוסר|
|`details`|אובייקט או ריק|הקשר נוסף (למשל, ספירת אזהרות, שם פריט)|
|`createdAt`|חותמת זמן|מתי בוצעה הפעולה|

### שאילתות היסטוריה

|פונקציה|תיאור|
|---|---|
|`getModerationHistoryByUser(userId, limit)`|קבל את כל פעולות הניהול עבור משתמש (מגבלת ברירת מחדל: 50)|
|`getModerationHistoryByReport(reportId)`|קבל פעולות ניהול המקושרות לדוח ספציפי|

שתי פונקציות השאילתה מעשירות את התוצאות במידע על פרופיל המשתמש ובפרטי המבצע של המנהל.

**מקור:** `template/lib/db/queries/moderation.queries.ts`

## ניהול סטטוס משתמש

### ערכי סטטוס

|סטטוס|תיאור|
|---|---|
|`active`|חשבון רגיל, כל התכונות זמינות|
|`suspended`|מוגבל באופן זמני, לא יכול ליצור תוכן|
|`banned`|מוגבל לצמיתות, חסום מכל הפעולות|

### פעולות מסד נתונים

|פונקציה|תיאור|
|---|---|
|`suspendUser(userId)`|מגדיר את המצב ל-`"suspended"`, מתעד `suspendedAt`|
|`unsuspendUser(userId)`|משחזר את המצב ל-`"active"`, מנקה את `suspendedAt`|
|`banUser(userId)`|מגדיר את המצב ל-`"banned"`, מתעד `bannedAt`|
|`unbanUser(userId)`|משחזר את המצב ל-`"active"`, מנקה את `bannedAt`|
|`incrementWarningCount(userId)`|מגדילים `warningCount` באמצעות SQL `COALESCE`|

### בדיקות משתמש חסומות

שתי פונקציות עוזר בודקות את מצב המשתמש בכל האפליקציה:

- **`isUserBlocked(status)`** -- מחזירה `true` אם המצב הוא `"suspended"` או `"banned"`
- **`getBlockReasonMessage(status)`** -- מחזירה הודעה הפונה למשתמש המסבירה מדוע הפעולה מוגבלת

בדיקות אלו משמשות את נקודות הקצה של ההערות, ההצבעות והדיווחים כדי למנוע ממשתמשים חסומים ליצור תוכן.

**מקור:** `template/lib/db/queries/moderation.queries.ts`

## הודעות אימייל

ה-`EmailNotificationService` שולח התראות לא חוסמות עבור פעולות ניהול:

|שיטה|טריגר|
|---|---|
|`sendContentRemovedEmail(email, type, reason)`|תוכן הוסר על ידי אדמין|
|`sendUserWarningEmail(email, reason, count)`|הוצאה אזהרה|
|`sendUserSuspensionEmail(email, reason)`|החשבון הושעה|
|`sendUserBanEmail(email, reason)`|החשבון נאסר|

כל שליחת הדואר האלקטרוני השתמש ב-`.catch()` כדי למנוע מכשלים להפריע לזרימת העבודה של הניהול. הודעת אימייל שנכשלה אינה גורמת לפעולת הניהול עצמה להיכשל.

## פרטי יישום מרכזיים

- **דפוס שכבת שירות:** לוגיקה של ניהול חי ב-`lib/services/moderation.service.ts`, לא במטפלי נתיב API. זה מאפשר שימוש חוזר בנקודות כניסה שונות.
- **מסלול הביקורת:** כל פעולת ניהול יוצרת רשומה `moderationHistory`, המספקת יומן ביקורת שלם לציות ובדיקה.
- **הודעות דוא"ל לא חוסמות:** הודעות דוא"ל נשלחות באופן אסינכרוני עם מטפלי `.catch()`. אם שירות הדוא"ל אינו זמין, פעולת הניהול עדיין מצליחה.
- **משמרות האידמפוטנציה:** כל פעולה בודקת את סטטוס המשתמש הנוכחי לפני שתמשיך. איסור על משתמש שכבר נחסם מחזיר שגיאה במקום יצירת פעולה כפולה.
- **מחיקה רכה לעומת מחיקה קשה:** הערות נמחקות ברכות (הגדרה `deletedAt`), בעוד פריטים מוסרים במלואם ממאגר Git. הבדל זה משקף את מודל האחסון (בסיס נתונים לעומת תוכן מבוסס קבצים).
