# מדרגיות וזמינות גבוהה

מדריך זה מכסה אסטרטגיות להרחבת Ever Works Template מפריסת מופע יחיד לתצורת ייצור בעלת זמינות גבוהה, כולל תצורת serverless, מאגר חיבורים, אופטימיזציית CDN ופונקציות קצה.

## ארכיטקטורות פריסה

התבנית תומכת בארכיטקטורות פריסה מרובות:

| ארכיטקטורה | מתאים ביותר ל | מודל הרחבה |
|---|---|---|
| Vercel (Serverless) | רוב הפריסות | הרחבה אופקית אוטומטית |
| Docker (עצמאי) | אחסון עצמאי, מקומי | ידני או מבוסס מתזמר |
| Node.js (ישיר) | פיתוח, פריסות פשוטות | מופע יחיד או אשכול PM2 |

## תצורת Serverless (Vercel)

### פלט עצמאי

התבנית מוגדרת עם פלט עצמאי לפריסת serverless מאופטמת:

```typescript
// next.config.ts
const nextConfig: NextConfig = {
  output: "standalone",
};
```

מצב עצמאי יוצר בנייה עצמאית ב-`.next/standalone/` המכילה רק את הקבצים הדרושים להרצת האפליקציה. מצמצם זמני הפעלה קרה על ידי הקטנת גודל חבילת הפריסה.

### תצורת פונקציות

הגדר הגדרות פונקציות serverless ב-`vercel.json` או דרך תצורה ברמת מסלול:

```typescript
// app/api/heavy-computation/route.ts
export const maxDuration = 60; // שניות (תוכנית Pro: עד 300 שניות)
export const dynamic = 'force-dynamic';
```

### הגדרות פונקציות מומלצות

| סוג מסלול | משך מקסימלי | זיכרון | הערות |
|---|---|---|---|
| מסלולי API (פשוטים) | 10 שניות | 1024 MB | ברירת מחדל לרוב נקודות הקצה |
| מסלולי API (עיבוד נתונים) | 30 שניות | 1024 MB | לפעולות אצווה |
| משימות Cron | 60 שניות | 1024 MB | ביצוע משימות ברקע |
| מטפלי Webhook | 30 שניות | 1024 MB | Callbacks לתשלומים ו-OAuth |
| דפים סטטיים | לא רלוונטי | לא רלוונטי | מרונדרים מראש בעת בנייה |

### אופטימיזציית הפעלה קרה

מזעור הפעלות קרות עם טכניקות אלו:

| טכניקה | יישום | השפעה |
|---|---|---|
| מזעור גודל פונקציה | `serverExternalPackages` בתצורה | מקצר זמן אתחול |
| הימנעות מייבוא ברמת מודול | `import()` דינמי למודולים כבדים | דוחה טעינה עד לצורך |
| שימוש ב-edge runtime בכל מקום אפשרי | `export const runtime = 'edge'` | כמעט אפס הפעלה קרה |
| שמירת פונקציות חמות | נקודות קצה לבדיקת בריאות עם ניטור | שומר פונקציות פעילות |

## מאגר חיבורים לבסיס נתונים

### הבעיה

בסביבות serverless, כל קריאת פונקציה עשויה לפתוח חיבור חדש לבסיס הנתונים. ללא מאגר חיבורים, זה עלול לרוקן את מגבלת החיבורים של בסיס הנתונים.

### פתרון: מאגר חיבורים

השתמש במאגר חיבורים בין האפליקציה לבסיס הנתונים:

| מאגר | ספק | תצורה |
|---|---|---|
| PgBouncer | Supabase (מובנה) | שימוש במחרוזת חיבור מאוגמת (פורט 6543) |
| Neon Pooler | Neon (מובנה) | שימוש במחרוזת חיבור `-pooler` |
| PgBouncer | אחסון עצמאי | פריסת PgBouncer לצד PostgreSQL |

### תצורה

השתמש במחרוזות חיבור שונות לחיבורים מאוגמים וישירים:

```bash
# חיבור מאוגם לשאילתות אפליקציה (בטוח ל-serverless)
DATABASE_URL=postgresql://user:pass@host:6543/db?pgbouncer=true

# חיבור ישיר למיגרציות בלבד
DIRECT_DATABASE_URL=postgresql://user:pass@host:5432/db
```

עדכן `drizzle.config.ts` להשתמש בחיבור הישיר למיגרציות:

```typescript
export default {
  schema: "./lib/db/schema.ts",
  out: "./lib/db/migrations",
  dialect: "postgresql",
  dbCredentials: {
    url: process.env.DIRECT_DATABASE_URL || process.env.DATABASE_URL,
  },
} satisfies Config;
```

### מגבלות חיבור

| רמה | חיבורים מקסימליים | גודל מאגר מומלץ |
|---|---|---|
| Hobby (Neon/Supabase) | 50–100 | 10–20 |
| Pro (Neon/Supabase) | 200–500 | 50–100 |
| Enterprise | 1000+ | 100–200 |

### ניהול חיבורים בקוד

מודול בסיס הנתונים של התבנית צריך לשאר מאגר חיבורים יחיד לכל מופע פונקציה:

```typescript
// lib/db/index.ts
import { drizzle } from 'drizzle-orm/postgres-js';
import postgres from 'postgres';

// יצירת מאגר חיבורים פעם אחת לכל מופע serverless
const connectionString = process.env.DATABASE_URL!;
const client = postgres(connectionString, {
  max: 10,          // מספר מקסימלי של חיבורים במאגר
  idle_timeout: 20, // סגירת חיבורים לא פעילים אחרי 20 שניות
  connect_timeout: 10,
});

export const db = drizzle(client);
```

## CDN ומטמון

### רשת Vercel Edge

בפריסה על Vercel, רשת Edge מספקת אוטומטית:

- הפצת CDN גלובלית ביותר מ-30 אזורים
- מטמון אוטומטי לנכסים סטטיים
- מטמון Edge לדפי ISR (יצירה סטטית מדורגת)
- הגנת DDoS

### כותרות Cache-Control

הגדר מטמון לסוגי תוכן שונים:

```typescript
// מסלול API עם כותרות מטמון
export async function GET() {
  const data = await fetchData();

  return Response.json(data, {
    headers: {
      'Cache-Control': 'public, s-maxage=60, stale-while-revalidate=300',
    },
  });
}
```

### אסטרטגיית מטמון לפי סוג תוכן

| סוג תוכן | אסטרטגיית מטמון | TTL | הערות |
|---|---|---|---|
| נכסים סטטיים (JS, CSS, תמונות) | בלתי ניתן לשינוי | שנה אחת | שמות קבצים עם hash תוכן |
| דפים ציבוריים | ISR | 60–300 שניות | אימות מחדש לפי דרישה |
| תגובות API (ציבורי) | `s-maxage` | 10–60 שניות | מטמון ברמת CDN |
| תגובות API (מאומת) | `no-store` | 0 | לעולם לא לאחסן נתוני משתמש |
| דפי תוכן CMS | ISR | 300 שניות | אימות מחדש אחרי סנכרון תוכן |

### ISR (יצירה סטטית מדורגת)

השתמש ב-ISR לדפים עשירי תוכן שמשתנים לעיתים נדירות:

```typescript
// app/[locale]/discover/[page]/page.tsx
export const revalidate = 300; // יצירה מחדש כל 5 דקות

export default async function DiscoverPage({ params }) {
  const items = await fetchItems(params.page);
  return <ItemGrid items={items} />;
}
```

### אימות מחדש לפי דרישה

הפעל אימות מחדש לאחר עדכוני תוכן:

```typescript
// app/api/revalidate/route.ts
import { revalidatePath } from 'next/cache';

export async function POST(request: Request) {
  const { secret, path } = await request.json();

  if (secret !== process.env.REVALIDATION_SECRET) {
    return Response.json({ error: 'Invalid secret' }, { status: 401 });
  }

  revalidatePath(path);
  return Response.json({ revalidated: true });
}
```

## פונקציות קצה

### מתי להשתמש ב-Edge Runtime

פונקציות קצה רצות על Cloudflare Workers (דרך Vercel) ומספקות כמעט אפס זמן הפעלה קרה. השתמש בהן עבור:

| מקרה שימוש | דוגמה |
|---|---|
| ניתוב מבוסס מיקום גיאוגרפי | הפניית משתמשים לתוכן אזורי |
| בדיקות A/B | הפניה לגרסאות ניסוי |
| בדיקות אימות | אימות סשן מהיר |
| המרת תגובה | הוספת כותרות, שינוי תגובות |
| נקודות קצה API פשוטות | אחזור נתונים קל |

### מגבלות Edge Runtime

| מגבלה | פרטים |
|---|---|
| אין Node.js APIs | לא ניתן להשתמש ב-`fs`, `child_process` וכו' |
| אין מודולים מקוריים | לא ניתן להשתמש ב-`bcryptjs`, `postgres` ישירות |
| זמן ביצוע מוגבל | מקסימום 30 שניות (Vercel Pro) |
| זיכרון מוגבל | 128 MB |
| אין Drizzle ORM | השתמש בלקוחות בסיס נתונים תואמי-Edge |

### דוגמת פונקציית קצה

```typescript
// app/api/geo/route.ts
export const runtime = 'edge';

export async function GET(request: Request) {
  const country = request.headers.get('x-vercel-ip-country') || 'US';
  const city = request.headers.get('x-vercel-ip-city') || 'Unknown';

  return Response.json({
    country,
    city,
    timestamp: Date.now(),
  });
}
```

## אסטרטגיות הרחבה אופקית

### עיצוב אפליקציה חסרת מצב

התבנית מתוכננת כחסרת מצב ברמת האפליקציה:

| רכיב | מיקום נתונים | השפעה על הרחבה |
|---|---|---|
| סשנים | בסיס נתונים או JWT | אין מצב משותף בין מופעים |
| משימות רקע | מנהל משימות (לכל מופע או Trigger.dev) | השתמש ב-Trigger.dev למופעים מרובים |
| העלאות קבצים | אחסון חיצוני (S3, Supabase) | אין תלות במערכת קבצים מקומית |
| תוכן CMS | מאגר Git (משובט בעת בנייה/הפעלה) | לקריאה בלבד, זהה לכל מופע |
| מטמון | בזיכרון (לכל מופע) או Redis | שקול Redis למטמון משותף |

### שיקולים לריבוי מופעים

בהרצת מופעים מרובים (Docker Swarm, Kubernetes או פונקציות Vercel מרובות):

1. **משימות רקע**: השתמש ב-Trigger.dev או Vercel Cron במקום `LocalJobManager` למניעת ביצוע כפול.
2. **חיבורי בסיס נתונים**: הפעל מאגר חיבורים למניעת מיצוי חיבורים.
3. **אחסון סשן**: השתמש בסשנים מבוססי-בסיס נתונים במקום מאגרי זיכרון.
4. **ביטול מטמון**: יישם מטמון משותף (Redis) או קבל עקביות סופית עם מטמוני מופע.

### מדדים מרכזיים למעקב

| מדד | כלי | סף |
|---|---|---|
| זמן תגובה (p95) | Sentry, Vercel Analytics | פחות מ-500 אלפיות שנייה |
| שיעור שגיאות | Sentry | פחות מ-1% |
| חיבורי בסיס נתונים | לוח בקרה בסיס נתונים | פחות מ-80% מהמקסימום |
| הפעלות קרות של פונקציות | Vercel Analytics | מעקב תדירות |
| שיעור פגיעות מטמון | לוגי אפליקציה | מעל 80% |
| שימוש בזיכרון | מדדי Vercel/Docker | פחות מ-80% מהמגבלה |

### ניטור ביצועים ב-Sentry

התבנית מגדירה Sentry עם דגימת מעקב:

```typescript
Sentry.init({
  tracesSampleRate: process.env.NODE_ENV === 'production' ? 0.1 : 1.0,
});
```

כוונן `tracesSampleRate` בהתאם לנפח תעבורה:

| בקשות יומיות | קצב דגימה מומלץ |
|---|---|
| פחות מ-10,000 | 1.0 (100%) |
| 10,000–100,000 | 0.1 (10%) |
| 100,000–1,000,000 | 0.01 (1%) |
| מעל 1,000,000 | 0.001 (0.1%) |
