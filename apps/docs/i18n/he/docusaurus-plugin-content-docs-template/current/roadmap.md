# מפת דרכים

דף זה מתאר את כיוון הפיתוח הנוכחי של Directory Web Template וכיצד הקהילה יכולה להשתתף בעיצוב עתידו.

## חזון המוצר

Directory Web Template שואף להיות הפתרון הקוד-פתוח המקיף ביותר לבניית אתרי ספרייה מקצועיים. החזון ארוך הטווח כולל:

- **אתרי ספרייה מוכנים לייצור** שהם יפים, בעלי ביצועים גבוהים וניתנים להתאמה אישית מלאה
- **ניהול תוכן קל** דרך ה-CMS מבוסס Git עם יצירת תוכן אופציונלית מונעת בינה מלאכותית דרך [פלטפורמת Ever Works](https://docs.ever.works)
- **תשלומים ואימות שניתן להרחבה** עם תמיכה בספקים מרובים מחוץ לקופסה
- **בינאום ברמה ראשונה** עם תמיכה מלאה ב-RTL וכיסוי שפות הולך וגדל

## תחומי פיתוח פעיל

### ביצועים ומדדי Core Web Vitals

- אופטימיזציה של Largest Contentful Paint (LCP) עבור דפי רשימות ופרטים של פריטים
- הפחתת גודל חבילת ה-JavaScript דרך פיצול קוד טוב יותר ו-tree shaking
- שיפור צינור אופטימיזציית התמונות עבור צילומי מסך ולוגואים של פריטי הספרייה
- יישום עיבוד מקדים חלקי לטעינות עמוד ראשוניות מהירות יותר

### שיפורי תכונות

- הוספת עוד יכולות סינון וחיפוש (חיפוש רב-ממדי, מסננים מתקדמים)
- יישום תכונות תוכן שנוצר על ידי משתמשים (ביקורות, דירוגים, תגובות)
- הוספת עוד אינטגרציות עם ספקי תשלום ותכונות ניהול מנויים
- הרחבת מערכת הנושאים עם נושאים מובנים נוספים והתאמה אישית קלה יותר

### חוויית מפתחים

- שיפור הגדרת סביבת הפיתוח המקומית עם תיעוד טוב יותר והודעות שגיאה
- הוספת כיסוי בדיקות E2E מקיף יותר עם Playwright
- יצירת תבניות התחלה לסוגי ספרייה נפוצים (SaaS, עסקים מקומיים, משאבים)
- שיפור בטיחות הסוגים ב-TypeScript לאורך כל בסיס הקוד

### בינאום

- הוספת ותרגומי שפות מובנים נוספים
- שיפור תמיכת פריסת RTL עבור ערבית ועברית
- תמיכה בתצורת שפה לפי ספרייה
- הוספת תהליכי עבודה לתרגום אוטומטי

### תיעוד

- הרחבת תיעוד ה-API עם דוגמאות נוספות
- הוספת הדרכות וידאו למשימות נפוצות
- יצירת רשומות החלטת ארכיטקטורה (ADR) להחלטות עיצוב מרכזיות
- בניית מדריכים אינטראקטיביים וסביבות משחק

## כיצד להציע תכונות

### GitHub Issues

הדרך הראשית להציע תכונות היא דרך GitHub Issues בכתובת [github.com/ever-works/directory-web-template/issues](https://github.com/ever-works/directory-web-template/issues).

בעת יצירת בקשת תכונה:

1. **בדוק issues קיימים** תחילה כדי להימנע מכפילויות.
2. **תאר את הבעיה** שאתה מנסה לפתור, לא רק את הפתרון שאתה רוצה.
3. **ספק הקשר** לגבי מקרה השימוש, סוג הספרייה והיקף שלך.
4. **כלול דוגמאות** (מוקאפים, סכמות API, דוגמאות תצורה).

### GitHub Discussions

לרעיונות רחבים יותר הדורשים קלט מהקהילה: [github.com/ever-works/directory-web-template/discussions](https://github.com/ever-works/directory-web-template/discussions)

### רמות עדיפות

- **P0 (קריטי):** פרצות אבטחה, באגים של אובדן נתונים, או בעיות חוסמות. מטופלות מיידית.
- **P1 (גבוה):** תכונות או תיקונים שעובדים עליהם באופן פעיל לקראת הגרסה הבאה.
- **P2 (בינוני):** תכונות מאושרות שתוכננו אך טרם תוזמנו.
- **P3 (נמוך):** שיפורים רצויים. מועמדים טובים לתרומות קהילה.

## תרומה למפת הדרכים

1. **הגש בקשות תכונה כתובות היטב** עם הצהרות ברורות של בעיות ומקרי שימוש.
2. **תרום קוד.** Pull requests הם הדרך המהירה ביותר מרעיון למציאות. ראה את [מדריך תרומה לפרויקט](/contributing).
3. **השתתף בדיונים.** ספק משוב על הצעות ושתף את הניסיון שלך.
4. **דווח על באגים.** דוחות באגים אמינים עוזרים לתעדף תיקונים ולשפר יציבות.

## קצב שחרור גרסאות

גרסאות משוחררות כאשר קבוצה משמעותית של תכונות ותיקונים מוכנה:

- **גרסאות Patch** (תיקוני באגים) מתפרסמות כנדרש, לעתים קרובות שבועית בזמן פיתוח פעיל.
- **גרסאות Minor** (תכונות חדשות) מתפרסמות בערך מדי חודש.
- **גרסאות Major** (שינויים שוברים) נדירות ומלוות במדריכי העברה.

ראה את דף [יומן שינויים](/changelog) לפרטים.
