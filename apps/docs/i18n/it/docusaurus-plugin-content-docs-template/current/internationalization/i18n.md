# Internazionalizzazione (i18n)

Ever Works Template utilizza `next-intl` per l'internazionalizzazione, supportando 21 localizzazioni con routing basato su URL, rilevamento automatico del browser, fallback dei messaggi, layout da destra a sinistra e tag SEO hreflang. Questa pagina illustra l'architettura i18n, la configurazione e i modelli d'uso.

## Architettura

| File | Scopo |
|------|---------|
| `i18n/routing.ts` | Configurazione delle rotte (localizzazioni supportate, strategia del prefisso) |
| `i18n/request.ts` | Risoluzione della localizzazione e caricamento dei messaggi per richiesta |
| `i18n/navigation.ts` | Link, redirect, usePathname, useRouter compatibili con la localizzazione |
| `lib/constants.ts` | Elenco delle localizzazioni, localizzazione predefinita, definizioni RTL |
| `lib/seo/hreflang.ts` | Generazione di tag SEO hreflang per i motori di ricerca |
| `components/language-switcher.tsx` | Selettore lingua nell'interfaccia utente |
| `messages/*.json` | File di traduzione (uno per localizzazione) |
| `app/[locale]/layout.tsx` | Layout radice con NextIntlClientProvider |

## Localizzazioni Supportate

Il template definisce 21 localizzazioni in `lib/constants.ts`:

```typescript
export const DEFAULT_LOCALE = 'en';
export const LOCALES = [
  'en', 'fr', 'es', 'de', 'zh', 'ar', 'he',
  'ru', 'uk', 'pt', 'it', 'ja', 'ko', 'nl',
  'pl', 'tr', 'vi', 'th', 'hi', 'id', 'bg'
] as const;

export type Locale = (typeof LOCALES)[number];

/** Localizzazioni che utilizzano la direzione del testo da destra a sinistra */
export const RTL_LOCALES: readonly Locale[] = ['ar', 'he'] as const;
```

## Configurazione del Routing

Il modulo di routing in `i18n/routing.ts` definisce come le localizzazioni vengono mappate agli URL:

```typescript
import { defineRouting } from "next-intl/routing";
import { DEFAULT_LOCALE, LOCALES } from "@/lib/constants";

export const routing = defineRouting({
  locales: LOCALES,
  defaultLocale: DEFAULT_LOCALE,
  localeDetection: true,
  localePrefix: "as-needed",
});
```

### Strategia del Prefisso della Localizzazione

La strategia `"as-needed"` mantiene gli URL puliti per la localizzazione predefinita:

| URL | Localizzazione risolta | Spiegazione |
|-----|-----------------|-------------|
| `/about` | `en` (predefinita) | Nessun prefisso necessario per la localizzazione predefinita |
| `/fr/about` | `fr` | Prefisso richiesto per le localizzazioni non predefinite |
| `/ar/about` | `ar` | Prefisso richiesto per le localizzazioni non predefinite |

### Rilevamento Localizzazione

Con `localeDetection: true`, il sistema controlla automaticamente:

1. Un cookie di localizzazione (impostato dal componente LanguageSwitcher)
2. L'intestazione HTTP `Accept-Language` del browser
3. Ritorna a `DEFAULT_LOCALE` (`'en'`) se non viene trovata una corrispondenza

## Caricamento dei Messaggi e Fallback

Il modulo `i18n/request.ts` viene eseguito a ogni richiesta del server per risolvere la localizzazione e caricare i messaggi di traduzione:

```typescript
export default getRequestConfig(async ({ requestLocale }) => {
  let locale = await requestLocale;

  if (!locale || !routing.locales.includes(locale as any)) {
    locale = routing.defaultLocale;
  }

  const userMessages = (await import(`../messages/${locale}.json`)).default;
  const defaultMessages = (await import(`../messages/en.json`)).default;
  const messages = deepmerge(defaultMessages, userMessages);

  return { locale, messages };
});
```

La strategia `deepmerge` garantisce che le chiavi non tradotte mostrino il testo inglese invece dei nomi grezzi delle chiavi.

## File di Traduzione

I messaggi di traduzione sono memorizzati come file JSON nella directory `messages/`, un file per localizzazione:

```
messages/
  en.json    # Inglese (sorgente, tutte le chiavi presenti)
  fr.json    # Francese
  es.json    # Spagnolo
  de.json    # Tedesco
  ar.json    # Arabo
  it.json    # Italiano
  ... (21 file in totale)
```

### Utilizzo delle Traduzioni nei Componenti

**Componenti client** usano il hook `useTranslations`:

```tsx
import { useTranslations } from "next-intl";

function LoginButton() {
  const t = useTranslations("auth");
  return <button>{t("SIGN_IN")}</button>;
}
```

**Componenti server** usano il `getTranslations` asincrono:

```tsx
import { getTranslations } from "next-intl/server";

export default async function Page() {
  const t = await getTranslations("common");
  return <h1>{t("DISCOVER")}</h1>;
}
```

## Utilità di Navigazione

Il modulo `i18n/navigation.ts` esporta sostituti compatibili con la localizzazione per la navigazione standard di Next.js:

```typescript
import { createNavigation } from "next-intl/navigation";
import { routing } from "./routing";

export const { Link, redirect, usePathname, useRouter, getPathname } =
  createNavigation(routing);
```

## Aggiungere una Nuova Localizzazione

1. Aggiungere il codice della localizzazione all'array `LOCALES` in `lib/constants.ts`
2. Se la localizzazione è RTL, aggiungerla a `RTL_LOCALES`
3. Creare `messages/<codice>.json` con le chiavi tradotte
4. Aggiungere una SVG della bandiera in `public/flags/<codice>.svg`
5. Aggiungere la voce della localizzazione alla `languageMap` in `components/language-switcher.tsx`
6. Aggiungere la mappatura hreflang in `lib/seo/hreflang.ts`

## Best Practice

1. **Usare `Link` da `@/i18n/navigation`** invece di `next/link`
2. **Aggiungere le nuove chiavi prima a `en.json`** poiché è la lingua di fallback per tutte le localizzazioni
3. **Non codificare mai stringhe inglesi** nei componenti — usare `useTranslations` o `getTranslations`
4. **Affidarsi al fallback `deepmerge`** — i file di localizzazione parzialmente tradotti funzionano correttamente
5. **Testare i layout RTL** passando all'arabo o all'ebraico nel LanguageSwitcher
