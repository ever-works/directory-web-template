# Internationalisering (i18n)

De Ever Works Template gebruikt `next-intl` voor internationalisering en ondersteunt 21 taalregio's met URL-gebaseerde routering, automatische browserdetectie, berichtenuitwijking, rechts-naar-links layouts en SEO hreflang-tags. Deze pagina behandelt de i18n-architectuur, configuratie en gebruikspatronen.

## Architectuur

| Bestand | Doel |
|------|---------|
| `i18n/routing.ts` | Routeconfiguratie (ondersteunde taalregio's, voorvoegsels) |
| `i18n/request.ts` | Per-verzoek taalregio-oplossing en berichtenlading |
| `i18n/navigation.ts` | Taalregio-bewuste Link, redirect, usePathname, useRouter |
| `lib/constants.ts` | Taalregio-lijst, standaard taalregio, RTL-definities |
| `lib/seo/hreflang.ts` | SEO hreflang-tag generatie voor zoekmachines |
| `components/language-switcher.tsx` | Gebruikersinterface taalkeuzeschakelaar |
| `messages/*.json` | Vertaalbestanden (één per taalregio) |
| `app/[locale]/layout.tsx` | Root-layout met NextIntlClientProvider |

## Routingconfiguratie

De routingmodule in `i18n/routing.ts` bepaalt hoe taalregio's aan URL's worden gekoppeld:

```typescript
import { defineRouting } from "next-intl/routing";
import { DEFAULT_LOCALE, LOCALES } from "@/lib/constants";

export const routing = defineRouting({
  locales: LOCALES,
  defaultLocale: DEFAULT_LOCALE,
  localeDetection: true,
  localePrefix: "as-needed",
});
```

### Taalregio-voorvoegsels strategie

De `"as-needed"`-strategie houdt URL's schoon voor de standaard taalregio:

| URL | Opgeloste taalregio | Uitleg |
|-----|-----------------|-------------|
| `/about` | `en` (standaard) | Geen voorvoegsel nodig voor de standaard taalregio |
| `/fr/about` | `fr` | Voorvoegsel vereist voor niet-standaard taalregio's |
| `/ar/about` | `ar` | Voorvoegsel vereist voor niet-standaard taalregio's |

### Taaldetectie

Met `localeDetection: true` controleert het systeem automatisch:

1. Een taalregio-cookie (ingesteld door de LanguageSwitcher-component)
2. De `Accept-Language`-HTTP-header van de browser
3. Valt terug op `DEFAULT_LOCALE` (`'en'`) als er geen overeenkomst wordt gevonden

## Berichten laden en uitwijken

De module `i18n/request.ts` wordt bij elk serververzoek uitgevoerd om de taalregio op te lossen en vertaalberichten te laden:

```typescript
export default getRequestConfig(async ({ requestLocale }) => {
  let locale = await requestLocale;

  if (!locale || !routing.locales.includes(locale as any)) {
    locale = routing.defaultLocale;
  }

  const userMessages = (await import(`../messages/${locale}.json`)).default;
  const defaultMessages = (await import(`../messages/en.json`)).default;
  const messages = deepmerge(defaultMessages, userMessages);

  return { locale, messages };
});
```

## Vertaalbestanden

Vertaalberichten worden opgeslagen als JSON-bestanden in de map `messages/`, één bestand per taalregio:

```
messages/
  en.json    # Engels (bron, alle sleutels aanwezig)
  fr.json    # Frans
  es.json    # Spaans
  de.json    # Duits
  ar.json    # Arabisch
  zh.json    # Chinees
  ... (21 bestanden totaal)
```

### Gebruik van vertalingen in componenten

**Clientcomponenten** gebruiken de `useTranslations`-hook:

```tsx
import { useTranslations } from "next-intl";

function LoginButton() {
  const t = useTranslations("auth");
  return <button>{t("SIGN_IN")}</button>;
}
```

**Servercomponenten** gebruiken de asynchrone `getTranslations`:

```tsx
import { getTranslations } from "next-intl/server";

export default async function Page() {
  const t = await getTranslations("common");
  return <h1>{t("DISCOVER")}</h1>;
}
```

## Navigatiehulpmiddelen

De module `i18n/navigation.ts` exporteert taalregio-bewuste vervangingen voor standaard Next.js-navigatie:

```typescript
import { createNavigation } from "next-intl/navigation";
import { routing } from "./routing";

export const { Link, redirect, usePathname, useRouter, getPathname } =
  createNavigation(routing);
```

| Export | Vervangt | Gedrag |
|--------|----------|----------|
| `Link` | `next/link` | Voegt automatisch taalregio-voorvoegsel toe aan `href` |
| `redirect` | `next/navigation` redirect | Leidt door binnen de huidige taalregio |
| `usePathname` | `next/navigation` usePathname | Geeft pad zonder taalregio-voorvoegsel terug |
| `useRouter` | `next/navigation` useRouter | push/replace bewaren de huidige taalregio |
| `getPathname` | N/A | Serverside padresolutie met taalregio |

## Nieuwe taalregio toevoegen

1. Taalregio-code toevoegen aan het `LOCALES`-array in `lib/constants.ts`
2. Als de taalregio RTL is, toevoegen aan `RTL_LOCALES`
3. `messages/<code>.json` aanmaken met vertaalde sleutels (ontbrekende sleutels wijken uit naar Engels)
4. Een vlag-SVG toevoegen aan `public/flags/<code>.svg`
5. De taalregio-invoer toevoegen aan de `languageMap` in `components/language-switcher.tsx`
6. De hreflang-mapping toevoegen in `lib/seo/hreflang.ts`

## Best Practices

1. **Gebruik `Link` uit `@/i18n/navigation`** in plaats van `next/link`
2. **Voeg nieuwe sleutels eerst toe aan `en.json`** omdat het de uitwijktaal is voor alle taalregio's
3. **Codeer nooit Engelse strings hard** in componenten — gebruik `useTranslations` of `getTranslations`
4. **Vertrouw op de `deepmerge`-uitwijking** — gedeeltelijk vertaalde bestanden werken correct
5. **Test RTL-layouts** door over te schakelen naar Arabisch of Hebreeuws in de LanguageSwitcher
