# Internacjonalizacja (i18n)

Ever Works Template używa `next-intl` do internacjonalizacji, obsługując 21 regionów językowych z routingiem opartym na URL, automatycznym wykrywaniem przez przeglądarkę, fallbackiem wiadomości, układami od prawej do lewej oraz tagami SEO hreflang. Ta strona dokumentuje architekturę i18n, konfigurację i wzorce użycia.

## Architektura

| Plik | Cel |
|------|---------|
| `i18n/routing.ts` | Konfiguracja routingu (obsługiwane regiony, strategia prefiksu) |
| `i18n/request.ts` | Rozwiązywanie regionu i ładowanie wiadomości dla każdego żądania |
| `i18n/navigation.ts` | Link, redirect, usePathname, useRouter zgodne z regionem |
| `lib/constants.ts` | Lista regionów, domyślny region, definicje RTL |
| `lib/seo/hreflang.ts` | Generowanie tagów SEO hreflang dla wyszukiwarek |
| `components/language-switcher.tsx` | Wybór języka w interfejsie użytkownika |
| `messages/*.json` | Pliki tłumaczeń (jeden na region) |
| `app/[locale]/layout.tsx` | Główny layout z NextIntlClientProvider |

## Obsługiwane Regiony

Szablon definiuje 21 regionów w `lib/constants.ts`:

```typescript
export const DEFAULT_LOCALE = 'en';
export const LOCALES = [
  'en', 'fr', 'es', 'de', 'zh', 'ar', 'he',
  'ru', 'uk', 'pt', 'it', 'ja', 'ko', 'nl',
  'pl', 'tr', 'vi', 'th', 'hi', 'id', 'bg'
] as const;

export type Locale = (typeof LOCALES)[number];

/** Regiony używające pisma od prawej do lewej */
export const RTL_LOCALES: readonly Locale[] = ['ar', 'he'] as const;
```

| Kod | Język | Kierunek |
|------|----------|-----------|
| `en` | Angielski | LTR |
| `fr` | Francuski | LTR |
| `es` | Hiszpański | LTR |
| `de` | Niemiecki | LTR |
| `zh` | Chiński uproszczony | LTR |
| `ar` | Arabski | **RTL** |
| `he` | Hebrajski | **RTL** |
| `it` | Włoski | LTR |
| `pt` | Portugalski | LTR |
| `nl` | Holenderski | LTR |
| `pl` | Polski | LTR |
| `ru` | Rosyjski | LTR |

## Konfiguracja Routingu

Moduł routingu w `i18n/routing.ts` definiuje mapowanie regionów na URL-e:

```typescript
import { defineRouting } from "next-intl/routing";
import { DEFAULT_LOCALE, LOCALES } from "@/lib/constants";

export const routing = defineRouting({
  locales: LOCALES,
  defaultLocale: DEFAULT_LOCALE,
  localeDetection: true,
  localePrefix: "as-needed",
});
```

### Strategia Prefiksu Regionu

Strategia `"as-needed"` utrzymuje czyste URL-e dla domyślnego regionu:

| URL | Rozwiązany Region | Wyjaśnienie |
|-----|-----------------|-------------|
| `/about` | `en` (domyślny) | Brak prefiksu dla domyślnego regionu |
| `/fr/about` | `fr` | Prefiks wymagany dla regionów niedomyślnych |
| `/ar/about` | `ar` | Prefiks wymagany dla regionów niedomyślnych |

### Wykrywanie Regionu

Przy `localeDetection: true` system automatycznie sprawdza:

1. Cookie regionu (ustawiane przez komponent LanguageSwitcher)
2. Nagłówek HTTP `Accept-Language` przeglądarki
3. Wraca do `DEFAULT_LOCALE` (`'en'`) jeśli nie ma dopasowania

## Ładowanie Wiadomości i Fallback

Moduł `i18n/request.ts` wykonuje się przy każdym żądaniu serwera:

```typescript
export default getRequestConfig(async ({ requestLocale }) => {
  let locale = await requestLocale;

  if (!locale || !routing.locales.includes(locale as any)) {
    locale = routing.defaultLocale;
  }

  const userMessages = (await import(`../messages/${locale}.json`)).default;
  const defaultMessages = (await import(`../messages/en.json`)).default;
  const messages = deepmerge(defaultMessages, userMessages);

  return { locale, messages };
});
```

Strategia `deepmerge` zapewnia, że nieprzetłumaczone klucze pokazują tekst angielski.

## Pliki Tłumaczeń

Wiadomości tłumaczeń są przechowywane jako pliki JSON w katalogu `messages/`:

```
messages/
  en.json    # Angielski (źródło, wszystkie klucze obecne)
  fr.json    # Francuski
  pl.json    # Polski
  ... (21 plików łącznie)
```

### Używanie Tłumaczeń w Komponentach

**Komponenty klienckie** używają hooka `useTranslations`:

```tsx
import { useTranslations } from "next-intl";

function LoginButton() {
  const t = useTranslations("auth");
  return <button>{t("SIGN_IN")}</button>;
}
```

**Komponenty serwerowe** używają asynchronicznego `getTranslations`:

```tsx
import { getTranslations } from "next-intl/server";

export default async function Page() {
  const t = await getTranslations("common");
  return <h1>{t("DISCOVER")}</h1>;
}
```

## Dodawanie Nowego Regionu

1. Dodać kod regionu do tablicy `LOCALES` w `lib/constants.ts`
2. Jeśli region jest RTL, dodać go do `RTL_LOCALES`
3. Stworzyć `messages/<kod>.json` z przetłumaczonymi kluczami
4. Dodać SVG flagi w `public/flags/<kod>.svg`
5. Dodać wpis regionu do `languageMap` w `components/language-switcher.tsx`
6. Dodać mapowanie hreflang w `lib/seo/hreflang.ts`

## Najlepsze Praktyki

1. **Używać `Link` z `@/i18n/navigation`** zamiast `next/link`
2. **Dodawać nowe klucze najpierw do `en.json`** gdyż jest językiem fallback
3. **Nigdy nie kodować na stałe angielskich ciągów** w komponentach
4. **Polegać na fallbacku `deepmerge`** — częściowo przetłumaczone pliki działają poprawnie
5. **Testować układy RTL** przełączając na arabski lub hebrajski
