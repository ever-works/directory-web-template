# Internacionalização (i18n)

O Ever Works Template usa `next-intl` para internacionalização, suportando 21 localizações com roteamento baseado em URL, detecção automática do navegador, fallback de mensagens, layouts da direita para a esquerda e tags SEO hreflang. Esta página documenta a arquitetura i18n, configuração e padrões de uso.

## Arquitetura

| Arquivo | Finalidade |
|------|---------|
| `i18n/routing.ts` | Configuração de rotas (localizações suportadas, estratégia de prefixo) |
| `i18n/request.ts` | Resolução de localização e carregamento de mensagens por requisição |
| `i18n/navigation.ts` | Link, redirect, usePathname, useRouter compatíveis com localização |
| `lib/constants.ts` | Lista de localizações, localização padrão, definições RTL |
| `lib/seo/hreflang.ts` | Geração de tags SEO hreflang para mecanismos de busca |
| `components/language-switcher.tsx` | Seletor de idioma na interface do usuário |
| `messages/*.json` | Arquivos de tradução (um por localização) |
| `app/[locale]/layout.tsx` | Layout raiz com NextIntlClientProvider |

## Localizações Suportadas

O template define 21 localizações em `lib/constants.ts`:

```typescript
export const DEFAULT_LOCALE = 'en';
export const LOCALES = [
  'en', 'fr', 'es', 'de', 'zh', 'ar', 'he',
  'ru', 'uk', 'pt', 'it', 'ja', 'ko', 'nl',
  'pl', 'tr', 'vi', 'th', 'hi', 'id', 'bg'
] as const;

export type Locale = (typeof LOCALES)[number];

/** Localizações que usam direção de texto da direita para esquerda */
export const RTL_LOCALES: readonly Locale[] = ['ar', 'he'] as const;
```

| Código | Idioma | Direção |
|------|----------|-----------|
| `en` | Inglês | LTR |
| `fr` | Francês | LTR |
| `es` | Espanhol | LTR |
| `de` | Alemão | LTR |
| `zh` | Chinês Simplificado | LTR |
| `ar` | Árabe | **RTL** |
| `he` | Hebraico | **RTL** |
| `it` | Italiano | LTR |
| `pt` | Português | LTR |

## Configuração de Roteamento

O módulo de roteamento em `i18n/routing.ts` define como as localizações são mapeadas para URLs:

```typescript
import { defineRouting } from "next-intl/routing";
import { DEFAULT_LOCALE, LOCALES } from "@/lib/constants";

export const routing = defineRouting({
  locales: LOCALES,
  defaultLocale: DEFAULT_LOCALE,
  localeDetection: true,
  localePrefix: "as-needed",
});
```

### Estratégia de Prefixo de Localização

A estratégia `"as-needed"` mantém URLs limpos para a localização padrão:

| URL | Localização Resolvida | Explicação |
|-----|-----------------|-------------|
| `/about` | `en` (padrão) | Nenhum prefixo necessário para a localização padrão |
| `/fr/about` | `fr` | Prefixo obrigatório para localizações não padrão |
| `/ar/about` | `ar` | Prefixo obrigatório para localizações não padrão |

### Detecção de Localização

Com `localeDetection: true`, o sistema verifica automaticamente:

1. Um cookie de localização (definido pelo componente LanguageSwitcher)
2. O cabeçalho HTTP `Accept-Language` do navegador
3. Retorna para `DEFAULT_LOCALE` (`'en'`) se nenhuma correspondência for encontrada

## Carregamento de Mensagens e Fallback

O módulo `i18n/request.ts` executa em cada requisição do servidor para resolver a localização e carregar mensagens de tradução:

```typescript
export default getRequestConfig(async ({ requestLocale }) => {
  let locale = await requestLocale;

  if (!locale || !routing.locales.includes(locale as any)) {
    locale = routing.defaultLocale;
  }

  const userMessages = (await import(`../messages/${locale}.json`)).default;
  const defaultMessages = (await import(`../messages/en.json`)).default;
  const messages = deepmerge(defaultMessages, userMessages);

  return { locale, messages };
});
```

A estratégia `deepmerge` garante que chaves não traduzidas mostrem texto em inglês em vez dos nomes brutos das chaves.

## Arquivos de Tradução

As mensagens de tradução são armazenadas como arquivos JSON no diretório `messages/`, um arquivo por localização:

```
messages/
  en.json    # Inglês (fonte, todas as chaves presentes)
  fr.json    # Francês
  es.json    # Espanhol
  de.json    # Alemão
  ar.json    # Árabe
  pt.json    # Português
  ... (21 arquivos no total)
```

### Usando Traduções em Componentes

**Componentes cliente** usam o hook `useTranslations`:

```tsx
import { useTranslations } from "next-intl";

function LoginButton() {
  const t = useTranslations("auth");
  return <button>{t("SIGN_IN")}</button>;
}
```

**Componentes servidor** usam o `getTranslations` assíncrono:

```tsx
import { getTranslations } from "next-intl/server";

export default async function Page() {
  const t = await getTranslations("common");
  return <h1>{t("DISCOVER")}</h1>;
}
```

## Utilitários de Navegação

O módulo `i18n/navigation.ts` exporta substitutos compatíveis com localização para a navegação padrão do Next.js:

```typescript
import { createNavigation } from "next-intl/navigation";
import { routing } from "./routing";

export const { Link, redirect, usePathname, useRouter, getPathname } =
  createNavigation(routing);
```

## Adicionando uma Nova Localização

1. Adicionar o código da localização ao array `LOCALES` em `lib/constants.ts`
2. Se a localização for RTL, adicioná-la a `RTL_LOCALES`
3. Criar `messages/<código>.json` com as chaves traduzidas
4. Adicionar um SVG de bandeira em `public/flags/<código>.svg`
5. Adicionar a entrada de localização ao `languageMap` em `components/language-switcher.tsx`
6. Adicionar o mapeamento hreflang em `lib/seo/hreflang.ts`

## Melhores Práticas

1. **Usar `Link` de `@/i18n/navigation`** em vez de `next/link`
2. **Adicionar novas chaves primeiro em `en.json`** pois é o idioma de fallback para todas as localizações
3. **Nunca codificar strings em inglês** em componentes — usar `useTranslations` ou `getTranslations`
4. **Confiar no fallback `deepmerge`** — arquivos de localização parcialmente traduzidos funcionam corretamente
5. **Testar layouts RTL** alternando para árabe ou hebraico no LanguageSwitcher
