# Оптимизация производительности

В этом руководстве рассматриваются оптимизации производительности, встроенные в шаблон Ever Works, и методы обеспечения быстрой загрузки по мере роста вашего приложения.

## Конфигурация Next.js

Шаблон `next.config.ts` включает в себя несколько настроек, ориентированных на производительность:

### Автономный выход

```typescript
const nextConfig: NextConfig = {
  output: "standalone",
  // ...
};
```

Режим вывода `standalone` создает автономную сборку, включающую только файлы, необходимые для запуска приложения. Это уменьшает размер контейнера и время запуска в производство.

### Оптимизация импорта пакетов

```typescript
experimental: {
  optimizePackageImports: ["@heroui/react", "lucide-react"],
},
```

Этот параметр включает встряхивание дерева для пакетов с большим количеством бочковых файлов. Вместо импорта всей библиотеки `@heroui/react` или `lucide-react` в комплект включаются только фактически используемые компоненты.

### Оптимизация наблюдения за веб-пакетом

```typescript
if (dev) {
  config.watchOptions = {
    ...config.watchOptions,
    ignored: ['**/node_modules/**', '**/.git/**', '**/.content/**']
  };
}
```

Каталог `.content/` (CMS на базе Git с более чем 220 файлами уценки) исключен из средства наблюдения за файлами веб-пакета, находящегося в разработке. Это предотвращает ненужные перестройки при изменении файлов содержимого и значительно снижает загрузку ЦП во время разработки.

### Подавленные предупреждения

Подробное ведение журнала инфраструктуры подавляется в средах CI и Vercel:

```typescript
if (process.env.CI || process.env.VERCEL) {
  config.infrastructureLogging = { level: 'error' };
}
```

## Оптимизация изображения

### Удаленные шаблоны

Шаблон динамически генерирует разрешенные шаблоны удаленных изображений, используя `generateImageRemotePatterns()` . Это гарантирует, что изображения из настроенных CDN и внешних источников оптимизируются с помощью встроенного конвейера изображений Next.js.

### Обработка SVG

```typescript
images: {
  dangerouslyAllowSVG: true,
  contentDispositionType: 'attachment',
  contentSecurityPolicy: "default-src 'self'; script-src 'none'; sandbox;",
  unoptimized: false,
},
```

Изображения SVG разрешены, но помещены в «песочницу» со строгой политикой безопасности контента, которая запрещает выполнение сценариев. Это позволяет использовать логотипы и значки SVG, одновременно предотвращая XSS посредством внедрения SVG.

### Лучшие практики для изображений

| Техника | Реализация | Влияние |
|---|---|---|
| Используйте `next/image` | Встроенный компонент с отложенной загрузкой | Автоматический WebP/AVIF, адаптивные размеры |
| Установить явные размеры | `width` и `height` реквизит | Предотвращает накопительный сдвиг макета (CLS) |
| Используйте `priority` для LCP | `<Image priority />` для главных изображений | Предварительная загрузка самого большого содержательного изображения Paint |
| Используйте опору `sizes` | `sizes="(max-width: 768px) 100vw, 50vw"` | Предотвращает загрузку изображений слишком большого размера |
| Размытие заполнителей | `placeholder="blur"` с `blurDataURL` | Улучшает воспринимаемую скорость загрузки |

## Стратегии кэширования

### HTTP-заголовки

Шаблон устанавливает заголовки, связанные с кешем, в `next.config.ts` :

```typescript
headers: [
  { key: "X-DNS-Prefetch-Control", value: "on" },
]
```

Предварительная выборка DNS включена глобально, чтобы уменьшить задержку поиска DNS для внешних ресурсов.

### Статическая генерация

Шаблон использует щедрый тайм-аут для генерации статической страницы:

```typescript
staticPageGenerationTimeout: 180, // 3 minutes
```

Сюда входят страницы, которые извлекают данные из внешних API или CMS на основе Git во время сборки.

### Конфигурация ETag

```typescript
generateEtags: false,
```

ETags отключены на уровне Next.js, поскольку CDN/обратный прокси-сервер (Vercel Edge Network или Cloudflare) более эффективно обрабатывает проверку кэша.

### Кэширование на уровне приложения

Фоновый аналитический процессор предварительно разогревает кэши через определенные промежутки времени:

| Тип кэша | Интервал обновления | Данные |
|---|---|---|
| Тенденции роста пользователей | 10 минут | Ежемесячный прирост пользователей за 6, 12, 24 месяца |
| Тенденции деятельности | 5 минут | Данные о деятельности для 7, 14, 30-дневных окон |
| Рейтинг лучших товаров | 15 минут | Топ 10, 20, 50 товаров |
| Недавняя активность | 2 минуты | Последние 10 и 20 записей о деятельности |
| Показатели производительности | 30 секунд | Статистика производительности запросов |
| Очистка кэша | 1 час | Удаление просроченной записи кэша |

## Ленивая загрузка

### Отложенная загрузка на уровне компонентов

Используйте `next/dynamic` для тяжелых компонентов, которые не нужны при первоначальном рендеринге:

```typescript
import dynamic from 'next/dynamic';

const HeavyChart = dynamic(() => import('@/components/charts/HeavyChart'), {
  loading: () => <ChartSkeleton />,
  ssr: false, // disable SSR for client-only components
});
```

### Разделение кода на уровне маршрута

Маршрутизатор приложений Next.js автоматически разделяет код по маршруту. Каждая страница в `app/[locale]/` получает свой собственный пакет, поэтому пользователи загружают только тот JavaScript, который необходим для текущей страницы.

### Динамический импорт в фоновых заданиях

Шаблон использует динамический импорт внутри обратных вызовов заданий, чтобы веб-пакет не мог втягивать серверные модули в клиентский пакет:

```typescript
manager.scheduleJob('repository-sync', 'Repository Synchronization', async () => {
  const { syncManager } = await import('@/lib/services/sync-service');
  await syncManager.performSync();
}, 5 * 60 * 1000);
```

## Оптимизация размера пакета

### Анализ пакета

Запустите следующее, чтобы проверить состав пакета:

```bash
ANALYZE=true pnpm build
```

Если настроено `@next/bundle-analyzer` , создается интерактивная древовидная карта, показывающая, какие модули влияют на размер пакета.

### Общие методы оптимизации

| Техника | Пример | Экономия |
|---|---|---|
| Оптимизация файла ствола | `optimizePackageImports` в конфигурации | Предотвращает импорт целых библиотек значков/UI |
| Серверные модули | `import 'server-only'` в файлах lib | Предотвращает случайное объединение клиентов |
| Динамический импорт | `await import('@/lib/services/...')` | Откладывает загрузку до тех пор, пока она не понадобится |
| Внешние пакеты | `serverExternalPackages: ['postgres', 'bcryptjs', 'drizzle-orm']` | Исключает из комплектации веб-пакета |

Конфигурация `serverExternalPackages` особенно важна:

```typescript
serverExternalPackages: ['postgres', 'bcryptjs', 'drizzle-orm'],
```

Эти пакеты исключены из комплектации веб-пакетов и загружаются изначально во время выполнения, что сокращает время сборки и позволяет избежать проблем совместимости с собственными модулями.

## Советы по оптимизации маяка

### Основные цели веб-показателей

| Метрическая | Цель | Ключевые факторы |
|---|---|---|
| **LCP** (самая большая по содержанию краска) | < 2,5 с | Оптимизация изображения, приоритетная загрузка, время ответа сервера |
| **FID** (первая входная задержка) | < 100 мс | Разделение кода, минимальная блокировка основного потока |
| **CLS** (накопительный сдвиг макета) | < 0,1 | Явные размеры изображения, стратегия загрузки шрифтов |
| **TTFB** (время до первого байта) | < 800 мс | Кэширование CDN, пограничные функции, оптимизация запросов к базе данных |

### Практический контрольный список

1. **Изображения**: используйте `next/image` с явными реквизитами `width` , `height` и `sizes` . Отметьте изображения над сгибом с помощью `priority` .
2. **Шрифты**: используйте `next/font` для самостоятельного размещения шрифтов с помощью `display: swap` и предварительной загрузки важных файлов шрифтов.
3. **JavaScript**: просмотрите `optimizePackageImports` и добавьте все большие библиотеки, использующие бочковые файлы.
4. **CSS**: в шаблоне используется CSS Tailwind, который уже удален в производственных сборках. Избегайте импорта неиспользуемых модулей CSS.
5. **Сторонние сценарии**: отложите некритичные сценарии, используя `next/script` с `strategy="lazyOnload"` .
6. **Компоненты сервера**: по умолчанию используются компоненты сервера React (RSC), используйте `"use client"` только там, где требуется интерактивность.

### Бегущий маяк

Шаблон включает конфигурацию `lighthouse-test.json` . Запустите автоматические тесты Lighthouse:

```bash
npx lhci autorun --config=lighthouse-test.json
```

Или используйте панель Chrome DevTools Lighthouse для аудита вручную.

## Производительность запросов к базе данных

### Пул соединений

Используйте пул соединений, чтобы избежать открытия нового соединения с базой данных при каждом запросе. Подробности настройки см. в [Руководстве по масштабированию](/deployment/scaling).

### Оптимизация запросов

- Используйте шаблон репозитория ( `lib/repositories/` ) для централизации и оптимизации запросов.
- Репозиторий аналитики включает встроенные уровни кэша с настраиваемым TTL.
- Отслеживайте медленные запросы с помощью фонового задания показателей производительности.

### Стратегия индексирования

Просмотрите `lib/db/schema.ts` для существующих индексов. Добавьте индексы для:
- Столбцы, используемые в `WHERE` предложениях
- Столбцы внешнего ключа
- Столбцы, используемые в `ORDER BY` предложениях
- Составные индексы для поиска по нескольким столбцам.

## Мониторинг производительности

### Интеграция Sentry

Шаблон интегрирует Sentry для мониторинга производительности в `instrumentation.ts` :

```typescript
Sentry.init({
  tracesSampleRate: process.env.NODE_ENV === 'production' ? 0.1 : 1.0,
});
```

Трассировки отбираются на 10 % при производстве и на 100 % при разработке. Отрегулируйте `tracesSampleRate` в зависимости от объема вашего трафика и ограничений плана Sentry.

### Пользовательские маркеры производительности

Используйте Web Performance API для настройки времени:

```typescript
performance.mark('data-fetch-start');
const data = await fetchData();
performance.mark('data-fetch-end');
performance.measure('data-fetch', 'data-fetch-start', 'data-fetch-end');
```

## Резюме

| Площадь | Встроенная оптимизация | Дополнительные шаги |
|---|---|---|
| Изображения | Автоматическая песочница WebP/AVIF, SVG | Добавьте `priority` к изображениям LCP, используйте `sizes` |
| JavaScript | Оптимизация пакетов, разделение кода | Добавьте библиотеки в `optimizePackageImports` |
| Кэширование | Фоновое потепление кэша, предварительная выборка DNS | Настройка правил кэширования CDN |
| База данных | Пул соединений, шаблон репозитория | Добавляйте индексы, отслеживайте медленные запросы |
| Построить | Автономный выход, внешние пакеты | Включить анализатор пакетов |
| Мониторинг | Отслеживание караула, работа по показателям производительности | Настройка оповещений об ухудшении показателей |
