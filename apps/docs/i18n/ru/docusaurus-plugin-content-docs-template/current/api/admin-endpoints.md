# Конечные точки API администратора

API администратора содержит около 60 обработчиков маршрутов в 19 группах ресурсов. Все конечные точки администратора защищены промежуточным программным обеспечением `withAdminAuth`, которое проверяет аутентификацию и назначение роли администратора посредством запроса к базе данных.

## Аутентификация

Для каждой конечной точки администратора требуется:

1. Действительный сеанс JWT (проверен через `auth()`)
2. Роль администратора в таблице `user_roles` (проверено через `isAdmin()` от `lib/db/roles.ts`)

Неаутентифицированные запросы получают ответ `401`. Запросы, прошедшие проверку подлинности, но не относящиеся к администратору, получают ответ `403`.

## Группы ресурсов

### Категории (`/api/admin/categories`)

Управляйте категориями контента с помощью сохраняемости на основе Git.

|Метод|Путь|Описание|
|--------|------|-------------|
|`GET`|`/api/admin/categories`|Список категорий с нумерацией страниц|
|`POST`|`/api/admin/categories`|Создать новую категорию|
|`GET`|`/api/admin/categories/all`|Получить все категории (без нумерации страниц)|
|`POST`|`/api/admin/categories/git`|Синхронизировать категории с репозиторием Git.|
|`POST`|`/api/admin/categories/reorder`|Изменить порядок позиций категорий|
|`GET`|`/api/admin/categories/[id]`|Получить категорию по идентификатору|
|`PUT`|`/api/admin/categories/[id]`|Обновить категорию|
|`DELETE`|`/api/admin/categories/[id]`|Удалить категорию|

## Общие шаблоны

### Массовые операции

Несколько ресурсов поддерживают массовые операции через POST с массивом идентификаторов:

```json
POST /api/admin/items/bulk
{
  "action": "approve",
  "ids": ["item-1", "item-2", "item-3"]
}
```

### Конечные точки статистики

Большинство групп ресурсов включают конечную точку `/stats`, возвращающую совокупное количество:

```json
GET /api/admin/items/stats
{
  "success": true,
  "data": {
    "total": 1250,
    "published": 980,
    "pending": 120,
    "rejected": 50,
    "draft": 100
  }
}
```

### История аудита

Элементы поддерживают отслеживание истории аудита через конечную точку `/[id]/history`, фиксируя, кто и когда внес изменения.
