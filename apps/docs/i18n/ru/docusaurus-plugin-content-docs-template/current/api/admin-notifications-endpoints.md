# Конечные точки API уведомлений администратора

API уведомлений администратора управляет уведомлениями в приложении для пользователей с правами администратора. Он поддерживает перечисление уведомлений с количеством непрочитанных, создание новых уведомлений для конкретных пользователей и пометку уведомлений как прочитанных (индивидуально или массово). Уведомления хранятся в базе данных и предназначены для отдельных пользователей.

## Сводка маршрута

|Метод|Путь|Авторизация|Описание|
|--------|------|------|-------------|
|`GET`|`/api/admin/notifications`|Админ|Список уведомлений для текущего администратора|
|`POST`|`/api/admin/notifications`|Аутентифицированный|Создать новое уведомление|
|`PATCH`|`/api/admin/notifications/{id}/read`|Аутентифицированный|Отметить одно уведомление как прочитанное|
|`PATCH`|`/api/admin/notifications/mark-all-read`|Аутентифицированный|Отметить все уведомления как прочитанные|

## Аутентификация

Конечные точки уведомлений используют два уровня аутентификации:

**Только для администратора (список GET):** Требуется как аутентификация, так и роль администратора.

```typescript
const session = await auth();
if (!session?.user?.id) {
  return NextResponse.json({ success: false, error: 'Unauthorized' }, { status: 401 });
}
if (!session.user.isAdmin) {
  return NextResponse.json({ success: false, error: 'Forbidden' }, { status: 403 });
}
```

**Пользователь, прошедший проверку подлинности (POST, PATCH):** Требуется действительный сеанс, но не требуется роль администратора. Конечные точки, отмеченные как прочитанные, относятся к собственным уведомлениям аутентифицированного пользователя.

## Конечные точки

### ПОЛУЧИТЬ `/api/admin/notifications`

Извлекает последние 50 уведомлений для аутентифицированного пользователя-администратора, отсортированные по дате создания (сначала самые новые). Также возвращает общее количество непрочитанных уведомлений.

**Ответ (200):**

```json
{
  "success": true,
  "data": {
    "notifications": [
      {
        "id": "notif_123abc",
        "userId": "user_456def",
        "type": "item_approved",
        "title": "Item Approved",
        "message": "Your item 'Awesome Tool' has been approved and is now live.",
        "data": "{\"itemId\": \"item_789ghi\", \"itemName\": \"Awesome Tool\"}",
        "isRead": false,
        "readAt": null,
        "createdAt": "2024-01-20T10:30:00.000Z",
        "updatedAt": "2024-01-20T10:30:00.000Z"
      }
    ],
    "unreadCount": 3
  }
}
```

**Детали поведения:**
- На каждый запрос возвращается максимум 50 уведомлений.
- Результаты упорядочены по `createdAt` по убыванию (самые новые сначала)
- `unreadCount` рассчитывается отдельно путем подсчета уведомлений, где `isRead = false`
- Уведомления привязаны к идентификатору аутентифицированного пользователя.

### ПОСТ `/api/admin/notifications`

Создает новое уведомление для указанного пользователя. Поле `data` принимает объект, который перед сохранением будет преобразован в строку JSON. Эта конечная точка не требует прав администратора — любой прошедший проверку подлинности пользователь может создавать уведомления (обычно вызываемые внутренней системой).

**Тело запроса:**

```json
{
  "type": "item_approved",
  "title": "Item Approved",
  "message": "Your item 'Awesome Tool' has been approved and is now live.",
  "userId": "user_456def",
  "data": {
    "itemId": "item_789ghi",
    "itemName": "Awesome Tool",
    "action": "approved"
  }
}
```

|Поле|Тип|Требуется|Описание|
|-------|------|----------|-------------|
|`type`|строка|Да|Идентификатор типа уведомления (например, `"item_approved"`, `"comment_received"`)|
|`title`|строка|Да|Краткое название уведомления|
|`message`|строка|Да|Полное уведомление|
|`userId`|строка|Да|Целевой идентификатор пользователя для получения уведомления|
|`data`|объект|Нет|Дополнительные метаданные (в формате JSON в хранилище)|

**Ответ (200):**

```json
{
  "success": true,
  "notification": {
    "id": "notif_123abc",
    "userId": "user_456def",
    "type": "item_approved",
    "title": "Item Approved",
    "message": "Your item 'Awesome Tool' has been approved and is now live.",
    "data": "{\"itemId\": \"item_789ghi\", \"itemName\": \"Awesome Tool\", \"action\": \"approved\"}",
    "isRead": false,
    "readAt": null,
    "createdAt": "2024-01-20T10:30:00.000Z",
    "updatedAt": "2024-01-20T10:30:00.000Z"
  }
}
```

### ПАТЧ `/api/admin/notifications/{id}/read`

Отмечает конкретное уведомление как прочитанное. Устанавливает `isRead` на `true`, записывает текущую временную метку в `readAt` и обновляет `updatedAt`. Только владелец уведомления может отмечать свои собственные уведомления — запрос фильтруется как по идентификатору уведомления, так и по идентификатору аутентифицированного пользователя.

**Параметры пути:**

|Параметр|Тип|Описание|
|-----------|------|-------------|
|`id`|строка|Уникальный идентификатор уведомления|

**Ответ (200):**

```json
{
  "success": true,
  "notification": {
    "id": "notif_123abc",
    "userId": "user_456def",
    "type": "item_approved",
    "title": "Item Approved",
    "message": "Your item 'Awesome Tool' has been approved and is now live.",
    "isRead": true,
    "readAt": "2024-01-20T16:45:00.000Z",
    "createdAt": "2024-01-20T10:30:00.000Z",
    "updatedAt": "2024-01-20T16:45:00.000Z"
  }
}
```

### ПАТЧ `/api/admin/notifications/mark-all-read`

Помечает все непрочитанные уведомления для аутентифицированного пользователя как прочитанные в рамках одной массовой операции. Обновления `isRead`, `readAt` и `updatedAt` для каждого соответствующего уведомления. Возвращает количество обновленных уведомлений.

**Ответ (200):**

```json
{
  "success": true,
  "updatedCount": 5
}
```

**Детали поведения:**
- Обновляет только уведомления, где `isRead = false` для текущего пользователя
- `updatedCount` может быть `0`, если нет непрочитанных уведомлений
- Все соответствующие уведомления обновляются в одном запросе к базе данных.

## Модель данных уведомлений

|Поле|Тип|Обнуляемый|Описание|
|-------|------|----------|-------------|
|`id`|строка|Нет|Уникальный идентификатор уведомления|
|`userId`|строка|Нет|Идентификатор пользователя, получившего уведомление|
|`type`|строка|Нет|Тип уведомления (например, `"item_approved"`, `"comment_received"`)|
|`title`|строка|Нет|Краткое отображаемое название|
|`message`|строка|Нет|Полное уведомление|
|`data`|строка|Да|Дополнительные метаданные в формате JSON|
|`isRead`|логическое значение|Нет|Прочитано ли уведомление|
|`readAt`|дата и время|Да|Временная метка при пометке как прочитанная|
|`createdAt`|дата и время|Нет|Временная метка создания|
|`updatedAt`|дата и время|Да|Временная метка последнего обновления|

## Коды ошибок

|Статус|Ошибка|Причина|
|--------|-------|-------|
| `400` |Отсутствуют обязательные поля|В POST отсутствует тип, заголовок, сообщение или идентификатор пользователя.|
| `400` |Укажите идентификатор уведомления.|PATCH с пустым параметром ID|
| `401` |Несанкционированный|Нет активной сессии|
| `403` |Запрещено|Пользователь без прав администратора в конечной точке списка GET|
| `404` |Уведомление не найдено|Неверный идентификатор или уведомление принадлежит другому пользователю.|
| `500` |Внутренняя ошибка сервера|Сбой базы данных или сервера|

## Распространенные типы уведомлений

Поле `type` представляет собой строку произвольной формы, но приложение обычно использует следующие значения:

|Тип|Описание|
|------|-------------|
|`item_approved`|Товар одобрен администратором|
|`item_rejected`|Товар отклонен|
|`comment_received`|К элементу был опубликован новый комментарий|
|`submission_received`|Получен новый товар|
