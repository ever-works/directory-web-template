# Справочник по API флагов конфигурации и функций

## Обзор

Конечная точка Config Features предоставляет текущие флаги доступности функций для приложения. Эти флаги указывают, какие функции, зависящие от базы данных, активны, позволяя корректно снижать производительность внешнего интерфейса, когда функции недоступны. Это общедоступная кэшированная конечная точка, предназначенная для высокочастотного потребления.

## Конечные точки

### ПОЛУЧИТЬ /api/config/features

Возвращает текущую доступность функции на основе конфигурации системы и доступности базы данных.

**Запрос**

Никаких параметров или тела не требуется.

**Ответ**
```typescript
{
  ratings: boolean;         // Whether the ratings feature is available
  comments: boolean;        // Whether the comments feature is available
  favorites: boolean;       // Whether the favorites feature is available
  featuredItems: boolean;   // Whether the featured items feature is available
  surveys: boolean;         // Whether the surveys feature is available
}
```

**Пример**
```typescript
const response = await fetch('/api/config/features');
const features = await response.json();

if (features.ratings) {
  // Render rating component
}

if (!features.surveys) {
  // Hide survey section
}
```

## Аутентификация

Эта конечная точка является **общедоступной** – аутентификация не требуется. Он предназначен для использования интерфейсом при начальной загрузке страницы, чтобы определить, какие функции пользовательского интерфейса следует отобразить.

## Реакции на ошибки

|Статус|Описание|
|--------|-------------|
| 200 |Флаги функций успешно получены.|
| 500 |Внутренняя ошибка — для безопасности возвращаются все флаги как `false` с заголовком `no-cache`.|

В случае ошибки конечная точка возвращает все функции как `false`, чтобы гарантировать безопасный сбой приложения, а не раскрывать нарушенную функциональность.

## Ограничение скорости

Ответы кэшируются со следующими заголовками:
- `Cache-Control: public, s-maxage=300, stale-while-revalidate=600`
- Эффективно кэшируется в течение 5 минут на уровне CDN с 10-минутным окном устаревания при повторной проверке.

В ответах на ошибки используется `Cache-Control: no-cache` для предотвращения кэширования ухудшенного состояния.
