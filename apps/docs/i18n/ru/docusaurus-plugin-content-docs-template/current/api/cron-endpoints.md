# Конечные точки API заданий Cron

Шаблон включает три конечных точки задания cron, которые запускаются через запланированные интервалы через Vercel Cron. Эти конечные точки обрабатывают синхронизацию контента, напоминания о подписке и обработку истечения срока действия подписки.

## Конфигурация Крон

Расписания Cron определены в `vercel.json`:

```json
{
  "crons": [
    {
      "path": "/api/cron/sync",
      "schedule": "0 3 * * *"
    },
    {
      "path": "/api/cron/subscription-reminders",
      "schedule": "0 9 * * *"
    },
    {
      "path": "/api/cron/subscription-expiration",
      "schedule": "0 0 * * *"
    }
  ]
}
```

## Синхронизация контента (`/api/cron/sync`)

|Метод|Путь|Расписание|Описание|
|--------|------|----------|-------------|
|`GET`|`/api/cron/sync`|Ежедневно в 3:00 утра по всемирному координированному времени|Синхронизировать репозиторий контента на базе Git.|

### Что он делает

Задание синхронизации извлекает последний контент из настроенного репозитория данных Git (`DATA_REPOSITORY`) и обновляет локальный кеш контента. Это гарантирует, что приложение будет отражать любые изменения, внесенные непосредственно в репозиторий контента (например, посредством слияния GitHub PR).

### Процесс синхронизации

```
1. Verify CRON_SECRET authorization
2. Check if sync is already in progress (mutex lock)
3. Pull latest changes from remote Git repository
4. Parse and validate updated YAML content files
5. Update local content cache
6. Return sync result with duration
```

### Ключевые модели поведения

- **Блокировка мьютекса**: одновременно может выполняться только одна синхронизация. Одновременные запросы отклоняются с сообщением о состоянии
- **Тайм-аут**: операции синхронизации имеют 5-минутный тайм-аут, чтобы предотвратить неконтролируемые процессы.
- **Логика повтора**: неудачная синхронизация повторяется до 3 раз.
- **Режим разработки**: автосинхронизацию можно отключить с помощью переменной среды `DISABLE_AUTO_SYNC=true`.

### Ответ

```json
{
  "success": true,
  "message": "Sync completed successfully",
  "duration": 4523
}
```

## Напоминания о подписке (`/api/cron/subscription-reminders`)

|Метод|Путь|Расписание|Описание|
|--------|------|----------|-------------|
|`GET`|`/api/cron/subscription-reminders`|Ежедневно в 9:00 UTC|Отправлять напоминания о продлении подписки|

### Что он делает

Запрашивает подписки, приближающиеся к дате продления, и отправляет подписчикам напоминания по электронной почте. Это помогает уменьшить непроизвольный отток клиентов, предупреждая пользователей до обработки их платежей.

### Напоминание Логика

```
1. Verify CRON_SECRET authorization
2. Query subscriptions renewing within reminder window
3. Filter out already-notified subscriptions
4. Send reminder emails via email notification service
5. Mark subscriptions as notified
6. Return count of reminders sent
```

### Напоминание Windows

Типичные окна напоминаний:
- **7 дней до продления**: первое напоминание.
- **За 1 день до продления**: последнее напоминание.

### Ответ

```json
{
  "success": true,
  "message": "Subscription reminders sent",
  "data": {
    "reminders_sent": 15,
    "errors": 0
  }
}
```

## Срок действия подписки (`/api/cron/subscription-expiration`)

|Метод|Путь|Расписание|Описание|
|--------|------|----------|-------------|
|`GET`|`/api/cron/subscription-expiration`|Ежедневно в полночь UTC|Обработка подписок с истекшим сроком действия|

### Что он делает

Определяет подписки, срок действия которых истек, и обновляет их статус. При этом обрабатываются подписки, которые были отменены, но у которых осталось время, а также подписки, которые не удалось продлить.

### Процесс истечения срока действия

```
1. Verify CRON_SECRET authorization
2. Query subscriptions with expiration date in the past
3. Update subscription status to 'expired'
4. Revoke associated access/permissions
5. Send expiration notification emails
6. Log expiration events for audit trail
7. Return count of processed expirations
```

### Ответ

```json
{
  "success": true,
  "message": "Subscription expirations processed",
  "data": {
    "expired": 3,
    "errors": 0
  }
}
```

## Фоновые задания (`/api/cron/jobs`)

Файл `background-jobs-init.ts` в каталоге заданий cron инициализирует обработку фоновых заданий. Это настраивает любые повторяющиеся задачи, которые необходимо выполнять во время выполнения приложения.

## Безопасность

### CRON_SECRET Проверка

Все конечные точки cron проверяют заголовок `CRON_SECRET` или параметр запроса, чтобы предотвратить несанкционированное выполнение:

```typescript
// Typical cron authorization check
const authHeader = request.headers.get('authorization');
if (authHeader !== `Bearer ${process.env.CRON_SECRET}`) {
  return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
}
```

### Авторизация Vercel Cron

При развертывании на Vercel задания cron автоматически вызываются планировщиком cron Vercel с соответствующим заголовком `CRON_SECRET`. Секрет настраивается на панели управления Vercel в настройках проекта.

|Переменная среды|Описание|
|---------------------|-------------|
|`CRON_SECRET`|Общий секрет для авторизации заданий cron|

### Ручное выполнение

Конечные точки Cron можно запустить вручную для отладки, включив `CRON_SECRET` в заголовок авторизации:

```bash
curl -H "Authorization: Bearer your-cron-secret" \
  https://your-app.vercel.app/api/cron/sync
```

## Мониторинг

### Статус синхронизации

Статус задания синхронизации cron можно отслеживать с помощью:
- `/api/version/sync` — возвращает время и результат последней синхронизации.
- Журналы сервера — операции синхронизации регистрируются с префиксом `[SYNC_MANAGER]`.

### Обработка ошибок

Все задания cron реализуют комплексную обработку ошибок:
- Неудачные операции регистрируются с полной информацией об ошибках.
- Частичные сбои не мешают обработке оставшихся элементов.
- Количество ошибок включается в ответ для мониторинга.
- Критические сбои вызывают ошибки консоли для предупреждений агрегирования журналов.

## Ссылка на расписание

|Выражение Крон|Значение|
|----------------|---------|
| `0 3 * * *` |Каждый день в 3:00 утра по всемирному координированному времени.|
| `0 9 * * *` |Каждый день в 9:00 UTC|
| `0 0 * * *` |Каждый день в полночь UTC|

Все время указано в формате UTC. При настройке этих расписаний учитывайте распределение часовых поясов вашей базы пользователей.
