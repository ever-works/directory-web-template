# Система модерации

Система модерации обеспечивает программную модерацию контента через уровень обслуживания, а не через отдельные конечные точки API. Действия модерации запускаются автоматически, когда администраторы разрешают отчеты о контенте через API отчетов. Система поддерживает предупреждение пользователей, приостановку учетных записей, блокировку учетных записей и удаление контента с полной историей аудита и уведомлениями по электронной почте.

## Обзор

Модерация не предоставляется как отдельные конечные точки REST. Вместо этого он вызывается посредством рабочего процесса разрешения отчета:

```
PUT /api/admin/reports/[id]  -->  resolution triggers moderation action
```

Когда администратор устанавливает в отчете значение `resolution`, соответствующая функция модерации выполняется автоматически.

|Значение разрешения|Функция модерации|Эффект|
|---|---|---|
|`content_removed`|`removeContent()`|Мягкое удаление комментария или элемента, на который поступила жалоба.|
|`user_warned`|`warnUser()`|Увеличивает количество предупреждений пользователя.|
|`user_suspended`|`suspendUser()`|Устанавливает статус пользователя `"suspended"`|
|`user_banned`|`banUser()`|Устанавливает статус пользователя `"banned"`|
|`no_action`|Нет|Никаких действий по модерации не предпринято|

## Действия модерации

### Удалить контент

```typescript
removeContent(contentType, contentId, reportId, adminId): Promise<ModerationResult>
```

Удаляет указанный контент в зависимости от его типа. Для комментариев выполняется мягкое удаление (устанавливается `deletedAt`). Для элементов это удаляет элемент из репозитория контента на основе Git.

**Параметры:**

|Параметр|Тип|Описание|
|---|---|---|
|`contentType`|`"item"` или `"comment"`|Тип контента, который нужно удалить|
|`contentId`|строка|Идентификатор или фрагмент контента|
|`reportId`|строка|Идентификатор связанного отчета|
|`adminId`|строка|Пользователь-администратор, выполняющий действие|

**Этапы обработки:**

1. Найдите владельца контента через `getContentOwner()`
2. Если комментарий: мягкое удаление через `deleteComment()`
3. Если элемент: удалить из репозитория Git через `itemRepository.delete()`
4. Записать историю модерации с действием `CONTENT_REMOVED`
5. Отправьте электронное письмо с уведомлением об удалении контента владельцу контента.

**Источник:** `template/lib/services/moderation.service.ts`

### Предупредить пользователя

```typescript
warnUser(userId, reason, reportId, adminId): Promise<ModerationResult>
```

Выдает предупреждение пользователю, увеличивая его поле `warningCount`. Пользователи, которые уже забанены, не могут получать предупреждения.

**Параметры:**

|Параметр|Тип|Описание|
|---|---|---|
|`userId`|строка|Идентификатор клиентского профиля пользователя|
|`reason`|строка|Причина предупреждения|
|`reportId`|строка|Идентификатор связанного отчета|
|`adminId`|строка|Пользователь-администратор, выполняющий действие|

**Этапы обработки:**

1. Убедитесь, что пользователь существует и еще не забанен
2. Увеличение количества предупреждений через `incrementWarningCount()`
3. Записать историю модерации с действием `WARN`
4. Отправка предупреждающего уведомления по электронной почте с указанием текущего количества предупреждений.

**Результат успеха:**

```json
{
  "success": true,
  "message": "User warned successfully. Total warnings: 3"
}
```

**Источник:** `template/lib/services/moderation.service.ts`

### Заблокировать пользователя

```typescript
suspendUser(userId, reason, reportId, adminId): Promise<ModerationResult>
```

Приостанавливает учетную запись пользователя, устанавливая для нее статус `"suspended"` и записывая временную метку `suspendedAt`. Заблокированные пользователи не могут оставлять комментарии, отправлять голоса или отправлять отчеты.

**Охранники:**

- Возвращает ошибку, если пользователь уже заблокирован
- Возвращает ошибку, если пользователь уже забанен

**Этапы обработки:**

1. Убедитесь, что пользователь существует и не заблокирован или не заблокирован.
2. Установите статус `"suspended"` с отметкой времени `suspendedAt`.
3. Записать историю модерации с действием `SUSPEND`
4. Отправить уведомление о блокировке по электронной почте

**Источник:** `template/lib/services/moderation.service.ts`

### Заблокировать пользователя

```typescript
banUser(userId, reason, reportId, adminId): Promise<ModerationResult>
```

Навсегда блокирует учетную запись пользователя, устанавливая ее статус `"banned"` и записывая временную метку `bannedAt`. Забаненным пользователям блокируются все действия, прошедшие проверку подлинности.

**Охранники:**

- Возвращает ошибку, если пользователь уже забанен

**Этапы обработки:**

1. Убедитесь, что пользователь существует и еще не забанен
2. Установите статус `"banned"` с отметкой времени `bannedAt`.
3. Записать историю модерации с действием `BAN`
4. Отправить уведомление о запрете по электронной почте

**Источник:** `template/lib/services/moderation.service.ts`

## Разрешение владельца контента

Функция `getContentOwner()` определяет, кому принадлежит сообщаемый контент:

|Тип контента|Источник владельца|
|---|---|
|`comment`|Поле `comment.userId` из таблицы комментариев|
|`item`|Поле `item.submitted_by` из репозитория элементов|

Это используется всеми действиями модерации на уровне пользователя (`user_warned`, `user_suspended`, `user_banned`) для идентификации целевого пользователя для действия.

**Источник:** `template/lib/services/moderation.service.ts`

## История модерации

Все действия модерации создают контрольный журнал в таблице базы данных `moderationHistory`.

### Поля записи истории

|Поле|Тип|Описание|
|---|---|---|
|`id`|строка|Уникальный идентификатор записи|
|`userId`|строка|Идентификатор профиля клиента затронутого пользователя|
|`action`|строка|`"CONTENT_REMOVED"`, `"WARN"`, `"SUSPEND"` или `"BAN"`|
|`reason`|строка или ноль|Причина модерации|
|`reportId`|строка или ноль|Идентификатор связанного отчета|
|`performedBy`|строка или ноль|Идентификатор пользователя-администратора, выполнившего действие|
|`contentType`|строка или ноль|`"item"` или `"comment"` (для удаления контента)|
|`contentId`|строка или ноль|Идентификатор удаленного контента|
|`details`|объект или ноль|Дополнительный контекст (например, количество предупреждений, имя элемента)|
|`createdAt`|временная метка|Когда действие было совершено|

### Исторические запросы

|Функция|Описание|
|---|---|
|`getModerationHistoryByUser(userId, limit)`|Получить все действия модерации для пользователя (ограничение по умолчанию: 50).|
|`getModerationHistoryByReport(reportId)`|Получайте действия модерации, связанные с конкретным отчетом|

Обе функции запроса дополняют результаты информацией профиля пользователя и сведениями об исполнителе-администраторе.

**Источник:** `template/lib/db/queries/moderation.queries.ts`

## Управление статусом пользователя

### Значения статуса

|Статус|Описание|
|---|---|
|`active`|Обычный аккаунт, доступны все функции|
|`suspended`|Временно ограничено, невозможно создавать контент|
|`banned`|Постоянно ограничено, заблокировано для всех действий|

### Операции с базой данных

|Функция|Описание|
|---|---|
|`suspendUser(userId)`|Устанавливает статус `"suspended"`, записывает `suspendedAt`|
|`unsuspendUser(userId)`|Восстанавливает статус `"active"`, очищает `suspendedAt`|
|`banUser(userId)`|Устанавливает статус `"banned"`, записывает `bannedAt`|
|`unbanUser(userId)`|Восстанавливает статус `"active"`, очищает `bannedAt`|
|`incrementWarningCount(userId)`|Увеличивает `warningCount` с помощью SQL `COALESCE`|

### Заблокированные пользовательские проверки

Две вспомогательные функции проверяют статус пользователя в приложении:

- **`isUserBlocked(status)`** -- Возвращает `true`, если статус `"suspended"` или `"banned"`
- **`getBlockReasonMessage(status)`** — возвращает пользователю сообщение, объясняющее, почему действие ограничено.

Эти проверки используются конечными точками комментариев, голосований и отчетов, чтобы запретить заблокированным пользователям создавать контент.

**Источник:** `template/lib/db/queries/moderation.queries.ts`

## Уведомления по электронной почте

`EmailNotificationService` отправляет неблокирующие уведомления о действиях модератора:

|Метод|Триггер|
|---|---|
|`sendContentRemovedEmail(email, type, reason)`|Контент удален администратором|
|`sendUserWarningEmail(email, reason, count)`|Предупреждение выдано|
|`sendUserSuspensionEmail(email, reason)`|Аккаунт заблокирован|
|`sendUserBanEmail(email, reason)`|Аккаунт заблокирован|

Все отправляемые электронные письма используют `.catch()`, чтобы сбои не прерывали рабочий процесс модерации. Неудачное электронное письмо не приводит к сбою самого действия модерации.

## Ключевые детали реализации

- **Шаблон уровня обслуживания:** Логика модерации находится в `lib/services/moderation.service.ts`, а не в обработчиках маршрутов API. Это позволяет повторно использовать данные в разных точках входа.
- **Аудитный журнал.** При каждом действии модератора создается запись `moderationHistory`, предоставляющая полный журнал аудита для соответствия и проверки.
- **Неблокирующие электронные письма.** Уведомления по электронной почте отправляются асинхронно с обработчиками `.catch()`. Если служба электронной почты недоступна, действие модерации все равно будет успешным.
- **Защита идемпотентности:** перед продолжением каждого действия проверяется текущий статус пользователя. Запрет уже забаненного пользователя возвращает ошибку, а не создает дублирующее действие.
- **Мягкое и жесткое удаление:** Комментарии удаляются мягко (настройка `deletedAt`), а элементы полностью удаляются из репозитория Git. Эта разница отражает модель хранения (база данных или файловый контент).
