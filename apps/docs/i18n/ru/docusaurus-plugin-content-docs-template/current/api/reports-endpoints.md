# Конечные точки отчетов

Система отчетов позволяет прошедшим проверку подлинности пользователям отмечать неприемлемый контент и предоставляет администраторам инструменты для просмотра, модерации и разрешения отчетов. Отчеты поддерживают типы контента, включая элементы и комментарии, со встроенной защитой от дублирования.

## Обзор

|Конечная точка|Метод|Авторизация|Описание|
|---|---|---|---|
|`/api/reports`|ПОСТ|Пользователь|Отправить отчет о содержании|
|`/api/admin/reports`|ПОЛУЧИТЬ|Админ|Получение списка отчетов с фильтрацией|
|`/api/admin/reports/stats`|ПОЛУЧИТЬ|Админ|Получить статистику отчета|
|`/api/admin/reports/[id]`|ПОЛУЧИТЬ|Админ|Получите единый отчет|
|`/api/admin/reports/[id]`|ПОСТАВИТЬ|Админ|Обновить статус и разрешение отчета|

## Публичные конечные точки

### Отправить отчет

```
POST /api/reports
```

Аутентифицированные пользователи могут жаловаться на элементы или комментарии о неприемлемом содержании. Каждый пользователь может сообщить об одном и том же контенте только один раз (предотвращение дублирования посредством проверки `hasUserReportedContent`). Заблокированные пользователи (приостановленные или забаненные) не могут отправлять отчеты.

**Аутентификация:** Требуется (на основе сеанса)

**Тело запроса:**

```json
{
  "contentType": "item",
  "contentId": "awesome-productivity-tool",
  "reason": "spam",
  "details": "This tool is promoting malicious software"
}
```

|Поле|Тип|Требуется|Описание|
|---|---|---|---|
|`contentType`|строка|Да|Тип контента: `"item"` или `"comment"`|
|`contentId`|строка|Да|Идентификатор или фрагмент содержимого, о котором сообщается|
|`reason`|строка|Да|Один из: `"spam"`, `"harassment"`, `"inappropriate"`, `"other"`|
|`details`|строка|Нет|Дополнительный контекст отчета|

**Успешный ответ (200):**

```json
{
  "success": true,
  "message": "Report submitted successfully",
  "report": {
    "id": "rpt_abc123",
    "contentType": "item",
    "contentId": "awesome-productivity-tool",
    "reason": "spam",
    "status": "pending",
    "createdAt": "2024-01-20T10:30:00.000Z"
  }
}
```

**Ответы об ошибках:**

|Статус|Состояние|
|---|---|
| 400 |Неверный тип контента, отсутствует идентификатор контента или неверная причина.|
| 401 |Пользователь не авторизован|
| 403 |Требуется профиль клиента, или пользователь заблокирован/забанен|
| 404 |Профиль клиента не найден|
| 409 |Пользователь уже сообщил об этом контенте|
| 500 |Внутренняя ошибка сервера|

**Источник:** `template/app/api/reports/route.ts`

## Конечные точки администратора

Все конечные точки администратора требуют, чтобы `session.user.isAdmin` имело значение true.

### Получение списка отчетов

```
GET /api/admin/reports
```

Возвращает постраничный список отчетов о содержании с информацией о репортере. Поддерживает фильтрацию по статусу, типу контента и причине, а также текстовый поиск по идентификатору контента, сведениям и имени/электронному адресу репортера.

**Параметры запроса:**

|Параметр|Тип|По умолчанию|Описание|
|---|---|---|---|
|`page`|целое число| 1 |Номер страницы (минимум 1)|
|`limit`|целое число| 10 |Результатов на странице (1–100)|
|`search`|строка| - |Поиск по идентификатору контента, сведениям, имени/электронному адресу репортера|
|`status`|строка| - |Фильтр: `"pending"`, `"reviewed"`, `"resolved"`, `"dismissed"`|
|`contentType`|строка| - |Фильтр: `"item"`, `"comment"`|
|`reason`|строка| - |Фильтр: `"spam"`, `"harassment"`, `"inappropriate"`, `"other"`|

**Успешный ответ (200):**

```json
{
  "success": true,
  "data": {
    "reports": [
      {
        "id": "rpt_abc123",
        "contentType": "item",
        "contentId": "some-item-slug",
        "reason": "spam",
        "status": "pending",
        "details": "Suspicious content",
        "reportedBy": "client_456",
        "createdAt": "2024-01-20T10:30:00.000Z"
      }
    ],
    "pagination": {
      "total": 42,
      "page": 1,
      "limit": 10,
      "totalPages": 5
    }
  }
}
```

**Источник:** `template/app/api/admin/reports/route.ts`

### Получить статистику отчета

```
GET /api/admin/reports/stats
```

Возвращает совокупную статистику об отчетах, включая количество по статусу, типу контента и причине.

**Успешный ответ (200):**

```json
{
  "success": true,
  "data": {
    "total": 156,
    "pendingCount": 23,
    "resolvedCount": 120,
    "byStatus": {
      "pending": 23,
      "reviewed": 10,
      "resolved": 120,
      "dismissed": 3
    },
    "byContentType": {
      "item": 100,
      "comment": 56
    },
    "byReason": {
      "spam": 80,
      "inappropriate": 45,
      "harassment": 20,
      "other": 11
    }
  }
}
```

**Источник:** `template/app/api/admin/reports/stats/route.ts`

### Получить отчет по идентификатору

```
GET /api/admin/reports/[id]
```

Получает единый отчет с полной информацией, включая информацию о репортере и рецензенте.

**Параметры пути:**

|Параметр|Тип|Описание|
|---|---|---|
|`id`|строка|Идентификатор отчета|

**Успешный ответ (200):**

```json
{
  "success": true,
  "data": {
    "id": "rpt_abc123",
    "contentType": "item",
    "contentId": "some-item-slug",
    "reason": "spam",
    "status": "reviewed",
    "details": "Suspicious content",
    "reportedBy": "client_456",
    "reviewedBy": "admin_789",
    "reviewNote": "Confirmed as spam",
    "resolution": "content_removed",
    "createdAt": "2024-01-20T10:30:00.000Z",
    "updatedAt": "2024-01-21T09:00:00.000Z"
  }
}
```

|Статус|Состояние|
|---|---|
| 403 |Не администратор|
| 404 |Отчет не найден|

**Источник:** `template/app/api/admin/reports/[id]/route.ts`

### Обновить отчет

```
PUT /api/admin/reports/[id]
```

Обновляет статус отчета, разрешение и примечание к проверке. Когда разрешение установлено, система автоматически выполняет соответствующее действие модерации (удаление контента, предупреждение пользователя, приостановка или бан).

**Тело запроса:**

```json
{
  "status": "resolved",
  "resolution": "content_removed",
  "reviewNote": "Confirmed spam content, removed from listing"
}
```

|Поле|Тип|Требуется|Описание|
|---|---|---|---|
|`status`|строка|Нет|`"pending"`, `"reviewed"`, `"resolved"`, `"dismissed"`|
|`resolution`|строка|Нет|`"content_removed"`, `"user_warned"`, `"user_suspended"`, `"user_banned"`, `"no_action"`|
|`reviewNote`|строка|Нет|Примечания администратора об отзыве|

**Действия модератора по решению:**

В зависимости от значения разрешения запускаются следующие автоматические действия:

|Разрешение|Действие|
|---|---|
|`content_removed`|Вызывает `removeContent()`, чтобы удалить элемент или комментарий, о котором сообщили.|
|`user_warned`|Вызывает `warnUser()`, чтобы выдать предупреждение владельцу контента.|
|`user_suspended`|Вызывает `suspendUser()`, чтобы заблокировать учетную запись владельца контента.|
|`user_banned`|Вызывает `banUser()`, чтобы навсегда заблокировать владельца контента.|
|`no_action`|Никаких действий по модерации не предпринимается|

**Успешный ответ (200):**

```json
{
  "success": true,
  "message": "Report updated successfully",
  "data": {
    "id": "rpt_abc123",
    "status": "resolved",
    "resolution": "content_removed",
    "reviewNote": "Confirmed spam content"
  },
  "moderationResult": {
    "success": true,
    "message": "Content removed successfully"
  }
}
```

|Статус|Состояние|
|---|---|
| 400 |Неверный статус или значение разрешения; владелец контента не найден для действий на уровне пользователя|
| 403 |Не администратор|
| 404 |Отчет не найден|

**Источник:** `template/app/api/admin/reports/[id]/route.ts`

## Модель данных

В отчетах используются следующие перечисления, определенные в `lib/db/schema`:

- **ReportContentType:** `"item"`, `"comment"`
- **ReportReason:** `"spam"`, `"harassment"`, `"inappropriate"`, `"other"`
- **Состояние отчета:** `"pending"`, `"reviewed"`, `"resolved"`, `"dismissed"`
- **Решение отчета:** `"content_removed"`, `"user_warned"`, `"user_suspended"`, `"user_banned"`, `"no_action"`

## Интеграция с модерацией

Когда отчет разрешается с разрешением на уровне пользователя (`user_warned`, `user_suspended`, `user_banned`), система:

1. Ищет владельца контента через `getContentOwner()`.
2. Выполняет соответствующую функцию модерации от `lib/services/moderation.service`
3. Использует `reviewNote` в качестве причины модерации.
4. Записывает идентификатор администратора в качестве рецензента.

Если действие модерации завершается неудачей, обновление отчета все равно завершается успешно, но сбой регистрируется. Поле `moderationResult` в ответе указывает, удалось ли действие.
