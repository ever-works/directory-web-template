# Журнал Изменений

На этой странице объясняется, как Directory Web Template управляет версионированием, релизами и путями обновления.

## Семантическое Версионирование

Шаблон следует [Semantic Versioning (SemVer)](https://semver.org/). Номера версий используют формат **MAJOR.MINOR.PATCH**:

| Компонент  | Когда увеличивать                                               |
| ---------- | --------------------------------------------------------------- |
| **MAJOR**  | Обратно несовместимые изменения, требующие шагов миграции       |
| **MINOR**  | Новые функции, добавленные с соблюдением обратной совместимости |
| **PATCH**  | Обратно совместимые исправления ошибок и незначительные улучшения |

Предварительные версии могут использовать суффиксы, такие как `-alpha.1`, `-beta.2` или `-rc.1` для раннего тестирования.

## Миграции Базы Данных

Шаблон использует **Drizzle ORM** с PostgreSQL. Изменения схемы базы данных управляются через Drizzle Kit:

```bash
# Generate migration files from schema changes
pnpm db:generate

# Apply migrations to the database
pnpm db:migrate

# Open Drizzle Studio for visual database management
pnpm db:studio
```

Файлы миграции хранятся в каталоге `lib/db/migrations/`. Каждая миграция — это SQL-файл, сгенерированный из изменений в определениях схемы Drizzle в `lib/db/schema/`.

## Обновление Шаблона

При обновлении до более новой версии:

```bash
cd directory-web-template

# Pull latest changes
git pull origin main

# Install updated dependencies
pnpm install

# Apply database migrations
pnpm db:migrate

# Verify build
pnpm build
```

### Разрешение Конфликтов при Обновлении

Если вы настроили Шаблон, при получении обновлений могут возникнуть конфликты слияния. Рекомендуемый подход:

1. **Держите настройки в отдельных файлах**, когда это возможно (пользовательские компоненты, новые маршруты, дополнительные сервисы).
2. **Используйте CMS на основе Git** для изменений контента вместо изменения основных файлов.
3. **Просматривайте примечания к релизу** перед обновлением, чтобы понять, какие файлы изменились.
4. **Тщательно тестируйте** после устранения конфликтов, запустив `pnpm lint`, `pnpm tsc --noEmit` и `pnpm build`.

## Отслеживание Релизов

### GitHub Releases

Релизы публикуются на GitHub по адресу [github.com/ever-works/directory-web-template/releases](https://github.com/ever-works/directory-web-template/releases).

Каждый релиз включает:

- Тег версии (например, `v0.1.0`)
- Примечания к релизу с описанием изменений, новых функций, исправлений ошибок и критических изменений
- Ссылки на соответствующие pull request'ы и issues

### История Коммитов

Репозиторий использует [Conventional Commits](https://www.conventionalcommits.org/), что упрощает просмотр истории коммитов в поисках изменений:

```bash
# View recent commits with conventional commit prefixes
git log --oneline --since="2025-01-01"

# Filter for feature commits only
git log --oneline --grep="^feat:"

# Filter for breaking changes
git log --oneline --grep="BREAKING CHANGE"
```

## Политика Критических Изменений

Критические изменения принимаются всерьёз. Проект придерживается следующих принципов:

1. **Предварительное уведомление.** Критические изменения объявляются как минимум за один минорный релиз до вступления в силу, когда это возможно.
2. **Руководства по миграции.** Каждое критическое изменение включает руководство по миграции в примечаниях к релизу.
3. **Минимизация нарушений.** Критические изменения объединяются в мажорные релизы, а не распределяются по нескольким минорным релизам.
4. **Обратная совместимость базы данных.** Миграции разработаны так, чтобы быть неразрушающими. Добавление столбцов и создание таблиц предпочтительнее удалений или переименований.

### Примеры Критических Изменений

- Удаление или переименование публичного эндпоинта API
- Изменение структуры тел запросов или ответов API
- Удаление или переименование столбцов или таблиц базы данных
- Изменение обязательных переменных окружения
- Прекращение поддержки версии Node.js
- Изменение поведения аутентификации или авторизации
- Удаление или переименование экспортированных типов или интерфейсов TypeScript

### Примеры Некритических Изменений

- Добавление новых эндпоинтов API
- Добавление новых необязательных полей в тела запросов или ответов
- Добавление новых столбцов базы данных со значениями по умолчанию
- Добавление новых переменных окружения с разумными значениями по умолчанию
- Добавление новых функций или интеграций
- Улучшения производительности
- Исправления ошибок

## Формат Журнала Изменений

Примечания к релизу имеют следующую структуру:

```markdown
## [0.2.0] - 2025-04-15

### Added

- Category-based directory filtering
- New Polar payment provider integration

### Changed

- Improved authentication flow with better error messages

### Fixed

- Resolved race condition in concurrent directory updates
- Fixed pagination offset calculation for search results

### Deprecated

- Legacy REST endpoints under /api/v1/ (use /api/v2/ instead)

### Breaking Changes

- Removed `LEGACY_AUTH_MODE` environment variable
- Renamed `DirectoryItem` type to `Item` across all APIs
```

Этот формат следует конвенциям [Keep a Changelog](https://keepachangelog.com/).
