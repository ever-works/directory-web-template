# Конфигурация Мультиарендности

Этот документ объясняет, как работает поддержка мультиарендности в Directory Web Template.

## Обзор

Шаблон использует подход **общей базы данных с изоляцией на уровне строк**:

- Одна база данных PostgreSQL обслуживает несколько **арендаторов** (сайты-каталоги).
- Каждая таблица содержит столбец `tenant_id`, ограничивающий данные конкретным арендатором.
- Все запросы автоматически фильтруются по текущему арендатору — утечек данных между арендаторами нет.

## Быстрая Настройка

### 1. Установка Переменной Окружения

На платформе развёртывания (Vercel, Docker и т.д.) или в `.env.local`:

```bash
TENANT_ID="your-unique-tenant-id"
```

Это может быть любая уникальная строка (например, UUID или читаемый slug вроде `"my-directory"`).

### 2. Развёртывание

При первом запуске приложение:

1. Выполнит миграции базы данных (добавит столбец `tenant_id`, если он отсутствует)
2. Создаст строку арендатора, соответствующую значению `TENANT_ID`
3. Перенесёт существующие данные с NULL в `tenant_id` к вашему арендатору
4. Загрузит данные по умолчанию (пользователь-администратор, роли, разрешения)

Ручной SQL не требуется — всё происходит автоматически.

### 3. Проверка

Проверьте логи сервера на наличие:

```
[DB Init] Ensured environment tenant 'your-unique-tenant-id' exists
[Tenant Migration] ✓ users: updated 3 rows
[Tenant Migration] ✅ Migration complete: 15 total rows updated across all tables.
```

## Как Работает Определение Арендатора

Когда приложению нужно определить текущего арендатора, оно использует **каскадную** стратегию:

| Приоритет | Источник          | Описание                                                          |
| --------- | ----------------- | ----------------------------------------------------------------- |
| 1         | **Сессия**        | `user.tenantId` из JWT-токена (аутентифицированные пользователи) |
| 2         | **Переменная Env**| Переменная окружения `TENANT_ID`                                  |
| 3         | **HTTP-заголовок**| Заголовок `x-tenant-domain` (для маршрутизации по поддоменам)     |
| 4         | **База данных**   | Первая активная строка арендатора (последний резервный вариант)   |

Функция `getTenantId()` из `lib/auth/tenant.ts` реализует эту цепочку и вызывается при каждом запросе к базе данных.

## Архитектура

### Ключевые Файлы

| Файл                                     | Назначение                                                                        |
| ---------------------------------------- | --------------------------------------------------------------------------------- |
| `lib/auth/tenant.ts`                     | `getTenantId()` — серверное определение арендатора с кешированием                 |
| `lib/config/env.ts`                      | Валидация переменной окружения `TENANT_ID`                                        |
| `lib/db/schema.ts`                       | Таблица арендаторов + FK `tenant_id` во всех таблицах                             |
| `lib/db/initialize.ts`                   | Автоматически создаёт арендатора окружения + выполняет миграцию данных при запуске |
| `lib/db/migrate-tenant-data.ts`          | Присваивает строки с NULL в `tenant_id` текущему арендатору                       |
| `lib/auth/index.ts`                      | Callback-функции JWT/сессии внедряют `tenantId`                                   |
| `components/context/tenant-provider.tsx` | Контекст React для доступа к арендатору на стороне клиента                        |
| `app/api/tenant/route.ts`                | `GET /api/tenant` — возвращает информацию о текущем арендаторе                    |

### Поток Данных

```
Запрос пользователя → getTenantId() → Определяется из сессии/env/заголовков/БД
                                                ↓
                            Все запросы к БД фильтруются по этому tenant_id
                                                ↓
                            Возвращаются только данные для этого арендатора
```

### Интеграция с Аутентификацией

- **Вход по учётным данным**: Администраторы и клиенты получают `tenantId` из столбца `users.tenant_id`.
- **Вход через OAuth**: Адаптер Drizzle обёрнут для внедрения `tenantId` при создании пользователя.
- **Callback JWT**: Считывает `tenantId` из записи пользователя и встраивает его в токен.
- **Callback сессии**: Передаёт `tenantId` в `session.user.tenantId`.
- **Клиентские компоненты**: Используют хук `useTenant()` из `TenantProvider` для получения информации об арендаторе.

## Несколько Каталогов (Мультиарендность)

Для запуска нескольких сайтов-каталогов на одной базе данных:

1. **Каждый сайт** устанавливает отличный `TENANT_ID` в своём окружении:
    - Сайт A: `TENANT_ID="directory-a-uuid"`
    - Сайт B: `TENANT_ID="directory-b-uuid"`

2. **Все сайты** подключаются к **одной базе данных** (`DATABASE_URL`).

3. **Изоляция данных** происходит автоматически — Сайт A видит только строки, где `tenant_id = 'directory-a-uuid'`.

4. **Пользователи, роли, комментарии, подписки** и все остальные данные полностью изолированы по арендатору.

## Обработка Существующих Данных

При обновлении с версии без мультиарендности:

- Столбец `tenant_id` добавляется как **nullable** (не ломает существующие данные)
- При первом запуске `migrateNullTenantIds()` автоматически присваивает строки с NULL определённому арендатору
- Эта миграция **идемпотентна** — безопасна для многократного запуска
- После миграции все существующие данные видны под текущим арендатором

## Маршрутизация по Поддоменам (Расширенная)

Для маршрутизации арендаторов на основе поддоменов (например, `tenant-a.example.com`):

1. Настройте обратный прокси для добавления заголовка `x-tenant-domain`
2. Создайте записи арендаторов с полями `domain` или `slug`:
    ```sql
    INSERT INTO tenant (id, name, domain, slug, status)
    VALUES ('uuid', 'Tenant A', 'tenant-a.example.com', 'tenant-a', 'active');
    ```
3. Стратегия `resolveFromHeaders()` сопоставит домен и определит арендатора

## Схема Таблицы Арендаторов

```sql
CREATE TABLE tenant (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  website TEXT,
  domain TEXT UNIQUE,
  slug TEXT UNIQUE,
  status TEXT NOT NULL DEFAULT 'active',  -- 'active' | 'inactive'
  created_at TIMESTAMP NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMP NOT NULL DEFAULT NOW()
);
```
