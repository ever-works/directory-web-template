# Отношения схемы

На этой странице документируются все связи между таблицами, внешние ключи и соединительные таблицы в схеме базы данных шаблона. Схема определена в `lib/db/schema.ts` с использованием Drizzle ORM с PostgreSQL.

## Обзор отношений сущностей

База данных сосредоточена вокруг трех основных объектов: **users** (администратор), **client_profiles** (конечные пользователи) и **items** (хранятся в Git, на которые ссылается слаг). Большинство таблиц вовлеченности и коммерции относятся к этим трем.

## Ролевой контроль доступа

Система RBAC использует три таблицы по схеме «многие ко многим».

### роли

Именованные роли с флагом администратора.

### разрешения

Индивидуальные ключи доступа (например, `items:create`).

### role_permissions (таблица соединений)

Связывает роли с разрешениями.

|Столбец|Цель|При удалении|
|--------|--------|-----------|
|`role_id`|`roles.id`|КАСКАД|
|`permission_id`|`permissions.id`|КАСКАД|

Составной первичный ключ на `(role_id, permission_id)`.

### user_roles (таблица соединений)

Назначает роли пользователям.

|Столбец|Цель|При удалении|
|--------|--------|-----------|
|`user_id`|`users.id`|КАСКАД|
|`role_id`|`roles.id`|КАСКАД|

Составной первичный ключ на `(user_id, role_id)`.

### Диаграмма сущности RBAC

```
users ---< user_roles >--- roles ---< role_permissions >--- permissions
```

Пользователь может иметь множество ролей, каждая роль может иметь множество разрешений, и одну и ту же роль могут использовать несколько пользователей.

## Таблицы вовлеченности

### комментарии

|Отношения|Цель|При удалении|
|-------------|--------|-----------|
|`userId`|`client_profiles.id`|КАСКАД|

В столбце `itemId` хранится фрагмент элемента (а не внешний ключ, поскольку элементы находятся в Git).

### голоса

|Отношения|Цель|При удалении|
|-------------|--------|-----------|
|`userid`|`client_profiles.id`|КАСКАД|

Уникальный индекс `(userid, item_id)` гарантирует один голос каждому пользователю за каждый элемент. В столбце `item_id` хранится ярлык элемента.

### избранное

|Отношения|Цель|При удалении|
|-------------|--------|-----------|
|`userId`|`users.id`|КАСКАД|

Уникальный индекс `(userId, item_slug)` гарантирует каждому пользователю одно избранное для каждого элемента. В столбце `item_slug` хранится ярлык элемента.

### item_views

Никаких внешних ключей. Использует уникальный индекс `(item_id, viewer_id, viewed_date_utc)` для ежедневной дедупликации.

## Сводка каскадного удаления

При удалении **пользователя** каскадно удаляются следующие данные:

- Аккаунты, сессии, аутентификаторы
- Профили клиентов (и транзитивно: комментарии, голоса, отчеты этого клиента, история модерации)
- Подписки
- Платежные счета
- Уведомления
- Избранное
- Назначение ролей пользователей
- Журналы активности
- Спонсорские объявления

При удалении **профиля клиента**:

- Комментарии этого пользователя
- Голоса этого пользователя
- Отчеты, отправленные этим пользователем
- История модерации для этого пользователя
- Журналы активности для этого клиента

Когда **роль** удаляется:

- Все назначения ролей-разрешений для этой роли
- Все назначения ролей пользователей для этой роли

## Ссылки на товары

Элементы хранятся в CMS на базе Git, а не в базе данных. В нескольких таблицах элементы ссылаются на слаги:

- `comments.itemId` -- номер элемента
- `votes.item_id` -- номер элемента
- `favorites.item_slug` -- номер элемента
- `item_views.item_id` -- номер элемента
- `featured_items.item_slug` -- номер элемента
- `sponsor_ads.item_slug` -- номер элемента

Это столбцы с обычным текстом без ограничений внешнего ключа.
