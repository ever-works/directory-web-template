# CI/CD Конвейер

Шаблон Ever Works включает полный CI/CD конвейер, построенный на GitHub Actions. Это руководство описывает структуру рабочих процессов, сканирование безопасности, стратегию защиты веток и поток продвижения развёртываний.

## Обзор Рабочих Процессов

Конвейер состоит из шести файлов рабочих процессов в `.github/workflows/`:

| Рабочий Процесс | Файл | Триггер | Назначение |
|---|---|---|---|
| CI | `ci.yml` | Push/PR в `main`, `develop` | Lint, проверка типов, сборка |
| CodeQL | `codeql.yml` | Push/PR в `main`, `develop` + еженедельное расписание | Анализ уязвимостей безопасности |
| Dev Deploy | `deploy_dev.yaml` | Push в `develop` | Развёртывание в среде предпросмотра |
| Prod Deploy | `deploy_prod.yaml` | Push в `main` | Развёртывание в производственной среде |
| Vercel Deploy | `deploy_vercel.yaml` | Вызывается из dev/prod | Общая логика развёртывания Vercel |
| Disable CodeQL | `disable-default-codeql.yml` | Только вручную | Инструмент разрешения конфликтов CodeQL |

### Поток Конвейера

```
Feature Branch --> PR to develop --> CI runs
                                     |
                                     v
                               Merge to develop --> Dev Deploy (preview)
                                     |
                                     v
                               PR to main --> CI runs
                                     |
                                     v
                               Merge to main --> Prod Deploy (production)
```

## Рабочий Процесс CI (ci.yml)

CI запускается при каждом push и pull request в `main` и `develop`. Он проверяет качество кода и обеспечивает успешную сборку проекта.

### Задания

Рабочий процесс содержит одно задание `lint-and-build`, выполняемое на `ubuntu-latest`:

**Шаги**:

1. **Checkout кода** -- Клонирует репозиторий
2. **Определение менеджера пакетов** -- Автоматически определяет pnpm, yarn или npm по lockfiles
3. **Настройка pnpm** -- Устанавливает pnpm v9 при обнаружении
4. **Настройка Node.js** -- Устанавливает Node 20 с кешем менеджера пакетов
5. **Установка зависимостей** -- Запускает `pnpm install`
6. **Запуск lint** -- Запускает `pnpm lint` (продолжает при ошибках для PR)
7. **Проверка типов** -- Запускает `pnpm typecheck` или `pnpm check:types`
8. **Создание директорий контента** -- Создаёт `.content/data` для сборки
9. **Сборка проекта** -- Запускает `pnpm build` со всеми необходимыми переменными окружения
10. **Проверка успеха сборки** -- Проверяет, что директория `.next` была создана

### Управление Параллелизмом

```yaml
concurrency:
  group: ${{ github.ref }}-${{ github.workflow }}
  cancel-in-progress: true
```

Если в той же ветке произошёл новый push во время выполнения CI, предыдущий запуск автоматически отменяется. Это экономит минуты CI и гарантирует валидацию только последнего коммита.

### Переменные Окружения

CI рабочий процесс использует комбинацию жёстко закодированных значений по умолчанию и секретов GitHub:

| Переменная | Источник | Назначение |
|---|---|---|
| `NEXT_PUBLIC_APP_URL` | Жёстко закодирована | URL приложения для сборки |
| `DATABASE_URL` | Секрет или значение по умолчанию | Подключение к БД для сборки |
| `AUTH_SECRET` | Жёстко закодированное значение CI | Подпись токенов аутентификации (не для производства) |
| `DATA_REPOSITORY` | Секрет или значение по умолчанию | URL репозитория контента |
| `CONTENT_WARNINGS_SILENT` | Жёстко закодировано `true` | Подавляет предупреждения контента в CI |
| `CI` | Жёстко закодировано `true` | Указывает среду CI |
| OAuth секреты | Секреты GitHub | Учётные данные Google, GitHub, Facebook, Twitter |
| `RESEND_API_KEY` | Секрет GitHub | Email сервис для проверок при сборке |

### Разрешения

Рабочий процесс запрашивает минимальные разрешения:

```yaml
permissions:
  contents: read
```

Для задания CI необходим только доступ на чтение содержимого репозитория.

## Анализ Безопасности CodeQL (codeql.yml)

### Что Делает

CodeQL выполняет семантический анализ кода для обнаружения уязвимостей безопасности в JavaScript/TypeScript коде. Запускается:

- При каждом push и PR в `main` и `develop`
- Еженедельно по понедельникам в 6 AM UTC (запланированный анализ)
- При ручном запуске

### Шаги Анализа

1. **Checkout** и **настройка** Node.js + pnpm
2. **Инициализация CodeQL** с языком `javascript-typescript`
3. **Настройка среды CodeQL** через `scripts/codeql-setup.js`
4. **Установка зависимостей** для контекста анализа
5. **Autobuild** -- Автоматическое определение сборки CodeQL
6. **Анализ с загрузкой** -- Загружает результаты во вкладку Security GitHub
7. **Резервный анализ** -- Если загрузка не удаётся, анализ без загрузки

### Разрешения

CodeQL требует более широких разрешений для отчётности о событиях безопасности:

```yaml
permissions:
  actions: read
  contents: read
  security-events: write
  pull-requests: read
```

### Просмотр Результатов

После успешного запуска с загрузкой:
1. Перейдите в свой репозиторий на GitHub
2. Перейдите в **Security** > **Code scanning**
3. Просматривайте результаты, фильтруйте по серьёзности и управляйте предупреждениями

### Разрешение Конфликтов CodeQL

Если вы столкнулись с конфликтами обработки SARIF от стандартной конфигурации CodeQL GitHub, используйте рабочий процесс `disable-default-codeql.yml`:

```bash
# Trigger manually from GitHub Actions tab
# This disables the default configuration that may conflict with your custom setup
```

## Потоки Развёртывания

### Сопоставление Ветка-Среда

| Ветка | Рабочий Процесс | Среда | Домен |
|---|---|---|---|
| `develop` | `deploy_dev.yaml` | `preview` | URL предпросмотра Vercel |
| `main` | `deploy_prod.yaml` | `production` | Производственный домен |

### Шлюз Провайдера Развёртывания

Оба потока развёртывания проверяют переменную репозитория перед продолжением:

```yaml
jobs:
  Vercel:
    if: ${{ vars.DEPLOY_PROVIDER == 'vercel' }}
```

Установите `DEPLOY_PROVIDER=vercel` в **Настройки > Переменные** вашего репозитория, чтобы включить развёртывания Vercel. Это позволяет менять провайдеров развёртывания без изменения файлов рабочих процессов.

### Развёртывание Vercel (deploy_vercel.yaml)

Общий поток развёртывания Vercel обрабатывает как предпросмотровые, так и производственные развёртывания.

**Стратегия Развёртывания**: Рабочий процесс использует двухфазный подход:

1. **Развёртывание через API** (основное): Инициирует развёртывание через API Vercel для более быстрых сборок
2. **Резервный CLI**: Если вызов API не удаётся, откатывается к `vercel build` + `vercel deploy --prebuilt`

**Шаги**:

1. **Checkout** кода
2. **Определение менеджера пакетов** и настройка pnpm
3. **Глобальная установка Vercel CLI**
4. **Привязка проекта Vercel** с использованием `VERCEL_TOKEN` и опциональной области команды
5. **Установка переменных окружения** (DATA_REPOSITORY, GH_TOKEN, CRON_SECRET) через Vercel CLI
6. **Получение настроек Vercel** для целевой среды
7. **Инициация развёртывания через API** или откат к CLI сборке/развёртыванию
8. **Обновление расписания cron** через `scripts/update-cron.ts`

### Необходимые Секреты

Настройте их в секретах вашего GitHub репозитория:

| Секрет | Обязателен | Назначение |
|---|---|---|
| `VERCEL_TOKEN` | Да | Аутентификация API Vercel |
| `VERCEL_TEAM_SCOPE` | При использовании команд | Слаг команды Vercel |
| `DATA_REPOSITORY` | Да | Имя репозитория контента |
| `GH_TOKEN` | Да | Токен GitHub для клонирования контента |
| `CRON_SECRET` | Рекомендуется | Аутентифицирует вызовы cron endpoint |
| `DATABASE_URL` | Для сборки | Строка подключения к базе данных |
| OAuth секреты | При использовании OAuth | Учётные данные провайдера |

### Обновления Расписания Cron

После успешного развёртывания рабочий процесс запускает `scripts/update-cron.ts` для синхронизации расписаний cron:

```yaml
- name: Update cron schedule
  if: success() && steps.trigger_deployment.outputs.deployment_id != ''
  run: npx tsx scripts/update-cron.ts
```

## Правила Защиты Веток

### Рекомендуемые Настройки для `main`

| Настройка | Значение | Назначение |
|---|---|---|
| Требовать pull request | Да | Без прямых push в production |
| Необходимые проверки | 1+ | Проверка кода перед слиянием |
| Требовать проверки статуса | CI (lint-and-build) | CI должен пройти перед слиянием |
| Требовать CodeQL | Анализ CodeQL | Проверка безопасности должна пройти |
| Требовать актуальные ветки | Да | PR должен быть ребейзнут на последний main |
| Включить администраторов | Да | Правила применяются ко всем |

### Рекомендуемые Настройки для `develop`

| Настройка | Значение | Назначение |
|---|---|---|
| Требовать pull request | Опционально | Прямые push разрешены для быстрой итерации |
| Необходимые проверки статуса | CI (lint-and-build) | Базовый шлюз качества |
| Требовать актуальные ветки | Нет | Позволяет более быструю итерацию |

### Настройка Защиты Веток

1. Перейдите в **Настройки** > **Ветки** вашего репозитория
2. Нажмите **Добавить правило защиты ветки**
3. Введите шаблон имени ветки (например, `main`)
4. Настройте параметры из таблиц выше
5. Сохраните изменения

## Поток Продвижения

Шаблон следует стандартному потоку продвижения:

### Цикл Разработки

```
1. Create feature branch from develop
2. Implement changes
3. Open PR to develop
4. CI validates (lint, type check, build)
5. CodeQL scans for vulnerabilities
6. Code review and approval
7. Merge to develop --> automatic preview deployment
```

### Производственный Релиз

```
1. Open PR from develop to main
2. CI validates against main
3. CodeQL security scan
4. Final code review
5. Merge to main --> automatic production deployment
```

### Процесс Хотфикса

```
1. Create hotfix branch from main
2. Implement fix
3. Open PR directly to main
4. CI + CodeQL validation
5. Emergency review and merge
6. Backport to develop
```

## Настройка

### Добавление Новых Шагов CI

Чтобы добавить тесты или дополнительную валидацию, добавьте шаги в задание `ci.yml`:

```yaml
- name: Run unit tests
  run: ${{ steps.detect-pm.outputs.run-cmd }} test

- name: Run E2E tests
  run: ${{ steps.detect-pm.outputs.run-cmd }} test:e2e
```

### Добавление Уведомлений о Развёртывании

Добавьте шаг уведомления в конец потока развёртывания:

```yaml
- name: Notify Slack
  if: success()
  uses: slackapi/slack-github-action@v1
  with:
    payload: '{"text": "Deployed to ${{ inputs.environment }}"}'
  env:
    SLACK_WEBHOOK_URL: ${{ secrets.SLACK_WEBHOOK_URL }}
```

### Переменные Для Конкретных Сред

Используйте **Среды** GitHub для ограничения секретов конкретными целями развёртывания:

1. Перейдите в **Настройки** > **Среды**
2. Создайте среды `production` и `preview`
3. Добавьте секреты, специфичные для среды
4. Ссылайтесь на них в рабочих процессах с конфигурацией `environment:`
