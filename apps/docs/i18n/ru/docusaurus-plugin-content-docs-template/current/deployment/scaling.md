# Масштабирование и Высокая Доступность

Это руководство охватывает стратегии масштабирования Ever Works Template от развёртывания на одном экземпляре до продакшен-конфигурации с высокой доступностью, включая конфигурацию serverless, connection pooling, оптимизацию CDN и edge-функции.

## Архитектуры Развёртывания

Шаблон поддерживает несколько архитектур развёртывания:

| Архитектура | Лучше всего для | Модель Масштабирования |
|---|---|---|
| Vercel (Serverless) | Большинство развёртываний | Автоматическое горизонтальное масштабирование |
| Docker (Standalone) | Self-hosted, on-premise | Ручное или на основе оркестратора |
| Node.js (Прямой) | Разработка, простые развёртывания | Единственный экземпляр или кластер PM2 |

## Конфигурация Serverless (Vercel)

### Вывод Standalone

Шаблон настроен с выводом standalone для оптимизированного serverless-развёртывания:

```typescript
// next.config.ts
const nextConfig: NextConfig = {
  output: "standalone",
};
```

Режим standalone создаёт самодостаточную сборку в `.next/standalone/`, содержащую только файлы, необходимые для запуска приложения. Это минимизирует время холодного запуска за счёт уменьшения размера пакета развёртывания.

### Конфигурация Функций

Настройте параметры serverless-функций в `vercel.json` или через конфигурацию на уровне маршрута:

```typescript
// app/api/heavy-computation/route.ts
export const maxDuration = 60; // секунды (Pro план: до 300с)
export const dynamic = 'force-dynamic';
```

### Рекомендуемые Настройки Функций

| Тип Маршрута | Макс. Длительность | Память | Примечания |
|---|---|---|---|
| API-маршруты (простые) | 10с | 1024 МБ | По умолчанию для большинства конечных точек |
| API-маршруты (обработка данных) | 30с | 1024 МБ | Для пакетных операций |
| Cron-задания | 60с | 1024 МБ | Выполнение фоновых задач |
| Обработчики webhook | 30с | 1024 МБ | Колбэки платежей, OAuth |
| Статические страницы | Н/Д | Н/Д | Предрендерятся при сборке |

### Оптимизация Холодного Запуска

Минимизируйте холодные запуски с помощью этих техник:

| Техника | Реализация | Влияние |
|---|---|---|
| Минимизировать размер функции | `serverExternalPackages` в конфигурации | Сокращает время инициализации |
| Избегать импортов на уровне модуля | Динамический `import()` для тяжёлых модулей | Откладывает загрузку до необходимости |
| Использовать edge runtime где возможно | `export const runtime = 'edge'` | Почти нулевой холодный запуск |
| Поддерживать функции тёплыми | Конечные точки health check с мониторингом | Поддерживает функции активными |

## Connection Pooling Базы Данных

### Проблема

В serverless-окружениях каждый вызов функции может открывать новое соединение с базой данных. Без пулинга это может исчерпать лимит соединений базы данных.

### Решение: Connection Pooler

Используйте connection pooler между приложением и базой данных:

| Пулер | Провайдер | Конфигурация |
|---|---|---|
| PgBouncer | Supabase (встроенный) | Используйте pooled строку подключения (порт 6543) |
| Neon Pooler | Neon (встроенный) | Используйте строку подключения `-pooler` |
| PgBouncer | Self-hosted | Разверните PgBouncer вместе с PostgreSQL |

### Конфигурация

Используйте разные строки подключения для pooled и прямых соединений:

```bash
# Pooled соединение для запросов приложения (безопасно для serverless)
DATABASE_URL=postgresql://user:pass@host:6543/db?pgbouncer=true

# Прямое соединение только для миграций
DIRECT_DATABASE_URL=postgresql://user:pass@host:5432/db
```

Обновите `drizzle.config.ts` для использования прямого соединения при миграциях:

```typescript
export default {
  schema: "./lib/db/schema.ts",
  out: "./lib/db/migrations",
  dialect: "postgresql",
  dbCredentials: {
    url: process.env.DIRECT_DATABASE_URL || process.env.DATABASE_URL,
  },
} satisfies Config;
```

### Лимиты Соединений

| Уровень | Макс. Соединений | Рекомендуемый Размер Пула |
|---|---|---|
| Hobby (Neon/Supabase) | 50–100 | 10–20 |
| Pro (Neon/Supabase) | 200–500 | 50–100 |
| Enterprise | 1000+ | 100–200 |

### Управление Соединениями в Коде

Модуль базы данных шаблона должен повторно использовать единый пул соединений на экземпляр функции:

```typescript
// lib/db/index.ts
import { drizzle } from 'drizzle-orm/postgres-js';
import postgres from 'postgres';

// Создать пул соединений один раз на serverless-экземпляр
const connectionString = process.env.DATABASE_URL!;
const client = postgres(connectionString, {
  max: 10,          // Максимальное количество соединений в пуле
  idle_timeout: 20, // Закрыть неактивные соединения через 20с
  connect_timeout: 10,
});

export const db = drizzle(client);
```

## CDN и Кеширование

### Сеть Vercel Edge

При развёртывании на Vercel сеть Edge автоматически предоставляет:

- Глобальное распределение CDN в более чем 30 регионах
- Автоматическое кеширование статических ресурсов
- Edge-кеширование для ISR-страниц (Incremental Static Regeneration)
- Защиту от DDoS

### Заголовки Cache-Control

Настройте кеширование для разных типов контента:

```typescript
// API-маршрут с заголовками кеширования
export async function GET() {
  const data = await fetchData();

  return Response.json(data, {
    headers: {
      'Cache-Control': 'public, s-maxage=60, stale-while-revalidate=300',
    },
  });
}
```

### Стратегия Кеширования по Типу Контента

| Тип Контента | Стратегия Кеширования | TTL | Примечания |
|---|---|---|---|
| Статические ресурсы (JS, CSS, изображения) | Immutable | 1 год | Имена файлов с хешем контента |
| Публичные страницы | ISR | 60–300с | Ревалидация по запросу |
| Ответы API (публичные) | `s-maxage` | 10–60с | Кеширование на уровне CDN |
| Ответы API (аутентифицированные) | `no-store` | 0 | Никогда не хранить данные пользователя |
| Страницы контента CMS | ISR | 300с | Ревалидация после синхронизации контента |

### ISR (Incremental Static Regeneration)

Используйте ISR для страниц с большим объёмом контента, которые редко меняются:

```typescript
// app/[locale]/discover/[page]/page.tsx
export const revalidate = 300; // Регенерировать каждые 5 минут

export default async function DiscoverPage({ params }) {
  const items = await fetchItems(params.page);
  return <ItemGrid items={items} />;
}
```

### Ревалидация по Запросу

Запускайте ревалидацию после обновлений контента:

```typescript
// app/api/revalidate/route.ts
import { revalidatePath } from 'next/cache';

export async function POST(request: Request) {
  const { secret, path } = await request.json();

  if (secret !== process.env.REVALIDATION_SECRET) {
    return Response.json({ error: 'Invalid secret' }, { status: 401 });
  }

  revalidatePath(path);
  return Response.json({ revalidated: true });
}
```

## Edge-функции

### Когда Использовать Edge Runtime

Edge-функции работают на Cloudflare Workers (через Vercel) и обеспечивают почти нулевое время холодного запуска. Используйте их для:

| Случай Использования | Пример |
|---|---|
| Маршрутизация на основе геолокации | Перенаправление пользователей на региональный контент |
| A/B-тестирование | Направление к вариантам эксперимента |
| Проверки аутентификации | Быстрая валидация сессии |
| Трансформация ответов | Добавление заголовков, изменение ответов |
| Простые API-конечные точки | Лёгкое получение данных |

### Ограничения Edge Runtime

| Ограничение | Детали |
|---|---|
| Нет API Node.js | Нельзя использовать `fs`, `child_process` и т.д. |
| Нет нативных модулей | Нельзя напрямую использовать `bcryptjs`, `postgres` |
| Ограниченное время выполнения | Макс. 30 секунд (Vercel Pro) |
| Ограниченная память | 128 МБ |
| Нет Drizzle ORM | Используйте клиенты базы данных, совместимые с edge |

### Пример Edge-функции

```typescript
// app/api/geo/route.ts
export const runtime = 'edge';

export async function GET(request: Request) {
  const country = request.headers.get('x-vercel-ip-country') || 'US';
  const city = request.headers.get('x-vercel-ip-city') || 'Unknown';

  return Response.json({
    country,
    city,
    timestamp: Date.now(),
  });
}
```

## Стратегии Горизонтального Масштабирования

### Stateless-дизайн Приложения

Шаблон спроектирован как stateless на уровне приложения:

| Компонент | Расположение Состояния | Влияние на Масштабирование |
|---|---|---|
| Сессии | База данных или JWT | Нет общего состояния между экземплярами |
| Фоновые задачи | Менеджер задач (на экземпляр или Trigger.dev) | Используйте Trigger.dev для нескольких экземпляров |
| Загрузки файлов | Внешнее хранилище (S3, Supabase) | Нет зависимости от локальной файловой системы |
| Контент CMS | Git-репозиторий (клонируется при сборке/запуске) | Только чтение, идентично для каждого экземпляра |
| Кеш | In-memory (на экземпляр) или Redis | Рассмотрите Redis для общего кеша |

### Соображения для Нескольких Экземпляров

При запуске нескольких экземпляров (Docker Swarm, Kubernetes или несколько Vercel-функций):

1. **Фоновые задачи**: Используйте Trigger.dev или Vercel Cron вместо `LocalJobManager` для избежания дублирования.
2. **Соединения с базой данных**: Включите connection pooling для предотвращения исчерпания соединений.
3. **Хранение сессий**: Используйте сессии на основе базы данных вместо in-memory хранилищ.
4. **Инвалидация кеша**: Реализуйте общий кеш (Redis) или примите конечную согласованность с кешами на экземпляр.

## Мониторинг при Масштабировании

### Ключевые Метрики для Отслеживания

| Метрика | Инструмент | Порог |
|---|---|---|
| Время ответа (p95) | Sentry, Vercel Analytics | < 500мс |
| Частота ошибок | Sentry | < 1% |
| Количество соединений базы данных | Дашборд базы данных | < 80% максимума |
| Холодные запуски функций | Vercel Analytics | Мониторинг частоты |
| Коэффициент попаданий в кеш | Логи приложения | > 80% |
| Использование памяти | Метрики Vercel/Docker | < 80% лимита |

### Мониторинг Производительности Sentry

Шаблон настраивает Sentry с трассировочной выборкой:

```typescript
Sentry.init({
  tracesSampleRate: process.env.NODE_ENV === 'production' ? 0.1 : 1.0,
});
```

Настройте `tracesSampleRate` исходя из объёма трафика:

| Запросов в день | Рекомендуемая Частота Выборки |
|---|---|
| < 10 000 | 1,0 (100%) |
| 10 000–100 000 | 0,1 (10%) |
| 100 000–1 000 000 | 0,01 (1%) |
| > 1 000 000 | 0,001 (0,1%) |

## Нагрузочное Тестирование

### Рекомендуемые Инструменты

| Инструмент | Случай Использования | Сложность |
|---|---|---|
| `autocannon` | Быстрые HTTP-бенчмарки | Низкая |
| `k6` | Скриптованное нагрузочное тестирование | Средняя |
| `Artillery` | Сложные сценарии | Средняя |

### Пример Нагрузочного Теста

```bash
# Быстрый бенчмарк с autocannon
npx autocannon -c 50 -d 30 https://your-app.vercel.app/api/health

# k6-скрипт для более детального тестирования
k6 run load-test.js
```

### Контрольный Список Тестов

| Тест | Цель | Критерий Прохождения |
|---|---|---|
| Загрузка главной страницы | 100 одновременных пользователей | p95 < 1с |

| Поисковый запрос | 50 одновременных пользователей | p95 < 2с |
| Процесс аутентификации | 20 одновременных пользователей | Все успешно, без таймаутов |
