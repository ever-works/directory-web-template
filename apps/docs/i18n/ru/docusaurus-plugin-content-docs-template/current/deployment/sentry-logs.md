# Конфигурация Логов Sentry

Этот документ объясняет, как настроить и использовать Sentry Logs в репозитории Template и репозитории Ever Works.

## Обзор

Sentry Logs обеспечивает централизованное управление логами, позволяя захватывать, перенаправлять и анализировать логи приложения в Logs Explorer Sentry. Все логи автоматически перенаправляются в Sentry при включении, обеспечивая единый взгляд на поведение приложения в разных окружениях.

## Функциональность

- ✅ Автоматическое перенаправление логов в Sentry
- ✅ Поддержка всех уровней логирования (debug, info, warn, error)
- ✅ Контекстное логирование с автоматической пометкой
- ✅ Конфигурация под конкретное окружение
- ✅ Структурированное логирование с поддержкой метаданных
- ✅ Интеграция с существующей утилитой логгера

## Конфигурация

### Переменные Окружения

Добавьте эти переменные в файл `.env.local` для локальной разработки:

```env
# Конфигурация Sentry (обязательна для логов)
NEXT_PUBLIC_SENTRY_DSN=https://your-dsn@sentry.io/your-project-id
SENTRY_ORG=your-org-name
SENTRY_PROJECT=your-project-name
SENTRY_AUTH_TOKEN=your-auth-token

# Включить Sentry в разработке (опционально, по умолчанию только в продакшене)
SENTRY_ENABLE_DEV=true

# Режим отладки Sentry (опционально)
SENTRY_DEBUG=false

# Конфигурация логов Sentry
SENTRY_LOGS_ENABLED=true  # Включить/выключить Sentry Logs (по умолчанию: true)
SENTRY_LOGS_LEVEL=info    # Минимальный уровень логирования для захвата (по умолчанию: info)
```

### Конфигурация под Конкретное Окружение

#### Локальная Разработка

```env
SENTRY_ENABLE_DEV=true
SENTRY_LOGS_ENABLED=true
SENTRY_LOGS_LEVEL=debug  # Захватывать все логи в разработке
```

#### Разработка/Staging

```env
SENTRY_ENABLE_DEV=true
SENTRY_LOGS_ENABLED=true
SENTRY_LOGS_LEVEL=info  # Захватывать логи info, warn и error
```

#### Продакшен

```env
SENTRY_ENABLE_DEV=false  # Не нужно в продакшене
SENTRY_LOGS_ENABLED=true
SENTRY_LOGS_LEVEL=warn  # Захватывать только предупреждения и ошибки в продакшене
```

## Использование

### Базовое Логирование

Логгер автоматически перенаправляет логи в Sentry при включении:

```typescript
import { logger } from '@/lib/logger';

// Лог info
logger.info('User logged in', { userId: '12345' });

// Лог предупреждения
logger.warn('Rate limit approaching', { current: 90, limit: 100 });

// Лог ошибки
logger.error('Payment failed', { orderId: '67890', error: errorObject });

// Лог отладки (только в разработке)
logger.debug('API request', { method: 'GET', url: '/api/users' });
```

### Контекстное Логирование

Создайте логгер с конкретным контекстом для лучшей организации:

```typescript
import { Logger } from '@/lib/logger';

const paymentLogger = Logger.create('PaymentService');

paymentLogger.info('Processing payment', { amount: 100, currency: 'USD' });
paymentLogger.error('Payment failed', error);
```

### Уровни Логирования

Логгер поддерживает четыре уровня логирования, автоматически сопоставляемые с уровнями серьёзности Sentry:

| Уровень Логгера | Уровень Sentry | Описание |
|-------------|-------------|-------------|
| `DEBUG` | `debug` | Подробная отладочная информация (только в разработке) |
| `INFO` | `info` | Общие информационные сообщения |
| `WARN` | `warning` | Предупреждения о потенциальных проблемах |
| `ERROR` | `error` | Сообщения об ошибках и сбоях |

## Как Это Работает

### Инициализация

Sentry Logs включается как в инструментации клиента, так и в инструментации сервера:

1. **Серверная сторона** (`instrumentation.ts`): Инициализирует Sentry для Node.js runtime
2. **Клиентская сторона** (`instrumentation-client.ts`): Инициализирует Sentry для браузерного runtime

Обе конфигурации включают:
```typescript
_experiments: {
  enableLogs: SENTRY_LOGS_ENABLED,
}
```

### Перенаправление Логов

Утилита логгера (`lib/logger.ts`) автоматически:
1. Проверяет, включён ли Sentry Logs
2. Форматирует записи логов с контекстом и метаданными
3. Перенаправляет логи в Sentry с помощью `Sentry.captureMessage()` с соответствующими тегами и уровнями
4. Корректно переключается в режим fallback, если Sentry недоступен

### Структура Логов

Каждая запись лога, отправляемая в Sentry, содержит:
- **Сообщение**: Сообщение лога с опциональным префиксом контекста
- **Уровень**: Уровень серьёзности (debug, info, warning, error)
- **Теги**:
  - `logLevel`: Исходный уровень лога
  - `logType`: Всегда `application_log`
  - `context`: Опциональный идентификатор контекста
- **Дополнительные Данные**:
  - `data`: Любые предоставленные дополнительные данные
  - `timestamp`: ISO-метка времени

## Просмотр Логов в Sentry

### Logs Explorer

1. Перейдите в свой проект Sentry
2. Перейдите в **Logs** → **Logs Explorer**
3. Используйте фильтры для поиска конкретных логов:
   - Фильтровать по тегу `logLevel` (debug, info, warn, error)
   - Фильтровать по тегу `context` для просмотра логов конкретных модулей
   - Фильтровать по `logType:application_log` для просмотра только логов приложения

### Запросы к Логам

Примеры запросов в Logs Explorer Sentry:

```
# Все логи ошибок
logLevel:error

# Логи конкретного контекста
context:PaymentService

# Все логи приложения
logType:application_log

# Ошибки за конкретный временной диапазон
logLevel:error timestamp:>2024-01-01
```

## Интеграция с Пакетом Мониторинга

При использовании пакета `@ever-works/monitoring` убедитесь, что он настроен для работы с Sentry Logs:

1. Пакет мониторинга должен инициализировать Sentry с включёнными логами
2. Утилита логгера в этом шаблоне автоматически перенаправит логи в Sentry
3. Обе системы работают совместно для обеспечения комплексного мониторинга

## Устранение Неполадок

### Логи Не Отображаются в Sentry

1. **Проверить конфигурацию DSN**
   ```bash
   echo $NEXT_PUBLIC_SENTRY_DSN
   ```
   Убедитесь, что DSN корректно задан и доступен.

2. **Проверить, включены ли логи**
   ```bash
   echo $SENTRY_LOGS_ENABLED
   ```
   Должно быть `true` для перенаправления логов.

3. **Проверить инициализацию Sentry**
   - Убедитесь, что `SENTRY_ENABLED` равен true
   - Проверьте консоль браузера на наличие ошибок инициализации Sentry
   - Убедитесь, что `_experiments.enableLogs` установлен в `true`

4. **Проверить фильтрацию уровней логирования**
   - Убедитесь, что ваш уровень логирования соответствует порогу `SENTRY_LOGS_LEVEL`
   - Логи отладки захватываются только если уровень установлен на `debug`

### Соображения о Производительности

- Логи отправляются асинхронно и не блокируют приложение
- В продакшене рассмотрите установку `SENTRY_LOGS_LEVEL=warn` для снижения объёма логов
- Sentry автоматически обрабатывает ограничение частоты запросов и пакетирование

### Отключение Логов

Для отключения Sentry Logs без отключения Sentry полностью:

```env
SENTRY_LOGS_ENABLED=false
```

Логгер будет продолжать работать нормально, но логи не будут перенаправляться в Sentry.

## Лучшие Практики

1. **Использовать Соответствующие Уровни Логирования**
   - Используйте `debug` для детальной отладочной информации
   - Используйте `info` для общего потока приложения
   - Используйте `warn` для потенциальных проблем, не нарушающих функциональность
   - Используйте `error` для реальных ошибок и исключений

2. **Включать Контекст**
   - Используйте контекстные логгеры для лучшей организации
   - Включайте соответствующие метаданные в данные логов

3. **Избегать Конфиденциальных Данных**
   - Никогда не логируйте пароли, токены или персональные данные
   - Очищайте данные перед логированием

4. **Конфигурация Продакшена**
   - Установите `SENTRY_LOGS_LEVEL=warn` в продакшене
   - Мониторинг использования квоты Sentry
   - Регулярно проверяйте логи на наличие паттернов
