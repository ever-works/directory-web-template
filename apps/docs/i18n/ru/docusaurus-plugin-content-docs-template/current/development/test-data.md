# Тестовые Данные и Фикстуры

Шаблон Ever Works предоставляет несколько механизмов для генерации и управления тестовыми данными в контекстах разработки, заполнения (seeding) и E2E-тестирования. Эта страница охватывает фиктивные данные, сиды базы данных, фикстуры E2E и стратегии обеспечения согласованности данных.

## E2E Тестовые Данные (`e2e/helpers/test-data.ts`)

Набор E2E-тестов определяет свои тестовые данные через централизованный вспомогательный модуль:

```typescript
export const TEST_DATA = {
  get ADMIN_EMAIL()    { return requireEnv('SEED_ADMIN_EMAIL'); },
  get ADMIN_PASSWORD() { return requireEnv('SEED_ADMIN_PASSWORD'); },
  CLIENT_PASSWORD: 'TestClient123!',
  generateClientEmail: () =>
    `e2e-client-${Date.now()}-${Math.random().toString(36).slice(2, 8)}@test.local`,
  generateItemName: () =>
    `E2E Test Item ${Date.now()}-${Math.random().toString(36).slice(2, 6)}`,
  generateItemUrl: () =>
    `https://e2e-test-${Date.now()}.example.com`,
};
```

### Ключевые Проектные Решения

- **Учётные данные администратора из env** -- Email и пароль администратора читаются из переменных окружения `SEED_ADMIN_EMAIL` и `SEED_ADMIN_PASSWORD`, обеспечивая использование тестами тех же учётных данных, что и у засеянного пользователя-администратора.
- **Уникальные данные клиента** -- Email клиентов и названия элементов включают временные метки и случайные суффиксы для предотвращения коллизий при параллельных прогонах тестов.
- **Ленивая оценка** -- Учётные данные администратора используют геттер-функции, которые немедленно выбрасывают исключение при отсутствии переменных окружения, выявляя ошибки конфигурации заблаговременно.

## Фикстуры Состояния Аутентификации E2E

Состояние аутентификации управляется через файлы состояния хранилища Playwright:

```
e2e/auth-states/
  admin.json    # Сериализованная сессия администратора (cookies, localStorage)
  client.json   # Сериализованная сессия клиента
```

Эти файлы генерируются в ходе `global-setup.ts` путём программного входа с учётными данными администратора и клиента. Фикстура аутентификации (`e2e/fixtures/auth.fixture.ts`) предоставляет предварительно аутентифицированные контексты браузера:

- `adminContext` / `adminPage` -- Контекст браузера с загруженной сессией администратора
- `clientContext` / `clientPage` -- Контекст браузера с загруженной сессией клиента

Файлы тестов импортируют пользовательский объект `test` вместо стандартного Playwright:

```typescript
import { test, expect } from '@/e2e/fixtures';

test('admin can view dashboard', async ({ adminPage }) => {
  await adminPage.goto('/admin');
  await expect(adminPage.getByRole('heading')).toContainText('Dashboard');
});
```

## Заполнение Базы Данных

### Скрипт Заполнения (`lib/db/seed.ts`)

Скрипт заполнения базы данных выполняется через `pnpm db:seed` и заполняет базу данных начальными данными, необходимыми для работы приложения:

- **Пользователь-администратор** -- Создаётся из переменных окружения `SEED_ADMIN_EMAIL` и `SEED_ADMIN_PASSWORD`
- **Фиктивные пользователи** -- Генерируются на основе `SEED_FAKE_USER_COUNT` (по умолчанию: 10)
- **Демонстрационные данные** -- При `NEXT_PUBLIC_DEMO=true` засеиваются полные демонстрационные данные для всех функций

Скрипт заполнения идемпотентен — он проверяет существующие данные перед вставкой, чтобы избежать дублирования при повторных запусках.

### Демо-режим

При `NEXT_PUBLIC_DEMO=true` скрипт заполнения генерирует:

- Множество пользователей с разными ролями и профилями
- Примеры элементов в разных категориях и статусах
- Комментарии, голоса и данные вовлечённости
- Заявки на спонсорскую рекламу в различных состояниях
- Определения опросов с примерами ответов

## Стратегии Согласованности Данных

### Изоляция Между Прогонами Тестов

E2E-тесты используют несколько стратегий для предотвращения интерференции данных:

1. **Уникальные идентификаторы** -- Все генерируемые тестовые данные включают временные метки для предотвращения коллизий имён
2. **Очистка после теста** -- Тесты, создающие данные, должны убирать за собой
3. **Отдельные контексты аутентификации** -- Тесты администратора и клиента выполняются в изолированных контекстах браузера
4. **Глобальная настройка/очистка** -- `global-setup.ts` подготавливает состояние аутентификации, `global-teardown.ts` выполняет очистку

### Разработка vs Тестирование vs Производство

| Аспект | Разработка | Тестирование (E2E) | Производство |
|--------|-----------|-------------------|--------------|
| База данных | SQLite (`file:./dev.db`) или Postgres | То же, что dev (переиспользуемый сервер) | Postgres |
| Контент | Клонирован из `DATA_REPOSITORY` | Существующий контент из dev | Git-CMS |
| Пользователи | Засеянный admin + фиктивные пользователи | То же, что dev + пользователи, созданные E2E | Реальные пользователи |
| Демо-данные | При `NEXT_PUBLIC_DEMO=true` | Зависит от засеянных демо-данных | `NEXT_PUBLIC_DEMO=false` |

### Лучшие Практики

1. **Всегда заполняйте перед тестированием** -- Запустите `pnpm db:seed` перед E2E-тестами для гарантии существования пользователя-администратора
2. **Используйте генераторы уникальных данных** -- Никогда не задавайте жёстко имена элементов или email в тестах
3. **Проверяйте переменные окружения** -- Хелпер `requireEnv()` выдаёт понятные сообщения об ошибках при отсутствии обязательных переменных
4. **Держите фикстуры минимальными** -- Файлы состояния аутентификации содержат только необходимые cookies и записи хранилища
5. **Избегайте межтестовых зависимостей** -- Каждый spec-файл должен быть независимо запускаемым

## Переменные Окружения для Тестирования

```bash
# Обязательны для E2E-тестов
SEED_ADMIN_EMAIL=admin@changeme.com
SEED_ADMIN_PASSWORD=changeme_password

# Необязательные
BASE_URL=http://localhost:3000
SEED_FAKE_USER_COUNT=10
NEXT_PUBLIC_DEMO=true
```

## Связанные Файлы

- `e2e/helpers/test-data.ts` -- Генераторы тестовых данных и константы
- `e2e/fixtures/auth.fixture.ts` -- Фикстуры аутентификации для Playwright
- `e2e/global-setup.ts` -- Подготовка аутентификации перед тестами
- `e2e/global-teardown.ts` -- Очистка после тестов
- `lib/db/seed.ts` -- Скрипт заполнения базы данных
- `.env.example` -- Полный справочник переменных окружения
