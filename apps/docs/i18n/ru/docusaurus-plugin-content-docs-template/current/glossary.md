# Словарь Терминов

Ключевые термины и понятия, используемые во всей документации Directory Web Template.

## Основные Концепции Предметной Области

### Каталог

Коллекция организованных записей (элементов) вокруг конкретной темы или ниши. Каталог является сущностью верхнего уровня. Примеры: "SaaS Tools Directory", "Developer Resources Directory" или "Local Business Directory".

### Элемент

Отдельная запись или листинг в каталоге. Элемент представляет одну каталогизируемую сущность (инструмент, компанию, ресурс или сервис). Элементы имеют структурированные поля (название, описание, URL, логотип), принадлежат категориям и могут иметь теги.

### Категория

Иерархическая классификация для организации элементов. Категории образуют древовидную структуру (отношения «родитель/дочерний») и обеспечивают основной механизм навигации и фильтрации.

### Тег

Плоская, неиерархическая метка, прикреплённая к элементам для перекрёстной классификации. Теги используются для вторичной фильтрации и поиска. Элемент может иметь несколько тегов, например "open-source", "freemium" или "API-available".

### Коллекция

Кураторская группировка элементов, независимая от категорий или тегов. Коллекции — пользовательские или редакторски кураторские наборы, например "Топ 10 выборов" или "Новое в этом месяце".

### Таксономия

Общая система классификации каталога, включающая категории, теги и любые другие организационные структуры.

### Slug

Удобный для URL, человекочитаемый идентификатор, производный от имени сущности. Slugs используются в URL вместо числовых ID. Например, "Visual Studio Code" становится `visual-studio-code`.

## Паттерны Архитектуры

### Repository

Класс слоя доступа к данным, инкапсулирующий запросы и мутации базы данных для конкретной сущности. Repositories абстрагируют Drizzle ORM и предоставляют чистый интерфейс для сервисов. Находится в `lib/repositories/`.

### Service

Класс слоя бизнес-логики, оркестрирующий операции между репозиториями, внешними API и другими сервисами. Сервисы содержат основную логику приложения и вызываются обработчиками API-маршрутов. Находится в `lib/services/`.

### Webhook

HTTP-обратный вызов, инициируемый событием. Template использует вебхуки для уведомлений от платёжных провайдеров (Stripe, LemonSqueezy, Polar) и обновлений статуса деплоя. Эндпоинты вебхуков валидируют входящие запросы с помощью подписей или общих секретов.

## Управление Контентом

### CMS на основе Git

Подход к управлению контентом, используемый Template. Данные каталога (элементы, категории, метаданные) хранятся как структурированные файлы (YAML, Markdown) в Git-репозитории. Template клонирует этот репозиторий во время сборки и читает контент из локальной файловой системы. Изменения вносятся через коммиты и pull request'ы.

### Community PR

Pull request, отправленный членом сообщества для добавления или обновления элементов в Git-репозитории CMS каталога. Community PR'ы проходят процесс проверки перед слиянием.

## База Данных

### Drizzle ORM

Лёгкий, TypeScript-first ORM, используемый Template. Drizzle предоставляет SQL-подобный конструктор запросов с полной типовой безопасностью. Определения схемы пишутся в виде кода TypeScript, а миграции генерируются как обычные SQL-файлы через Drizzle Kit.

### Миграция

Версионированное изменение схемы базы данных. Миграции генерируются с помощью `pnpm db:generate` и применяются с помощью `pnpm db:migrate`. Файлы миграций хранятся в `lib/db/migrations/`.

## Аутентификация

### NextAuth.js

Библиотека аутентификации (v5), используемая Template. Она обеспечивает поддержку OAuth для нескольких провайдеров (Google, GitHub, Facebook, Twitter, Microsoft) с управлением сессиями и JWT-токенами.

### Supabase Auth

Альтернативный бэкенд аутентификации, поддерживаемый Template. Supabase Auth обеспечивает аутентификацию по email/паролю, magic links и социальный OAuth через управляемый сервис Supabase.

## Платежи

### Подписка

Периодическое платёжное соглашение, управляемое через одного из поддерживаемых платёжных провайдеров (Stripe, LemonSqueezy или Polar). Template обрабатывает создание подписок, управление ими и обработку вебхуков.

## Деплой

### Vercel

Основная платформа деплоя для Template. Vercel обеспечивает деплой без настройки для приложений Next.js, включая автоматические деплои предварительного просмотра, edge-функции и дистрибуцию через CDN.

### Docker

Альтернативный метод деплоя. Template может быть контейнеризован и развёрнут в любой Docker-совместимой среде хостинга.
