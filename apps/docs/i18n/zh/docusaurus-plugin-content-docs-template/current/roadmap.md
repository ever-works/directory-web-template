# 路线图与未来方向

本页面概述了 Directory Web Template 的当前开发方向，以及社区如何参与塑造其未来。

## 产品愿景

Directory Web Template 旨在成为构建专业目录网站最全面的开源解决方案。长期愿景涵盖：

- **生产级目录网站**，美观、高性能且完全可定制
- **简单的内容管理**，通过基于 Git 的 CMS，可选通过 [Ever Works 平台](https://docs.ever.works) 进行 AI 驱动的内容生成
- **可扩展的支付和身份验证**，开箱即用地支持多个提供商
- **一流的国际化**，提供完整的 RTL 支持和不断扩大的语言覆盖范围

## 积极开发领域

### 性能与核心 Web 指标

- 优化商品列表和详情页面的最大内容绘制（LCP）
- 通过更好的代码分割和 tree shaking 减少 JavaScript 包体积
- 改进目录条目截图和徽标的图像优化管道
- 实现部分预渲染以加快初始页面加载速度

### 功能增强

- 添加更多过滤和搜索功能（分面搜索、高级过滤器）
- 实现用户生成内容功能（评论、评分、留言）
- 添加更多支付提供商集成和订阅管理功能
- 通过更多内置主题和更简单的自定义扩展主题系统

### 开发体验

- 通过更好的文档和错误提示改进本地开发环境配置
- 使用 Playwright 添加更全面的 E2E 测试覆盖
- 为常见目录类型（SaaS、本地商业、资源）创建入门模板
- 改进整个代码库的 TypeScript 类型安全

### 国际化

- 添加更多内置语言翻译
- 改进阿拉伯语和希伯来语的 RTL 布局支持
- 支持每个目录的语言配置
- 添加自动化翻译工作流

### 文档

- 通过更多示例扩展 API 参考文档
- 为常见任务添加视频教程
- 为主要设计决策创建架构决策记录（ADR）
- 构建交互式指南和演练环境

## 如何提议功能

### GitHub Issues

提议功能的主要方式是通过 GitHub Issues：[github.com/ever-works/directory-web-template/issues](https://github.com/ever-works/directory-web-template/issues)

创建功能请求时：

1. **首先检查已有的 issue**，以避免重复。
2. **描述您要解决的问题**，而不仅仅是您想要的解决方案。
3. **提供上下文**，包括您的使用场景、目录类型和规模。
4. **附上示例**（原型图、API 模式、配置示例）。

### GitHub Discussions

对于需要社区投入的更广泛想法：[github.com/ever-works/directory-web-template/discussions](https://github.com/ever-works/directory-web-template/discussions)

### Discord

加入 [Ever Works Discord](https://discord.gg/ever)，进行关于功能和项目方向的实时对话。

## 优先级如何决定

| 因素                 | 权重 | 描述                                                 |
| -------------------- | ---- | ---------------------------------------------------- |
| **用户需求**         | 高   | 请求数量、投票数和社区兴趣                           |
| **战略一致性**       | 高   | 功能与产品愿景的契合程度                             |
| **实现工作量**       | 中   | 复杂性、时间投入和维护负担                           |
| **破坏性变更风险**   | 中   | 对现有用户造成干扰的潜力                             |
| **贡献者可用性**     | 中   | 维护人员或社区成员是否能够承担                       |

### 优先级层次

- **P0（紧急）：** 安全漏洞、数据丢失错误或阻塞性问题。立即处理。
- **P1（高）：** 正在积极为下一个版本开发的功能或修复。
- **P2（中）：** 已批准计划但尚未排期的功能。
- **P3（低）：** 锦上添花的改进。适合社区贡献的候选项。

## 为路线图做出贡献

1. **提交写得好的功能请求**，附有清晰的问题陈述和使用场景。
2. **贡献代码。** Pull request 是从想法到实现的最快路径。参见[贡献指南](/contributing)。
3. **参与讨论。** 对提案提供反馈并分享您的经验。
4. **报告错误。** 可靠的错误报告有助于确定修复优先级并提高稳定性。

## 发布节奏

当有意义的功能和修复集合就绪时就会发布：

- **Patch 版本**（错误修复）根据需要发布，在积极开发期间通常每周发布。
- **Minor 版本**（新功能）大约每月发布。
- **Major 版本**（破坏性更改）不频繁，并附有迁移指南。

详情请参见[变更日志](/changelog)页面。
