# Internationalization Overview

The Ever Works Template uses `next-intl` for internationalization, supporting 21 locales with URL-based routing, automatic browser detection, message fallback, right-to-left layouts, and SEO hreflang tags. This page covers the i18n architecture, configuration, and usage patterns.

## Architecture

| File | Purpose |
|------|---------|
| `i18n/routing.ts` | Route configuration (supported locales, prefix strategy) |
| `i18n/request.ts` | Per-request locale resolution and message loading |
| `i18n/navigation.ts` | Locale-aware Link, redirect, usePathname, useRouter |
| `lib/constants.ts` | Locale list, default locale, RTL locale definitions |
| `lib/seo/hreflang.ts` | SEO hreflang tag generation for search engines |
| `components/language-switcher.tsx` | User-facing language selector dropdown |
| `messages/*.json` | Translation files (one per locale) |
| `app/[locale]/layout.tsx` | Root layout with NextIntlClientProvider |

## Routing Configuration

The routing module at `i18n/routing.ts` defines how locales map to URLs:

```typescript
import { defineRouting } from "next-intl/routing";
import { DEFAULT_LOCALE, LOCALES } from "@/lib/constants";

export const routing = defineRouting({
  locales: LOCALES,
  defaultLocale: DEFAULT_LOCALE,
  localeDetection: true,
  localePrefix: "as-needed",
});
```

### Locale Prefix Strategy

The `"as-needed"` strategy keeps URLs clean for the default locale:

| URL | Resolved Locale | Explanation |
|-----|-----------------|-------------|
| `/about` | `en` (default) | No prefix needed for the default locale |
| `/fr/about` | `fr` | Prefix required for non-default locales |
| `/ar/about` | `ar` | Prefix required for non-default locales |

The default locale (`en`) never appears in the URL. All other locales require an explicit prefix.

### Locale Detection

With `localeDetection: true`, the system automatically:

1. Checks for a locale cookie (set by the LanguageSwitcher component)
2. Reads the `Accept-Language` HTTP header from the browser
3. Falls back to `DEFAULT_LOCALE` (`'en'`) if no match is found

## Message Loading and Fallback

The `i18n/request.ts` module runs on every server request to resolve the locale and load translation messages:

```typescript
export default getRequestConfig(async ({ requestLocale }) => {
  let locale = await requestLocale;

  // Validate locale against supported list
  if (!locale || !routing.locales.includes(locale as any)) {
    locale = routing.defaultLocale;
  }

  // Load locale-specific messages and English fallback
  const userMessages = (await import(`../messages/${locale}.json`)).default;
  const defaultMessages = (await import(`../messages/en.json`)).default;
  // Deep merge: locale messages override English defaults
  const messages = deepmerge(defaultMessages, userMessages);

  return { locale, messages };
});
```

The `deepmerge` strategy ensures that untranslated keys display the English text rather than showing raw key names. Partially translated locale files work correctly -- any missing key falls back to its English equivalent.

## Translation Files

Translation messages are stored as JSON files in the `messages/` directory, one file per locale:

```
messages/
  en.json    # English (source of truth, all keys present)
  fr.json    # French
  es.json    # Spanish
  de.json    # German
  ar.json    # Arabic
  zh.json    # Chinese
  ... (21 files total)
```

### Message Structure

Messages use a flat namespace structure with dot-separated keys:

```json
{
  "auth": {
    "SIGN_IN": "Sign In",
    "SIGN_UP": "Sign up",
    "FORGOT_PASSWORD": "Forgot password?",
    "ENTER_YOUR_EMAIL": "Enter your email"
  },
  "common": {
    "DISCOVER": "Discover",
    "HOME": "Home",
    "ABOUT": "About",
    "SAVE": "Save",
    "SHARE": "Share"
  }
}
```

### Using Translations in Components

**Client components** use the `useTranslations` hook:

```tsx
import { useTranslations } from "next-intl";

function LoginButton() {
  const t = useTranslations("auth");
  return <button>{t("SIGN_IN")}</button>;
}
```

**Server components** use the async `getTranslations`:

```tsx
import { getTranslations } from "next-intl/server";

export default async function Page() {
  const t = await getTranslations("common");
  return <h1>{t("DISCOVER")}</h1>;
}
```

## Navigation Utilities

The `i18n/navigation.ts` module exports locale-aware replacements for standard Next.js navigation:

```typescript
import { createNavigation } from "next-intl/navigation";
import { routing } from "./routing";

export const { Link, redirect, usePathname, useRouter, getPathname } =
  createNavigation(routing);
```

### Usage

```tsx
import { Link, useRouter, usePathname } from "@/i18n/navigation";

// Locale-aware Link component
<Link href="/about">About</Link>
// Renders: /about (en) or /fr/about (fr)

// Locale-aware router
const router = useRouter();
router.push("/about");  // Preserves current locale

// Current pathname without locale prefix
const pathname = usePathname();  // Returns "/about" regardless of locale
```

Always use these exports instead of `next/link` and `next/navigation` to ensure locale-aware routing.

## Root Layout Integration

The root layout at `app/[locale]/layout.tsx` validates the locale and wraps the page tree with the i18n provider:

```typescript
export default async function RootLayout({ children, params }) {
  const { locale } = await params;

  // Reject invalid locales
  if (!routing.locales.includes(locale as Locale)) {
    notFound();
  }

  setRequestLocale(locale);
  const messages = await getMessages();

  return (
    <>
      <NextIntlClientProvider messages={messages}>
        {/* Settings, theme, and other providers */}
        <Providers config={config}>
          {children}
        </Providers>
      </NextIntlClientProvider>
    </>
  );
}
```

The layout also generates static params for the default locale, with other locales rendered on demand:

```typescript
export async function generateStaticParams() {
  return [{ locale: 'en' }];
}
```

## SEO: Hreflang Tags

The `lib/seo/hreflang.ts` module generates hreflang alternate URLs for search engines:

```typescript
export function generateHreflangAlternates(
  path: string
): Record<string, string> {
  const languages: Record<string, string> = {};

  for (const locale of LOCALES) {
    const hreflang = LOCALE_TO_HREFLANG[locale];
    languages[hreflang] = getLocalizedUrl(path, locale);
  }

  // x-default points to the default locale version
  languages['x-default'] = getLocalizedUrl(path, DEFAULT_LOCALE);
  return languages;
}
```

Use it in page metadata:

```typescript
export async function generateMetadata({ params }) {
  const { locale } = await params;
  return {
    alternates: {
      canonical: locale === DEFAULT_LOCALE ? '/' : `/${locale}`,
      languages: generateHreflangAlternates('/')
    }
  };
}
```

This outputs hreflang tags for all 21 locales plus an `x-default` entry, helping search engines serve the correct language version to users.

## LanguageSwitcher Component

The `LanguageSwitcher` component at `components/language-switcher.tsx` provides a dropdown for users to change locale:

```tsx
import { LanguageSwitcher } from "@/components/language-switcher";

// Standard size with flag + label
<LanguageSwitcher />

// Compact mode (flag only)
<LanguageSwitcher compact />
```

### How It Works

1. Renders the current locale flag and two-letter code in a button
2. Opens a scrollable dropdown listing all 21 locales with flag icons
3. On selection, calls `router.replace(pathname, { locale })` via `useTransition`
4. `next-intl` handles cookie persistence and URL rewriting automatically
5. The active locale is highlighted with the theme primary color
6. Escape key and outside-click close the dropdown

### Accessibility

The component includes `aria-label`, `aria-expanded`, and `aria-controls` attributes. Buttons are disabled during locale transitions to prevent double-clicks.

## Adding a New Locale

1. Add the locale code to the `LOCALES` array in `lib/constants.ts`
2. If the locale is RTL, add it to `RTL_LOCALES`
3. Create `messages/<code>.json` with translated keys (missing keys fall back to English)
4. Add a flag SVG to `public/flags/<code>.svg`
5. Add the locale entry to the `languageMap` in `components/language-switcher.tsx`
6. Add the hreflang mapping in `lib/seo/hreflang.ts`

## Best Practices

1. **Use `Link` from `@/i18n/navigation`** instead of `next/link` to ensure locale-aware routing
2. **Add new keys to `en.json` first** since it is the fallback language for all locales
3. **Never hardcode English strings** in components -- use `useTranslations` or `getTranslations`
4. **Rely on the `deepmerge` fallback** -- partially translated locale files work correctly
5. **Test RTL layouts** by switching to Arabic or Hebrew in the LanguageSwitcher
6. **Use `localePrefix: "as-needed"`** to keep default-locale URLs clean while supporting SEO prefixes for other locales
